현재 Machi_MK5에서 gemma3:4b 같은 LLM은 “생각의 주체”라기보다, 구조화된 내부 결과를 해석하고 정리하는 보조 실행기 역할을 맡고 있습니다.

이게 중요합니다. MK5는 전통적인 “LLM이 다 알아서 답하는 챗봇” 구조가 아니라,

그래프 ingest
activation
thought engine
search decision / slot planning / coverage refinement
같은 내부 파이프라인이 먼저 움직이고,
LLM은 그 뒤 특정 지점에서 제한된 역할로 호출됩니다.
지금 기준으로 gemma3:4b가 맡는 역할은 크게 세 군데입니다.

1. 질문을 검색 가능한 슬롯 구조로 바꾸는 역할
관련 파일:

question_slot_planner.py
여기서 LLM은 사용자의 질문을 보고:

어떤 entity를 검색해야 하는지
어떤 search_aspect가 필요한지
어떤 comparison_axis가 있는지
를 JSON 형태로 뽑아냅니다.
즉, 이 단계의 LLM은 “답변 생성기”가 아니라
질문 분해기 / 검색 구조화기입니다.

예:

판금갑옷, 사슬갑옷, 미늘갑옷
구조, 방어력, 기동성
이런 식으로 search slot을 만드는 역할이죠.

2. 검색 결과가 질문의 요구 슬롯을 얼마나 메웠는지 다시 판정하는 역할
관련 파일:

search_coverage_refiner.py
여기서는 LLM이 검색 결과 요약을 보고:

어떤 slot은 채워졌는지
어떤 slot은 아직 비었는지
를 다시 구조화해서 돌려줍니다.
즉 이 단계의 LLM은
검색 커버리지 판정기입니다.

중요한 점은, MK5가 string heuristic로
“이 단어가 있으면 됐다”
이런 식으로 안 가고, 지금은 이 판정을 LLM에게 맡기고 있다는 점입니다.

3. 최종 사용자 응답을 자연어로 만드는 역할
관련 파일:

ollama_verbalizer.py
verbalizer.py
이 단계에서 LLM은:

CoreConclusion
DerivedActionLayer
search_context
를 바탕으로
최종 답변 문장을 만듭니다.
즉 이때는
언어화기 / surface realization engine 역할입니다.

다만 그냥 마음대로 말하는 건 아니고,

search context
do_not_claim
meaning_preserver
에 의해 제약받습니다.
그래서 이 단계에서도 LLM은 “지식의 원천”이라기보다
내부 결정 결과를 사용자 문장으로 풀어주는 렌더러에 더 가깝습니다.

정리하면, 지금 MK5에서 gemma3:4b의 역할은 이렇습니다.

질문을 검색 가능한 구조로 분해
검색 결과가 충분한지 판정
내부 결론을 자연어 응답으로 표현
반대로 LLM이 맡지 않는 역할도 분명합니다.

LLM이 주도하지 않는 것

그래프 저장/갱신
activation 계산
contradiction/trust revision
핵심 추론 상태 보존
search backend 실행
evidence ingest
meaning preservation 최종 가드
즉 MK5에서 LLM은 시스템의 “두뇌 전체”가 아니라,
그래프 기반 코어 위에서 구조화/판정/언어화에 참여하는 전문 하위 모듈입니다.

비유하면 이렇게 볼 수 있습니다.

그래프/파이프라인: 실제 작업자, 상태 관리자, 내부 추론 엔진
LLM: 해석 비서, 분류 비서, 문장 작성 비서
그래서 gemma3:4b가 느리거나 품질이 안 좋으면 영향은 크지만,
Machi의 전체 철학이 무너지는 건 아닙니다.
왜냐하면 MK5는 애초에 “LLM 하나에 전부 의존하지 않도록” 설계되고 있기 때문입니다.

한 줄 요약:
gemma3:4b는 MK5에서 “모든 걸 아는 두뇌”가 아니라, 질문 구조화기 + 검색 커버리지 판정기 + 최종 언어화기 역할을 맡고 있습니다.

그럼 그래프가 맡는 역할은? 그리고 그 그래프가 그 각 역할들을 진행하는 원리는?
오전 8:58


그래프가 맡는 역할은, MK5 전체에서 보면 LLM이 즉흥적으로 판단하지 않도록, 시스템의 내부 상태와 의미 관계를 유지하는 중심 저장소이자 작업 공간입니다.

조금 더 정확히 말하면, 그래프는 단순한 “메모장”이 아닙니다.
Machi_MK5에서 그래프는 적어도 아래 역할을 맡고 있습니다.

1. 사용자 입력과 시스템 출력을 구조화해서 저장하는 역할
사용자 발화, 검색 결과, 시스템 답변은 그냥 텍스트로만 남지 않습니다.
이들은 의미 블록으로 쪼개지고, 노드/엣지/포인터 형태로 그래프 안에 들어갑니다.

즉 그래프는:

어떤 개념이 등장했는지
어떤 개념과 어떤 개념이 연결되는지
이 연결이 사용자 발화에서 왔는지, 검색에서 왔는지, 시스템 답변에서 왔는지
어떤 메시지와 어떤 이벤트에서 생겼는지
를 구조적으로 보존합니다.

그래서 그래프의 첫 번째 역할은
대화를 “의미 단위”로 축적하는 기억 기반입니다.

2. 현재 질문과 관련된 내부 맥락을 꺼내오는 역할
사용자가 새 질문을 하면, MK5는 전체 DB를 다 읽는 게 아니라:

입력에서 seed block을 만들고
그 seed와 연결된 node/edge/pointer를 따라가며
현재 질문과 가까운 국소 서브그래프를 만듭니다
이게 activation 단계입니다.

즉 그래프의 두 번째 역할은
현재 질문에 필요한 지역적 맥락을 활성화하는 기억 네트워크입니다.

중요한 건, 이 덕분에 MK5는 매 턴마다 “모든 걸 처음부터 생각하는” 게 아니라,
이미 쌓인 구조를 다시 불러와서 그 위에서 생각합니다.

3. 생각(thinking)의 재료를 제공하는 역할
ThoughtEngine은 허공에서 추론하지 않습니다.
이미 activation된 그래프 상태를 보고:

어떤 관계가 핵심인지
어떤 구조 충돌이 있는지
어떤 edge는 신뢰가 낮아졌는지
어떤 revision이 필요한지
를 판단합니다.
즉 그래프는 thought engine에게:

현재 활성 개념
관계망
모순 신호 후보
revision 대상
을 제공합니다.
그래프의 세 번째 역할은
추론 대상이자 추론 재료입니다.

4. trust와 revision의 누적 기반 역할
Machi의 핵심은 “한 번 들어온 정보를 무조건 사실로 고정하지 않는다”는 점입니다.
그래프 안의 edge나 relation은:

support를 받을 수도 있고
conflict를 받을 수도 있고
trust가 오르내릴 수도 있고
deactivation될 수도 있습니다
이 말은 그래프가 단순한 정적 지식베이스가 아니라
변하는 신뢰 상태를 가진 동적 구조라는 뜻입니다.

즉 네 번째 역할은
사실 여부가 아니라, 현재 신뢰 상태를 가진 관계망입니다.

5. 검색이 필요한지 판단하는 내부 기준점 역할
검색도 완전히 LLM이 독단적으로 정하는 게 아닙니다.
현재 그래프를 보고:

어떤 slot이 비어 있는지
어떤 entity는 이미 내부에 어느 정도 있는지
무엇이 빠져 있는지
를 바탕으로 search need를 판정합니다.
그래프의 다섯 번째 역할은
외부 검색이 필요한지 판단하는 내부 기준면입니다.

이제 “그럼 그래프가 이런 역할들을 어떤 원리로 수행하느냐”가 핵심인데,
원리는 크게 다섯 개로 설명할 수 있습니다.

1. 텍스트를 그대로 저장하지 않고 의미 블록으로 분해한다
Machi는 입력을 바로 큰 문장 단위로만 보지 않고,

statement phrase
noun phrase
같은 의미 블록으로 나눕니다.
이 블록들이 다시 노드/포인터/관계 생성의 출발점이 됩니다.

즉 원리는:

텍스트
의미 블록
그래프 요소
순으로 변환하는 것입니다.
이렇게 해야 나중에 “이 질문이 저번 질문의 어느 개념과 닿는지”를 잡아낼 수 있습니다.

2. 개념을 노드로, 관계를 엣지로, 출처를 이벤트/포인터로 유지한다
그래프의 기본 작동 원리는 매우 전통적입니다.

개념: node
관계: edge
출처/연결 흔적: pointer, event
그런데 Machi에서는 여기서 끝나지 않고,
각 요소에 메타데이터가 붙습니다.

source type
claim domain
trust 관련 상태
생성/재사용 여부
어느 message/event에서 왔는지
즉 그래프는 단순한 개념 연결도가 아니라
출처와 상태를 가진 의미 네트워크로 동작합니다.

3. 현재 질문에서 가까운 부분만 활성화한다
그래프 전체를 매번 쓰면 너무 무겁고 잡음도 많아집니다.
그래서 activation engine은:

현재 입력에서 seed를 만들고
그 seed와 연결된 국소 노드/엣지를 확장하고
서브그래프를 thought engine에 넘깁니다
이 원리는 뇌 비유로 하면 “전 기억을 다 꺼내지 않고, 관련 기억만 점화한다”에 가깝습니다.

그래프가 추론에 기여하는 핵심 원리는 바로 이
local activation 입니다.

4. 관계의 강도와 충돌을 누적적으로 갱신한다
Machi 그래프는 “A와 B가 연결되어 있다”에서 멈추지 않고,
그 연결이 얼마나 믿을 만한지, 지금 압력이 어떤지를 계속 갱신할 수 있게 설계돼 있습니다.

예를 들면:

어떤 검색 근거가 들어오면 기존 edge를 support
어떤 발화가 기존 관계와 어긋나면 conflict signal 생성
revision 단계에서 trust 감소 또는 deactivation
즉 그래프는 정답을 저장하는 구조가 아니라
관계의 상태 변화를 누적하는 구조로 작동합니다.

5. 내부 요약은 그래프 상태를 바탕으로만 만든다
최종적으로 CoreConclusion 같은 내부 결론은
LLM의 자유 생성물이 아니라,
활성 그래프 상태를 보고 만들어집니다.

그래서 conclusion에는:

activated_concepts
key_relations
detected_conflicts
revision_decisions
같은 그래프 기반 지표가 들어갑니다.
즉 그래프는 결과를 위한 참고자료가 아니라,
결론 생성의 근거 그 자체입니다.

조금 더 직관적으로 말하면, MK5에서 그래프는 이런 흐름으로 작동합니다.

입력이 들어온다
입력을 블록으로 나눈다
블록에서 개념과 관계를 그래프에 반영한다
새 질문에 맞는 관련 부분만 활성화한다
활성화된 서브그래프를 바탕으로 충돌/신뢰/의도를 본다
아직 비어 있는 부분이 있으면 검색을 붙인다
검색 결과도 다시 그래프에 넣는다
그래프 상태를 기반으로 conclusion을 만든다
그 conclusion을 verbalizer가 말로 바꾼다
즉 그래프는 중간 저장소가 아니라,
입력, 기억, 활성화, 추론, 검색 판단, 최종 응답 이전 결론 생성까지 모두 관통하는 중심 구조입니다.

LLM과 그래프의 차이도 여기서 선명해집니다.

그래프는 상태를 유지한다
그래프는 출처와 관계를 보존한다
그래프는 무엇이 연결되어 있고 무엇이 충돌하는지 구조로 가진다
그래프는 턴이 지나도 누적된다
반면 LLM은:

주어진 입력을 해석하고
구조를 정리하고
자연어로 풀어주는
보조 계산기입니다
즉 MK5 철학에서 그래프는
기억과 추론의 본체이고,
LLM은
그 본체 위에서 특정 작업을 돕는 도구입니다.

한 줄 요약으로 정리하면:

그래프는 MK5에서
의미를 저장하고, 관련 맥락을 활성화하고, 관계의 신뢰와 충돌을 누적하며, 최종 내부 결론의 근거를 제공하는 중심 구조이고,
그 원리는
텍스트를 의미 블록으로 분해해 노드/엣지/포인터로 축적하고, 현재 질문과 가까운 서브그래프만 활성화해 그 위에서 추론하는 방식입니다.

“그렇다면 현재 MK5 그래프가 아직 약한 부분은 어디인가”
또는
“search와 graph의 경계는 정확히 어디인가”
까지 이어서 설명해드릴게요.

오전 9:00



현재 MK5 그래프가 맡는 역할은 분명 크지만, 아직 “강한 기억-추론 그래프”라고 부르기엔 몇 군데가 약합니다. 그리고 두 번째 질문에 먼저 짧게 답하면, 사용자와의 대화에서 주제에 관한 사고, 추론, 결론 생성의 중심은 지금 그래프 기반 파이프라인이 맞습니다. 다만 “완전히 그래프만으로 한다”기보다는, 그래프가 중심 상태를 제공하고 그 위에 일부 LLM 보조 계층이 얹혀 있는 구조에 가깝습니다.

먼저 약한 부분부터 짚으면, 가장 큰 건 주제 구조의 지속성과 정교한 주제 전환 처리입니다. 지금 MK5는 입력을 그래프에 넣고 activation으로 관련 부분을 불러와 thought engine이 conclusion을 만들지만, “이번 대화의 장기적 주제”, “방금 전 질문과 지금 질문의 주제적 연속성”, “하위 주제로 내려갔다가 다시 상위 주제로 올라오는 흐름”을 명시적으로 관리하는 계층은 아직 약합니다. 즉 관련 노드가 활성화되긴 해도, 그걸 우리가 흔히 말하는 “대화 주제 관리” 수준으로 안정적으로 운영하는 단계는 아닙니다.

그다음으로 약한 건 관계 의미의 풍부함입니다. 지금 그래프는 노드와 엣지, trust, revision 같은 구조를 갖고 있지만, 엣지의 의미가 아직 충분히 풍부한 ontology로 정리되어 있다고 보긴 어렵습니다. 예를 들어 사용자가 “A와 B의 차이”, “A가 B의 일부인지”, “A와 B가 같은 범주인지”, “이건 예외인지 일반 규칙인지”를 물을 때, 그 차이를 엣지 타입만으로 안정적으로 구분하고 장기적으로 유지하는 힘은 더 키워야 합니다. 지금은 관계가 저장되고 추론 재료가 되긴 하지만, 관계 의미 체계가 더 세밀해져야 진짜 강한 그래프 사고가 됩니다.

또 하나는 검색 결과가 그래프의 장기 지식으로 흡수되는 방식입니다. 지금은 검색 결과를 ingest하고 그걸 바탕으로 같은 턴의 답변 품질을 높이는 쪽은 꽤 좋아졌습니다. 하지만 “검색으로 확인된 정보가 이후 턴에서도 안정적으로 재활용되는가”, “일회성 evidence와 장기 기억을 어떻게 구분할 것인가”, “외부 정보가 기존 내부 그래프와 충돌할 때 어떤 정책으로 재배선할 것인가”는 아직 더 다듬어야 합니다. 여기서 아까 이야기한 edge_update_service, graph_commit_service가 장기적으로 의미를 갖게 됩니다.

그리고 추론의 깊이와 형태도 아직 제한적입니다. 지금 thought engine은 activation된 그래프를 보고 contradiction, trust, revision, intent를 바탕으로 conclusion을 만듭니다. 이건 분명 그래프 기반 사고입니다. 다만 아직은 “여러 단계의 명시적 체인 추론”, “비교 축을 따라 체계적으로 결론을 조합”, “예외와 반례를 함께 유지한 추론” 같은 수준까지 강하다고 하긴 어렵습니다. 다시 말해 지금 그래프는 “관련 맥락을 꺼내고, 충돌과 부족을 감지하고, 보수적 conclusion을 만드는 것”에는 강해지고 있지만, 복합 주제에 대한 고차 추론 엔진으로는 아직 성장 중입니다.

또 약한 부분은 주제 단위 요약과 압축 기억입니다. 지금은 각 턴이 그래프에 들어가고 activation으로 다시 꺼내오는 방식인데, 대화가 길어질수록 “이 사용자는 지금까지 이 주제에서 무엇을 이미 이해했고, 무엇을 아직 묻고 있으며, 지금 논의 중인 핵심 축은 무엇인가”를 주제 단위로 압축해 관리하는 메커니즘은 더 필요합니다. 이게 생기면 그래프는 단순 저장소가 아니라 훨씬 더 강한 conversation state manager가 됩니다.

이제 두 번째 질문으로 돌아가면, “주제에 관한 대화 및 사고, 추론, 결론 생성 등을 관장하는 부분이 지금 그래프가 맞느냐”에 대해서는, 네, 중심은 그래프가 맞습니다. 이유는 이렇습니다.

사용자 입력은 그래프에 ingest됩니다.
activation은 그래프에서 현재 질문과 가까운 맥락을 꺼냅니다.
thought engine은 그 활성 그래프를 보고 CoreConclusion을 만듭니다.
search need도 그래프의 빈 슬롯과 부족한 grounding을 기준으로 판단합니다.
최종 verbalizer는 이미 생성된 CoreConclusion을 말로 바꾸는 후단입니다.
즉 “주제에 대해 무엇을 생각할 것인가”, “지금 어떤 관계가 중요하고 어떤 점이 비어 있는가”, “지금 결론을 얼마나 보수적으로 잡아야 하는가”는 그래프 기반 파이프라인이 주도합니다. 이 점에서 MK5는 확실히 그래프 중심입니다.

다만 주의해서 말하면, 주제의 ‘표현’은 그래프가 아니라 그래프+파이프라인 전체가 관장합니다.
그래프만 단독으로 생각하지는 않습니다. 실제로는:

그래프: 상태와 관계를 보존
activation: 주제 관련 부분을 활성화
thought engine: 활성 그래프를 해석해 conclusion 생성
search layer: 그래프의 빈 부분을 외부 근거로 보강
verbalizer: conclusion을 자연어로 표현
meaning preserver: 그 표현이 주제와 근거를 벗어나지 않게 감시
이 흐름 전체가 “주제에 대한 사고와 결론 생성”을 담당합니다.
그래프는 그중 중심 기억-상태 기반이고, thought/search/verbalization은 그 그래프를 사용하는 실행 계층입니다.

그래서 가장 정확한 표현은 이겁니다:

주제의 상태를 관장하는 중심은 그래프다
주제에 대한 사고와 결론 생성을 실행하는 것은 그래프 기반 파이프라인이다
최종 자연어 응답만 LLM이 담당한다
지금 MK5의 약점을 한 문장으로 요약하면,
그래프는 이미 중심 역할을 하고 있지만, 아직 “주제 관리”, “관계 의미의 풍부함”, “장기 기억화된 외부 근거”, “고차 추론” 측면은 더 강화되어야 한다 입니다.
