# 작업사항

## 현재 어디까지 되었는가

### 1. 저장 계층
- `storage/schema.sql` 기준으로 `chat_messages / nodes / edges / graph_events / node_pointers` 스키마가 있다.
- SQLite 구현체가 붙어 있고, `SqliteUnitOfWork`로 한 트랜잭션 안에서 접근 가능하다.
- repository 계층은 CRUD가 아니라 그래프 엔진 중심 질의 기준으로 잡혀 있다.

### 2. 입력 인지 및 graph ingest
- `core/cognition/input_segmenter.py`에서 입력을 문장 경계와 의미블록으로 나눈다.
- 원문/문장은 provenance로 남기고, 영구 그래프에는 재사용 가능한 의미블록을 반영한다.
- `core/update/graph_ingest_service.py`에서 다음이 작동한다.
  - chat_message 저장
  - graph_event 기록
  - meaning block 추출
  - 기존 node 재사용 또는 새 node 생성
  - source_type / claim_domain 기반 초기 trust 부여
  - 같은 입력 내부 공출현 기반 약한 edge 생성/강화
  - partial reuse pointer 생성
  - 반복 등장 시 node / edge trust 상승

### 3. source-aware trust policy
- `core/update/source_trust_policy.py`가 있다.
- 세계 그래프는 하나로 유지하되, source_type과 claim_domain에 따라:
  - 초기 trust
  - support 시 trust 상승률
  - conflict 시 trust 하락률
을 다르게 준다.

### 4. 국부 활성화
- `core/activation/activation_engine.py`와 `thought_view_builder.py`가 있다.
- 현재 입력 기준으로 seed block -> seed node -> local node / local edge / pointer를 모아 `ThoughtView`를 만든다.
- `PatternDetector`가 `ThoughtView`를 분석해 `SubgraphPattern`을 감지하고,
  `ThoughtView.activated_patterns`에 활성 패턴을 저장한다.
- `ConflictResolutionPolicy`가 노드/그래프 수준 패턴 충돌을 통일된 방식으로 해결한다.

### 5. thinking 계층
현재는 아래 축이 살아 있다.
- `ContradictionDetector`
- `TrustManager`
- `StructureRevisionService`
- `IntentManager`
- `ThoughtEngine`

현재 흐름:
- 구조 충돌 감지
- trust 하락
- revision candidate 표기
- revision 단계 review
- 필요 시 shallow duplicate merge 또는 edge 비활성화
- 현재 그래프 상태와 recent continuity를 기반으로 intent snapshot 결정

### 6. shallow node merge / pointer rewrite
- `NodeMergeService`
- `PointerRewriteService`
가 있다.
- 현재 merge는 **revision 단계에서만** 트리거된다.
- 누적량은 조금 얕게 잡되,
- 실제 merge 허용 조건은 duplicate-like node로 보수적으로 제한되어 있다.

### 7. 설명형 core conclusion + 행동형 파생 레이어
- `CoreConclusion`은 설명형 본체다.
- `activated_concepts`는 node id 참조 목록이다.
- `key_relations`는 edge id 참조 목록이다.
- `DerivedActionLayer`는 설명형 conclusion 위에 얹힌 얇은 행동형 레이어다.

### 8. verbalization
- `TemplateVerbalizer`
- `OllamaVerbalizer`
- `Verbalizer`
가 있다.
- 현재 `core/verbalization/ollama_verbalizer.py`는 직접 OLLAMA `/api/chat`을 호출한다.
- assistant 응답 생성 실패는 fallback으로 숨기기보다 오류로 드러내는 방향을 유지한다.

### 9. search sidecar
- `core/search/search_sidecar.py`가 있다.
- 현재는 explanation 계열 질문에서 위키피디아 중심 검색을 시도한다.
- 검색 결과는 별도 층이 아니라 같은 세계 그래프에 낮은 trust로 반영된다.

### 10. assistant 답변의 그래프 반영
- 최종 assistant 답변도 `source_type="assistant"`로 그래프에 반영된다.
- assistant message metadata 안에 `intent_snapshot`도 같이 남는다.
- 다음 turn에서는 이 snapshot을 읽어 continuity / shift 판단에 사용한다.

### 11. chat end-to-end
- `app/chat_pipeline.py` 기준으로
  - user ingest
  - activation
  - thinking
  - search enrichment
  - re-thinking
  - derived action
  - verbalization
  - assistant ingest
  이 한 줄이 닫혀 있다.

## 아직 안 된 것 / 추가적으로 필요한 것

### 1. Flask 실서버 실행 검증
- 이 작업 환경에서는 Flask 설치가 없어 실제 서버 기동을 검증하지 못했다.
- 즉 `python run.py`가 네 로컬에서 정상 동작하는지는 로컬 검증이 필요하다.

### 2. `requirements.txt`
- 아직 정리가 충분하지 않다.
- 최소 의존성 목록을 실제 코드 기준으로 다시 맞춰야 한다.

### 3. debug UI 최종 정리
- `chat.js` 최신본 반영 여부와 브라우저 캐시 문제 확인이 필요하다.
- 장기적으로는 debug 패널 UI를 더 정리해야 한다.

### 4. 검색을 위키피디아만이 아닌 trusted_search로 변경
- 현재 검색은 위키피디아 중심이다.
- 장기적으로는 source 신뢰도 관리가 가능한 trusted_search 레이어로 바꾸는 것이 맞다.

### 5. `tools/ollama_client.py` 실구현
- 현재 placeholder 상태다.
- 실제 OLLAMA 호출 계약을 이 파일로 모으고, verbalizer / tool 레이어가 이를 공유하도록 정리할 필요가 있다.

### 6. 검색 결과 corroboration
- 지금은 단일 검색 결과도 낮은 trust로 그래프에 들어간다.
- 이후에는 여러 검색 결과 / 여러 source 간 교차 일치가 있을 때 trust를 더 강하게 올리는 정책이 필요하다.

### 7. source_type × claim_domain 정책 정교화
- 현재 trust 정책은 시작점 수준이다.
- 사용자 자기진술 / 외부사실 / 모델 생성답변 / 검색결과 / 파일입력에 대한 세밀한 가중치 조정이 필요하다.

### 8. meaning block 정교화
- 현재 의미블록 추출은 시작점 수준이다.
- 명사구 / 짧은 판단구 / 관계구 중심인데, 실제 대화 품질을 높이려면 점진적 정교화가 필요하다.

### 9. revision 고도화
- 현재 revision은 보수적 규칙 기반이다.
- 장기적으로는 revision 히스토리 축적, 노드/엣지 안정성 평가, 구조 교체 후 후속 영향 관리가 필요하다.

### 10. `graph_commit_service.py` / `edge_update_service.py`
- 현재 placeholder 단계다.
- graph rewrite를 더 안정적으로 묶는 commit orchestration이 필요하다.

- 세계 그래프는 하나다.
- 사용자/모델/검색 결과 모두 그래프에 영향을 줄 수 있다.

- 프롬프트 문자열 prompts/ 외부화 정리

## 개념 업데이트 메모 (2026-04-16)

### 1. 존재 계층
- 존재 계층은 `instance_of`보다 `concept / flow` 중심으로 본다.
- `지성체 --[concept / flow]--> 사람`
- `지성체 --[concept / flow]--> AI`
- `사람 --[concept / flow]--> 나`
- `AI --[concept / flow]--> Machi`
- 즉 개별 존재도 상위 개념 아래의 더 구체적인 하위 개념으로 다룬다.

### 2. 이름 / 별칭 / 호칭
- `신재용`, `Jay`, `재용`은 각각 별도의 Node가 될 수 있다.
- 이들은 존재 계층 flow가 아니라 `concept / neutral` 연결로 묶는 방향이 맞다.
- 즉 존재 계층과 이름 표상 축을 분리한다.

### 3. 관계와 상태
- 관계와 상태는 `relation` family Edge가 맡는다.
- 세부 관계 의미는 Edge 내부 데이터에 넣는다.
- "존재"는 Node, "연결 방식"은 Edge라는 원칙을 유지한다.

### 3-1. connect_type
- 초기 `connect_type`은 `flow`, `neutral`, `opposite`
- `flow`: 분화/흐름/영향 방향이 있는 연결
- `neutral`: 유사/동류/표상/상호적 연결
- `opposite`: 같은 축 안의 반대 개념 연결
- 향후 새로운 connect type이 필요하면, 우선 Edge data 안에 후보로 누적한 뒤 충분히 반복될 때 정식 승격한다.

### 4. 시간성
- `2026년 3월의 Machi`를 새 subtype 노드로 만들지 않는다.
- 같은 `Machi` Node를 과거 시점 기준으로 역추적해서 본다.
- 따라서 시간성은 새 존재 생성보다 Node / Edge 이력 조회 문제로 본다.

### 5. Node / Edge 시점 정보
- 향후 역추적을 위해 Node와 Edge 모두 생성/업데이트 시점 정보를 가져야 한다.
- 예:
  - Node: `created_at`, `updated_at`, `created_event_id`, `last_updated_event_id`
  - Edge: `created_at`, `updated_at`, `created_event_id`, `last_supported_event_id`, `last_conflicted_event_id`
- 이렇게 해야 "이 시점보다 뒤에서 생긴 상태이므로 더 이전으로 가야 한다"는 판단이 가능해진다.
