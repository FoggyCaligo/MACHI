# 작업사항

## 현재 어디까지 되었는가

### 1. 저장 계층
- `storage/schema.sql` 기준으로 `chat_messages / nodes / edges / graph_events / node_pointers` 스키마가 있다.
- SQLite 구현체가 붙어 있고, `SqliteUnitOfWork`로 한 트랜잭션 안에서 접근 가능하다.
- repository 계층은 단순 CRUD가 아니라 `address_hash 조회`, `active relation 조회`, `revision candidate 조회`, `support/conflict 누적` 같은 그래프 엔진 중심 질의 기준으로 잡혀 있다.

### 2. 입력 인지 및 그래프 ingest
- `core/cognition/input_segmenter.py`에서 입력을 문장 경계와 의미블록으로 나눈다.
- 현재 정책은 **원문/문장 = provenance**, **영구 그래프 = 재사용 가능한 의미블록**이다.
- `core/update/graph_ingest_service.py`에서 다음 흐름이 작동한다.
  - chat_message 저장
  - graph_event 기록
  - meaning block 추출
  - 기존 node 재사용 또는 새 node 생성
  - source_type / claim_domain 기반 초기 trust 부여
  - 같은 입력 내부 공출현 기반 약한 edge 생성/강화
  - partial reuse pointer 생성
  - 반복 등장 시 node / edge trust 상승

### 3. source-aware trust policy
- `core/update/source_trust_policy.py`가 있다.
- 세계 그래프는 하나로 유지하되, source_type과 claim_domain에 따라:
  - 초기 trust
  - support 시 trust 상승률
  - conflict 시 trust 하락률
을 다르게 준다.
- 현재 기본 소스는 `user / assistant / search / file` 정도를 상정한다.

### 4. 국부 활성화
- `core/activation/activation_engine.py`와 `thought_view_builder.py`가 있다.
- 현재 입력 기준으로 seed block -> seed node -> local node / local edge / pointer를 모아 `ThoughtView`를 만든다.
- `PatternDetector`가 `ThoughtView`를 분석해 `SubgraphPattern`을 감지하고,
  `ThoughtView.activated_patterns`에 활성 패턴을 저장한다.
- `ConflictResolutionPolicy`가 노드/그래프 수준 패턴 충돌을 통일된 방식으로 해결한다.

### 5. thinking 계층
- `ContradictionDetector`
- `TrustManager`
- `StructureRevisionService`
- `ThoughtEngine`

현재는 아래 흐름이 들어가 있다.
- 구조 충돌 감지
- trust 하락
- revision candidate 검토
- 필요 시 edge 비활성화

즉, **기본은 구조 보존 / 반복 충돌 시 신뢰도 하락 / 바닥 도달 시 구조 교체 후보화** 방향이 코드에 반영되어 있다.

### 6. 설명형 core conclusion + 행동형 파생 레이어
- `CoreConclusion`은 설명형 본체다.
- `activated_concepts`는 **node id 참조 목록**이다.
- `key_relations`는 **edge id 참조 목록**이다.
- `DerivedActionLayer`는 설명형 conclusion 위에 얹힌 얇은 행동형 레이어다.

### 7. verbalization
- `mk5-graph-core`일 때는 template verbalizer 사용
- 다른 모델이 선택되면 `OllamaVerbalizer`가 실제 OLLAMA `/api/chat`을 호출
- 실패 시 template 응답으로 fallback

### 8. search sidecar
- `core/search/search_sidecar.py`가 있다.
- explanation 계열 질문에서는 외부 검색을 시도할 수 있다.
- 검색 결과는 별도 층이 아니라, **같은 세계 그래프에 낮은 trust로 반영**된다.
- 현재는 위키피디아 검색을 우선 사용한다.

### 9. assistant 답변의 그래프 반영
- 최종 assistant 답변도 `source_type="assistant"`로 그래프에 반영된다.
- 즉 MK5는 사용자 입력만이 아니라, 모델 자기 발화도 세계 인식의 일부로 누적한다.

### 10. chat end-to-end
- `app/chat_pipeline.py` 기준으로
  - user ingest
  - activation
  - thinking
  - search enrichment
  - re-thinking
  - derived action
  - verbalization
  - assistant ingest
  이 한 줄이 닫혀 있다.

## 아직 안 된 것 / 추가적으로 필요한 것

### 1. Flask 실서버 실행 검증
- 이 작업 환경에서는 Flask 설치가 없어 실제 서버 기동을 검증하지 못했다.
- 즉 `python run.py`가 네 로컬에서 정상 동작하는지는 로컬 검증이 필요하다.

### 2. `requirements.txt`
- 아직 없다.
- 최소한 `flask`는 명시되어야 하고, OLLAMA 연동/추가 도구까지 고려한 정리가 필요하다.

### 3. debug UI 최종 정리
- 현재 `chat.js` 최신본을 적용하면 debug를 숨길 수 있지만, 네 로컬에 실제 반영 여부와 브라우저 캐시 문제 확인이 필요하다.
- 장기적으로는 접고 펼치는 패널 UI를 더 정리해야 한다.

### 4. 검색 결과 corroboration
- 지금은 단일 검색 결과도 낮은 trust로 그래프에 들어간다.
- 이후에는 여러 검색 결과 / 여러 source 간 교차 일치가 있을 때 trust를 더 강하게 올리는 정책이 필요하다.

### 5. source_type × claim_domain 정책 정교화
- 현재 trust 정책은 시작점 수준이다.
- 사용자 자기진술 / 외부사실 / 모델 생성답변 / 검색결과 / 파일입력에 대한 세밀한 가중치 조정이 필요하다.

### 6. meaning block 정교화
- 현재 의미블록 추출은 시작점 수준이다.
- 명사구 / 짧은 판단구 / 관계구 / 정정구 중심인데, 실제 대화 품질을 높이려면 점진적 정교화가 필요하다.

### 7. revision 고도화
- 현재 revision은 보수적 규칙 기반이다.
- 장기적으로는 revision 히스토리 축적, 노드/엣지 안정성 평가, 구조 교체 후 후속 영향 관리가 필요하다.

## 참고 철학
- 세계 그래프는 하나다.
- 사용자/모델/검색 결과 모두 그래프에 영향을 줄 수 있다.
- 다만 같은 지위로 곧바로 안정화되지 않는다.
- source_type과 claim_domain에 따라 trust가 다르게 시작하고, support/conflict로 달라진다.
- 본체는 설명형 conclusion이고, 행동형은 얇은 파생 레이어다.
