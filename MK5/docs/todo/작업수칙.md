# MK5 작업수칙

이 문서는 MK5를 수정할 때 계속 지켜야 하는 작업 원칙이다.

## 1. 기준본 원칙
- 작업 기준은 항상 사용자가 가장 최근에 올린 로컬이다.
- ZIP은 sync 맞추기용이다.
- 이후 수정은 반드시 내가 풀어둔 최신 로컬 폴더 기준으로만 이어간다.
- 분기된 여러 버전을 따로 발전시키지 않는다.

## 2. sync 맞추는 방법
- 사용자가 최신 ZIP을 올리면, 그 ZIP을 풀어 현재 로컬 작업 폴더와 sync를 먼저 맞춘다.
- 이후 수정은 반드시 그 기준본 위에서만 이어서 적용한다.
- 예전 ZIP 스냅샷이나 과거 패치본 위로 덮어쓰지 않는다.

## 3. 파일 제공 방식
- 전체 ZIP 재배포보다 수정된 파일만 파일 단위로 제공하는 것을 기본으로 한다.
- 각 파일은 반드시 아래 형식으로 준다.
  - 원래 경로
  - 다운로드 링크
- 여러 ZIP 버전을 각각 발전시키는 식으로 결과를 쪼개지 않는다.

## 4. 오류 처리 원칙
- 실패했을 때는 실패로 드러나게 둔다.
- 오류를 문자열 비교, 임시 우회, fallback 남용으로 숨기지 않는다.
- 필수 계약 위반은 가능하면 명확하게 드러나게 둔다.
- 본체 응답 계약을 helper/fallback으로 우회하지 않는다.

## 5. 하드코딩 금지
- 문자열 하드코딩과 문자열 휴리스틱은 금지한다.
- 의미 해석을 `if text contains ...` 식으로 처리하지 않는다.
- 사람/AI/정체성/호칭/주제/관계 판단을 문자열 규칙으로 때우지 않는다.
- route 결정, tool 사용 여부, correction 의도 판별을 문자열 패턴으로 처리하지 않는다.

## 6. 그래프 우선 원칙
- 구조 판단은 prompt보다 그래프가 우선한다.
- 존재는 Node, 관계와 상태는 Edge가 맡는다.
- Edge의 본체는 `edge_family + connect_type + 방향 + 상태값`이다.
- `kind` / `connect_semantics` 같은 문자열 ontology를 본체 의미론으로 쓰지 않는다.
- 사고는 local activation에서 하고, direct access는 조회/관리 경로로 분리한다.

## 7. 시간성과 로그
- 과거 시점마다 새 노드를 계속 만들지 않는다.
- 현재 node/edge와 bounded append-only log를 바탕으로 필요할 때 과거 국소 그래프를 재형성한다.
- log는 현재 상태를 대체하지 않고, 재형성 재료로 사용한다.

## 8. 정책 판단이 필요한 경우
- 구조적 선택지가 갈리거나 철학에 영향을 주는 판단이 필요하면 먼저 사용자에게 묻는다.
- 애매한 경우 혼자 밀어붙이지 않는다.

## 9. 프로젝트 방향성
MK5는 답변을 잘하는 챗봇보다, 시간이 지날수록 더 구조적으로 세계와 사용자를 이해하는 그래프 중심 인지 시스템을 목표로 한다.

핵심 방향:
- 기억은 append-only 문자열 로그가 아니다.
- 장기 상태와 판단 근거의 중심은 그래프다.
- prompt-driven persona보다 graph-driven continuity를 지향한다.
- 개념/정체성/시간성도 예외 처리하지 않고 그래프 구조로 다룬다.
