# AGENTS.md

이 문서는 MK5 프로젝트에서 작업하는 모든 에이전트와 작업자가 반드시 따라야 하는 공통 작업 규칙이다.
이 프로젝트의 핵심은 "그럴듯한 답변 생성"이 아니라, 구조적으로 일관된 그래프 기반 기억/의미 모델을 안정적으로 구현하는 것이다.

## 1. 프로젝트 우선 목표

MK5는 사용자의 발화와 대상 정보를 그래프 구조로 다루는 시스템이다.

이 프로젝트에서 가장 중요한 것은 다음 4가지다.

1. 의미를 문자열 휴리스틱에 의존하지 않고 구조적으로 표현할 것
2. 검색, 저장, 연결, 응답이 서로 충돌하지 않도록 데이터 흐름을 일관되게 유지할 것
3. 임시 우회 패치보다 구조적 정합성을 우선할 것
4. "답변이 그럴듯한지"보다 "내부 구조가 정책에 맞는지"를 우선 검증할 것

## 2. 절대 금지 규칙

아래 항목은 명시적으로 금지한다.

### 2.1 문자열 의미론 회귀 금지
다음과 같은 방식으로 의미를 판별하지 말 것.

- kind, type, category 등의 문자열 라벨로 node의 본질을 판별
- note, text, label, summary 등 자유 텍스트를 정규식/키워드로 해석하여 의미를 복원
- 특정 단어 포함 여부로 relation의 성격을 판정
- 응답 문장 형태를 기준으로 저장 정책을 분기

문자열은 표시용 보조 정보일 수는 있지만, 핵심 의미 판정의 기준이 되어서는 안 된다.

### 2.2 kind 부활 금지
node 자체에 종류(kind) 개념을 다시 심지 말 것.
kind 필드를 새로 만들거나, 다른 이름으로 동일 개념을 우회 도입하는 것도 금지한다.

예:
- node_type
- semantic_kind
- inferred_kind
- entity_kind
- hidden kind 역할을 하는 enum/string field

### 2.3 direct access 성공 = search 불필요 라는 단순화 금지
어떤 단어/노드가 이미 존재한다는 이유만으로 search 필요성을 꺼 버리면 안 된다.

특히 다음 상황을 주의할 것.

- 초기에 비어 있는 노드가 생성된 뒤
- 이후 direct access는 성공하지만
- 실제 의미 보강 search는 영원히 생략되는 상태

이런 구조는 금지한다.

### 2.4 임시 패치성 분기 추가 금지
특정 버그를 막기 위해 아래와 같은 임시 분기를 넣지 말 것.

- "이 경우만 예외 처리"
- "이 문자열이면 저장하지 않음"
- "이 문구가 보이면 topic으로 간주"
- "이 답변 패턴이면 retry"

버그의 원인이 구조 문제라면 구조를 수정해야 한다.

## 3. 핵심 구조 원칙

### 3.1 의미는 구조에서 드러나야 한다
의미는 아래 계층에서 드러나야 한다.

1. relation 구조
2. structured data
3. provenance / evidence
4. confidence / status / lifecycle 정보

자유 텍스트 note는 설명 보조일 뿐, 핵심 의미 판단 근거가 아니다.

### 3.2 대상 중심 정보 귀속
정보는 가능하면 "대상"에 귀속되어야 한다.
"사용자 vs 웹" 같은 출처 기준 이분법보다, 어떤 정보가 누구/무엇에 대한 것인지가 우선이다.

### 3.3 검색과 저장은 분리하되 충돌하지 않아야 한다
검색(search)은 의미 보강과 연결 가능성 확장을 위한 과정이다.
저장(store/save)은 구조화된 결과를 그래프에 반영하는 과정이다.

둘은 다르지만, direct access / cached node / placeholder node 때문에 search가 봉쇄되어서는 안 된다.

### 3.4 placeholder는 종착점이 아니라 중간 상태다
비어 있는 노드, 임시 노드, 최소 정보 노드는 최종 상태가 아니다.
후속 search 또는 enrichment로 보강될 수 있어야 한다.

### 3.5 정책 위반보다 미완성이 낫다
정책을 어기면서 기능을 억지로 성립시키지 말 것.
구조적으로 맞는 구현이 아직 어렵다면, 일부 기능이 미완성인 상태가 더 낫다.

## 4. 작업 절차

모든 의미 있는 수정은 아래 순서를 따른다.

### 4.1 먼저 분석
코드를 바로 고치지 말고 먼저 아래를 정리한다.

- 문제 현상
- 재현 경로
- 실제 원인 후보
- 정책 위반 여부
- 영향을 받는 파일/모듈

### 4.2 그 다음 설계
수정 전 아래를 명시한다.

- 어떤 구조 원칙을 회복하려는지
- 어떤 안티패턴을 제거하는지
- 최소 수정인지, 리팩토링인지
- 어떤 부작용 가능성이 있는지

### 4.3 마지막에 코드 수정
코드 수정은 가능한 한 작고 명확해야 한다.
한 번에 너무 많은 파일을 동시에 건드리지 말 것.

## 5. 출력 형식 규칙

작업 결과를 보고할 때는 가능하면 아래 형식을 따른다.

### 5.1 분석 단계
1. 문제 요약
2. 원인 추정
3. 정책 위반 지점
4. 수정 계획

### 5.2 수정 단계
1. 변경 파일 목록
2. 파일별 변경 이유
3. 구조적으로 달라진 점
4. 남아 있는 리스크

### 5.3 검증 단계
1. 어떤 입력으로 재현 가능한지
2. 수정 후 기대 결과
3. 아직 확인 못 한 부분

## 6. 테스트 및 검증 규칙

아래 중 가능한 범위에서 반드시 수행한다.

- 최소 재현 입력으로 전/후 비교
- search 필요 여부가 placeholder/direct access 때문에 막히지 않는지 확인
- 문자열 기반 휴리스틱이 새로 들어오지 않았는지 확인
- node 내부에 kind 역할을 하는 구조가 재도입되지 않았는지 확인
- relation / structured data 중심으로 의미가 표현되는지 확인

## 7. 리뷰 기준

수정안은 아래 질문을 통과해야 한다.

1. 이 변경은 구조를 개선했는가, 아니면 분기를 추가했는가?
2. 의미 판정을 문자열에 다시 의존하게 만들지 않았는가?
3. placeholder node가 영구 빈 노드로 남는 경로를 만들지 않았는가?
4. node에 kind를 우회적으로 부활시키지 않았는가?
5. search / store / apply 흐름이 더 분명해졌는가?

## 8. MK5 특화 주의사항

- "응답 품질이 좋아 보인다"는 이유만으로 구조 문제를 덮지 말 것
- 템플릿 응답, verbalizer, formatter는 그래프/정책 오류를 가리는 수단이 되어선 안 됨
- extraction / normalization / apply 단계는 각각 역할이 달라야 하며, 서로의 책임을 침범하지 말 것
- topic, profile, general, relation, evidence의 저장 경계를 명확히 유지할 것
- 정책 위반이 의심되면 새 기능 추가보다 기존 분기 제거를 먼저 검토할 것

## 9. 작업자의 기본 태도

이 프로젝트에서는 다음 태도를 기본으로 한다.

- 추측보다 근거
- 임시 봉합보다 구조 정리
- 과도한 확신보다 검증 가능한 설명
- 많은 수정보다 정확한 수정

## 10. 작업 금지 예시

다음과 같은 수정은 하지 말 것.

- "지금은 급하니 문자열로만 판별"
- "일단 kind 비슷한 필드 추가"
- "검색이 너무 많으니 existing node면 무조건 search off"
- "특정 응답 패턴이면 저장 안 하도록 예외 처리"
- "note에 의미를 적어 두고 나중에 note 파싱으로 복원"

이런 수정은 단기적으로 편해 보여도 MK5의 구조 목표를 무너뜨린다.
