[2026-04-16 개념 업데이트 메모]

이번 정리에서 새로 확정한 방향은 아래와 같다.

1. 존재 계층은 `instance_of`보다 `subtype_of` 중심으로 본다.
- `지성체 -> 사람 -> 나`
- `지성체 -> AI -> Machi`
- 즉 개별 존재도 상위 개념 아래에서 더 구체화된 하위 개념으로 다룬다.

2. 이름/별칭/호칭은 존재 계층과 분리한다.
- `신재용`, `Jay`, `재용`은 각각 Node가 될 수 있다.
- 이들은 subtype보다는 `same_kind` 계열 Edge로 묶는 편이 맞다.
- 존재 자체와 이름 표상 축을 섞지 않는다.

3. 관계와 상태는 Node가 아니라 Edge가 맡는다.
- 예: `creator_of`, `calls_user_as`, `interacts_with`
- "누구인가"는 Node 축이고, "어떻게 연결되는가"는 Edge 축이다.

4. 시간성은 새 subtype 노드를 만드는 방식보다 이력 역추적으로 처리한다.
- `2026년 3월의 Machi`는 새 존재가 아니다.
- 같은 `Machi` Node를 과거 시점 기준으로 다시 읽는 것이다.
- 이를 위해 Node와 Edge에 생성/업데이트 시점 정보를 남기는 방향이 필요하다.

5. 역추적을 위해 Node / Edge 모두 시점 정보를 가져야 한다.
- Node: `created_at`, `updated_at` 또는 대응 event id
- Edge: `created_at`, `updated_at` 또는 support/conflict/revision 기준 event id
- 이렇게 해야 "지금 상태보다 더 이전이니 더 전으로 가야 한다"는 역추적이 자연스러워진다.

[이번 턴 추가 설계 초안]

이번엔 개념 문서를 한 단계 더 내려서 아래까지 같이 정리했다.

1. 현재 스키마 기준의 시점 정보 활용 원칙
- `nodes.created_at`, `nodes.updated_at`, `nodes.created_from_event_id`
- `edges.created_at`, `edges.updated_at`, `edges.last_supported_at`, `edges.last_conflicted_at`, `edges.created_from_event_id`
- 즉 완전 신설보다, 현재 필드를 "역추적 가능한 기준 정보"로 격상하는 방향이 1차적으로 맞다.

2. identity edge 규칙 초안
- `edge_family`: `concept`, `relation`
- 초기 `connect_type`: `flow`, `neutral`, `opposite`
- 세부 의미는 Edge data 안에 담는다
- 이름/별칭/호칭은 `concept / neutral`
- 존재 계층은 `concept / flow`
- 관계/역할/상태는 `relation / flow` 또는 `relation / neutral`

3. 파이프라인 연결 지점
- `graph_ingest_service.py`: identity anchor / naming cluster의 1차 주입 지점
- `activation_engine.py`: identity continuity와 향후 as-of cutoff 확장 지점
- `thought_engine.py`, `intent_manager.py`, `conclusion_builder.py`: memory_probe와 정체성 추론 연결 지점
- verbalizer: 그래프에서 읽힌 정체성/시간 구조를 표현만 하는 계층

4. 향후 구현 우선순위
- 시점 정보 원칙 확정
- identity anchor 1차 도입
- `concept / flow` 계층 도입
- 이후 as-of 조회 기반 마련

5. connect_type 확장 원칙
- 초기에는 `flow / neutral / opposite`의 작은 집합으로 시작
- 기존 타입으로 설명되지 않는 연결은 곧바로 정식 추가하지 않음
- 우선 Edge data 안에 `proposed_connect_type`, `proposal_reason`, `proposal_confidence` 같은 후보 정보로 누적
- 반복 누적 후 정식 `connect_type`으로 승격

[schema / edge / repository 정리]

이번엔 위 설계를 실제 코드 초안 수준으로 더 내려서 다음까지 정리했다.

1. `schema.sql`
- `edges` 테이블에 `edge_family`, `connect_type`를 추가하는 1차 마이그레이션 방향
- 기존 `edge_type`는 곧바로 삭제하지 않고 호환용 브리지 필드로 잠시 유지
- 기존 pair index는 장기적으로 `(source, target, edge_family, connect_type, is_active)` 쪽으로 이동하는 것이 맞음

2. `Edge` dataclass
- `edge_family: concept | relation`
- `connect_type: flow | neutral | opposite`
- `edge_type`는 1차 호환용으로 남길 수 있음
- 세부 의미는 `relation_detail`에 유지

3. repository dedupe 기준
- 현재의 `source + target + edge_type` 기준은 한계가 큼
- 1차적으로는 `source + target + edge_family + connect_type (+ connect_semantics)` 기준으로 바꾸는 방향이 맞음
- 같은 두 Node 사이에서도 family/type/semantics가 다르면 복수 Edge 공존 허용

4. 마이그레이션 원칙
- 한 번에 완전 교체하지 않음
- `edge_type` 제거보다 `edge_family` / `connect_type` 추가가 먼저
- 신규 생성 코드부터 새 구조를 쓰고, 기존 호출부는 점진적으로 교체
