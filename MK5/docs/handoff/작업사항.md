# 작업사항

## 현재 어디까지 되었는가

### 1. 저장 계층
- `storage/schema.sql` 기준으로 `chat_messages / nodes / edges / graph_events / node_pointers` 스키마가 있다.
- SQLite 구현체가 붙어 있고, `SqliteUnitOfWork`로 한 트랜잭션 안에서 접근 가능하다.
- repository 계층은 단순 CRUD가 아니라 `address_hash 조회`, `active relation 조회`, `revision candidate 조회`, `support/conflict 누적` 같은 그래프 엔진 중심 질의 기준으로 잡혀 있다.

### 2. 입력 인지 및 그래프 ingest
- `core/cognition/input_segmenter.py`에서 입력을 문장 경계와 의미블록으로 나눈다.
- 현재 정책은 **원문/문장 = provenance**, **영구 그래프 = 재사용 가능한 의미블록**이다.
- `core/update/graph_ingest_service.py`에서 다음 흐름이 작동한다.
  - chat_message 저장
  - graph_event 기록
  - meaning block 추출
  - 기존 node 재사용 또는 새 node 생성
  - 같은 입력 내부 공출현 기반 약한 edge 생성/강화
  - partial reuse pointer 생성

### 3. 국부 활성화
- `core/activation/activation_engine.py`와 `thought_view_builder.py`가 있다.
- 현재 입력 기준으로 seed block -> seed node -> local node / local edge / pointer를 모아 `ThoughtView`를 만든다.
- 즉, 전체 그래프를 다 쓰는 것이 아니라 **현재 사고용 국부 뷰**가 만들어진다.

### 4. thinking 계층
- `core/thinking/contradiction_detector.py`
- `core/thinking/trust_manager.py`
- `core/thinking/structure_revision_service.py`
- `core/thinking/thought_engine.py`

현재는 아래 흐름이 들어가 있다.
- 구조 충돌 감지
- trust 하락
- revision candidate 검토
- 필요 시 edge 비활성화

즉, **기본은 구조 보존 / 반복 충돌 시 신뢰도 하락 / 바닥 도달 시 구조 교체 후보화** 방향이 이미 코드에 반영되어 있다.

### 5. 설명형 core conclusion
- `core/entities/conclusion.py`에 설명형 conclusion dataclass가 있다.
- `activated_concepts`는 **node id 참조 목록**이다.
- `key_relations`는 **edge id 참조 목록**이다.
- `thought_engine`은 이제 `ThoughtView -> CoreConclusion`까지 만든다.

### 6. verbalization
- `core/verbalization/template_verbalizer.py`가 있다.
- 현재는 **설명형 core conclusion을 템플릿 기반 자연어 응답으로 바꾸는 단계**까지 연결되어 있다.
- 아직 로컬 LLM verbalizer는 붙어 있지 않다.

### 7. chat end-to-end
- `app/chat_pipeline.py` 기준으로
  - ingest
  - activation
  - thinking
  - verbalization
  이 한 줄이 닫혀 있다.
- `app/api.py`도 최소 Flask 라우트 셸로 연결되어 있다.
- `app/static/chat.html`, `chat.css`, `chat.js`를 통해 기본 UI 사용이 가능하다.

### 8. 테스트
현재 통과 확인된 테스트는 아래다.
- `tests/unit/test_sqlite_repository_smoke.py`
- `tests/integration/test_chat_graph_pipeline.py`
- `tests/integration/test_activation_engine_pipeline.py`
- `tests/integration/test_thinking_revision_pipeline.py`
- `tests/integration/test_end_to_end_chat_pipeline.py`

## 아직 안 된 것 / 추가적으로 필요한 것

### 1. Flask 실서버 실행 검증
- 이 작업 환경에서는 Flask 설치가 없어 실제 서버 기동을 검증하지 못했다.
- 즉, `python run.py`가 네 로컬에서 실제로 정상 동작하는지는 로컬 검증이 필요하다.

### 2. requirements 정리
- 아직 `requirements.txt`가 없다.
- 최소한 `flask`는 명시되어야 하고, 이후 LLM verbalizer나 추가 도구를 붙일 때 의존성을 정리해야 한다.

### 3. debug UI 구조화
- 지금은 debug 정보가 시스템 메시지 텍스트로 출력된다.
- 의미블록 / seed node / local edge / conflict / trust update / revision decision / core conclusion을 별도 패널로 분리할 필요가 있다.

### 4. 행동형 파생 출력
- 현재 본체는 설명형 conclusion이고, 이건 맞다.
- 하지만 실사용성을 위해 `suggested_actions`, `next_steps` 같은 **행동형 derived plan** 층이 추가로 필요하다.
- 단, 본체는 계속 설명형이어야 한다.

### 5. local LLM verbalizer
- 현재는 template verbalizer만 있다.
- 이후 작은 로컬 LLM을 **언어화 전용**으로 붙이는 작업이 필요하다.
- 조건: 의미 추가 금지, 사고 금지, 결론 구조 유지.

### 6. meaning block 정교화
- 현재 의미블록 추출은 시작점 수준이다.
- 명사구 / 짧은 판단구 / 관계구 / 정정구 중심인데, 실제 대화 품질을 높이려면 점진적 정교화가 필요하다.
- 단, 이건 큰 흐름 완성 후에 다듬는 것이 맞다.

### 7. graph revision 고도화
- 현재 revision은 보수적 규칙 기반에 가깝다.
- 장기적으로는
  - revision 히스토리 축적
  - edge/node별 안정성 평가
  - 구조 교체 후 후속 영향 관리
  같은 층이 필요하다.

### 8. 세션/프로젝트 처리 재정리
- 현재 MK5 철학상 project/profile/chat 분리는 제거되었고, 전부 chat 흐름으로 묶는다.
- 다만 UI에는 아직 project 셀렉터가 남아 있다.
- 이걸 완전히 제거할지, 아니면 “UI상의 grouping label” 정도로 남길지는 정리 필요.

### 9. 파일 입력 / zip 처리
- 현재 `/chat`은 파일 메타 정도만 처리하고 있다.
- MK4처럼 artifact/project ingest를 다시 붙일지, 아니면 MK5에서는 chat 입력 기반으로 더 단순화할지 결정 필요.
- 이건 철학/정책 분기점이 될 수 있으므로, 진행 전 확인 필요.

## 참고
