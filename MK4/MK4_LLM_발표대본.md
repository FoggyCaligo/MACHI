슬라이드 1. 제목

MK4 - LLM 기반 로컬 개인화 기억 시스템

“안녕하세요. 이번 발표에서는 제가 만들고 있는 MK4라는 프로젝트를 소개드리겠습니다.
MK4는 완전히 독립된 단일 프로젝트라기보다, 제가 장기적으로 구상하고 있는 MACHI 프로젝트의 일부입니다.
MACHI는 한 사람을 장기적으로 보조하는 로컬 Cognitive Partner, 더 크게는 개인 전용 인지 시스템을 목표로 하는 상위 프로젝트입니다.
그리고 그 안에서 MK4는 특히 기억 계층, 즉 사용자를 어떻게 기억하고, 틀리면 어떻게 고치고, 다음 대화에서 어떻게 다시 활용할 것인가를 다루는 단계입니다.
그래서 이번 발표에서는 단순히 챗봇 하나를 만든 이야기가 아니라, MACHI라는 더 큰 방향 안에서 왜 MK4가 필요했고, 지금 어디까지 구현되었는지를 설명드리려 합니다.”

슬라이드 2. 목차

프로젝트 배경과 목적 / 핵심 개념과 철학 / 시스템 구조 / 현재 구현 상태 / 트러블슈팅 / 향후 확장과 MK5

“발표는 크게 여섯 부분으로 구성했습니다.
먼저 왜 이런 프로젝트를 시작하게 되었는지 배경과 목적을 말씀드리고, 그 다음 MK4의 핵심 개념과 철학을 설명드리겠습니다.
이후에는 시스템 구조와 현재 구현 상태를 보여드리고, 실제 개발 과정에서 맞닥뜨린 구조적 문제들을 트러블슈팅 형태로 정리하겠습니다.
마지막으로 아직 남아 있는 과제와, 다음 단계인 MK5 방향까지 말씀드리겠습니다.”

슬라이드 3. 왜 기존 대화형 LLM만으로는 부족했는가

Problem Definition

“먼저 문제 정의입니다.
기존 대화형 LLM은 분명히 매우 유용합니다. 하지만 제가 느낀 한계는, 대화 로그는 남아도 이해 구조는 남지 않는다는 점이었습니다.
예를 들어 모델이 나를 어느 정도 이해하고 있는 것처럼 보여도, 그 이해가 왜 형성되었는지, 무엇을 수정해야 하는지, 어떤 기억이 지금 응답에 영향을 주는지는 분리되어 드러나지 않는 경우가 많습니다.
또 플랫폼이나 계정이 바뀌면 그동안 쌓인 맥락이 쉽게 끊기고, 저장된 기억이 틀렸더라도 correction으로 다시 깨고 재구성하는 구조는 상대적으로 약합니다.
즉 기존 LLM은 대화를 이어갈 수는 있어도, 사용자를 장기적으로 구조적으로 이해하는 시스템이라고 보기에는 부족하다고 느꼈습니다.”

슬라이드 4. 프로젝트 목적

Origin Story

“이 문제의식은 개인적인 경험에서 시작됐습니다.
이전에 퇴사하면서 회사 계정으로 쓰던 GPT에 더 이상 접근할 수 없게 됐습니다.
그래서 그 GPT에게 ‘나를 어떻게 이해하고 있었는지 정리해 달라’고 요청해서 문서로 옮긴 뒤, 새 개인 계정에 넣어봤습니다.
그런데 제가 기대했던 건 ‘이해의 이전’이었지만, 실제로는 모델이 그 문서를 그대로 이어받는 게 아니라 다시 검증하고 다시 해석하더라고요.
그때 느낀 게, 기억의 이전이 곧 이해의 이전은 아니라는 점이었습니다.
그리고 더 나아가, 기억이 계정이나 플랫폼에 묶여 있으면 사용자는 자기 장기 맥락을 직접 소유하지 못한다는 문제의식으로 이어졌습니다.
그래서 MK4는 기억을 플랫폼의 부속물이 아니라, 사용자가 보유하고 통제할 수 있는 구조로 만들자는 목적에서 시작됐습니다.”

슬라이드 5. MK4의 핵심 아이디어

Evidence-first / Correction-first / Revisable Memory

“이제 MK4의 핵심 아이디어를 설명드리겠습니다.
여기서 핵심은 세 가지입니다.
첫 번째는 evidence-first입니다. 채팅, 첨부 텍스트, ZIP artifact 같은 입력을 곧바로 사용자 profile로 확정하지 않고, 먼저 evidence로 저장해 검증 가능한 재료로 다룹니다.
두 번째는 correction-first입니다. 잘못 이해한 것은 나중에 correction으로 깨고 다시 세울 수 있어야 합니다.
세 번째는 revisable memory입니다. confirmed 상태조차 절대 진실이 아니라, 현재 시점의 가설적 사용자상으로 취급합니다.
즉 MK4는 ‘정답을 빨리 말하는 모델’보다는, ‘사용자를 더 일관되게 이해하고 틀리면 다시 고칠 수 있는 로컬 기억 구조’를 지향합니다.”

슬라이드 6. 기존 LLM 활용과 MK4의 차이

Role of the Model

“여기서 중요한 건 MK4가 LLM을 버리는 프로젝트는 아니라는 점입니다.
오히려 LLM은 evidence 추출, 응답 생성, retrieval 보조 같은 역할을 맡습니다.
다만 일반적인 챗봇이 프롬프트와 대화 맥락 중심으로 그때그때 답을 만드는 구조라면, MK4는 memory 구조 자체를 중심에 둡니다.
즉 LLM은 답변 엔진으로만 존재하는 것이 아니라, topic, correction, promotion, rebuild가 들어간 revisable memory substrate의 일부로 들어갑니다.
정리하면, 일반적인 챗봇은 ‘이번 답변’을 잘 만드는 데 초점이 있고, MK4는 ‘시간이 갈수록 더 일관된 사용자 모델’을 만드는 데 더 큰 초점을 둡니다.”

슬라이드 7. 시스템 전체 구조

입력 채널 → 공통 처리 계층 → 저장 / 재사용

“이제 전체 구조를 보겠습니다.
입력 채널은 여러 개입니다. 일반 채팅, 첨부 텍스트, ZIP artifact, project 질문, correction 입력이 각각 따로 들어옵니다.
하지만 중요한 건, 이 입력들이 각자 다른 시스템처럼 따로 놀면 안 된다는 점입니다.
그래서 중간에 공통 처리 계층을 두고, evidence 추출, normalization, topic routing, memory apply, promotion과 rebuild, response retrieval을 거치도록 했습니다.
그리고 그 결과가 topics, profiles, corrections, episodes, summaries, project chunks 같은 저장 계층으로 들어가고, 다음 턴에서 재사용됩니다.
즉 MK4의 핵심 구조는, 입력 채널은 달라도 기억으로 들어갈 때는 같은 의미론으로 수렴하게 하는 것입니다.”

슬라이드 8. 핵심 memory pipeline

Input → Evidence → Reuse

“이 구조를 더 단순하게 보면, 입력이 들어오고, evidence를 추출하고, topic을 정하고, tier를 부여하고, promotion과 rebuild를 거친 뒤, 다음 턴에서 다시 재사용하는 흐름입니다.
여기서 tier는 세 가지입니다.
general은 저장 가치는 있지만 기본 응답에는 주입하지 않는 층이고, candidate는 승격 후보, confirmed는 현재 사용자 모델에 반영되는 가설적 층입니다.
즉 MK4는 대화를 그대로 저장하는 것이 아니라, 해석 가능한 기억 계층으로 변환한 뒤 다시 사용하는 시스템이라고 보시면 됩니다.
이 차이가 결국 ‘대화 로그 저장’과 ‘개인화 기억 시스템’의 차이라고 생각합니다.”

슬라이드 9. 현재 구현 상태

What is already built

“현재 구현 상태를 말씀드리면, 아이디어 수준에 머무는 상태는 이미 지나왔습니다.
일반 채팅, 첨부 텍스트, ZIP artifact, project 질문의 경로 분리가 되어 있고, TopicRouter 기반의 active topic 유지와 attach, create 흐름도 들어가 있습니다.
또 EvidenceNormalizationService, MemoryIngressService, MemoryApplyService를 중심으로 공통 처리 구조를 만들었고, general, candidate, confirmed 분기와 promotion 구조, correction target kind 분류, semantic retrieval, response continuation, project chunk retrieval도 1차 반영했습니다.
즉 지금 단계는 ‘개념을 설명하는 단계’라기보다, 개념적 구조가 실제 코드로 꽤 내려온 단계입니다.
그래서 이제부터는 구조를 새로 만드는 것보다, 일반 채팅 품질과 재사용 품질을 더 끌어올리는 단계에 가깝다고 보고 있습니다.”

슬라이드 10. 트러블슈팅 03

입력 채널별로 기억 의미론이 갈라지던 문제

“이제부터는 트러블슈팅입니다.
첫 번째 큰 문제는 입력 채널마다 기억의 의미론이 갈라지던 점이었습니다.
일반 채팅은 임시 업데이트처럼 취급되고, 첨부 텍스트는 preview, 프로젝트 ZIP은 project 분석 전용처럼 따로 움직이기 쉬웠습니다.
이렇게 되면 같은 정보라도 어떤 채널로 들어왔는지에 따라 버려지거나 과하게 확정되는 문제가 생깁니다.
그래서 해결 방향은, 입력 채널은 달라도 기억으로 들어갈 때는 general, candidate, confirmed라는 공통 의미론으로 수렴하도록 구조를 재정렬하는 것이었습니다.
이때 얻은 핵심 교훈은, 입력 채널 분리는 기술적 편의일 뿐, 의미론 분리의 근거가 될 수 없다는 점입니다.”

슬라이드 11. 트러블슈팅 04

preview-first가 실제 업데이트를 가로막던 문제

“두 번째 문제는 preview-first 구조였습니다.
첨부 텍스트를 넣으면 겉으로는 그럴듯하게 분석해주는 답변이 나오지만, 실제 memory에는 거의 남지 않는 상태가 한동안 있었습니다.
즉 사용자는 이해받은 것처럼 느끼는데, 시스템은 실제로는 배우지 못하는 구조였던 겁니다.
그래서 이 부분을 update-first로 바꿨습니다.
source를 저장하고, relevant passage를 고르고, evidence를 추출하고, store와 sync를 먼저 수행한 뒤, 그 결과를 바탕으로 답변을 생성하도록 바꿨습니다.
핵심 교훈은, MK4에서 첨부 기반 프로필 요청은 설명 서비스가 아니라 실제 memory update 이벤트여야 한다는 점이었습니다.”

슬라이드 12. 트러블슈팅 08

저장된 memory를 너무 쉽게 믿던 문제

“세 번째 문제는 저장된 memory를 너무 쉽게 믿었던 점입니다.
처음에는 저장된 내용을 거의 참인 정보처럼 바로 주입하다 보니, 오래된 오염이나 잘못된 해석이 현재 응답을 덮어쓰는 일이 생겼습니다.
예를 들어 예전의 잘못된 응답 스타일이 계속 재주입되거나, correction보다 과거 profile이 더 강하게 작동하는 문제들이 있었습니다.
그래서 저장된 memory를 ‘신뢰 가능한 사실 저장소’가 아니라, 검증과 필터링이 필요한 가설 저장소로 취급하는 방향으로 바꿨습니다.
즉 저장된 memory도 다시 검토 대상이고, correction이 있으면 이전 profile보다 우선하며, 주입 전 최소한의 위생 필터가 필요하다는 원칙으로 재정렬했습니다.
이 문제를 통해 얻은 핵심 교훈은, 기억을 무조건 믿는 순간 시스템은 학습 시스템이 아니라 과거 오류의 증폭기가 된다는 점입니다.”

슬라이드 13. 트러블슈팅 09

topic 구조가 약해서 대화 의미 축이 흔들리던 문제

“네 번째 문제는 topic 구조였습니다.
초기에는 topic이 단순 문자열 라벨처럼 취급되거나, 애매하면 general로 뭉개지는 경우가 많았습니다.
그러다 보니 서로 다른 의미 축이 같은 bucket에 쌓이고, correction이 어느 범위에 적용되는지도 모호해졌습니다.
그래서 topic을 단순 꼬리표가 아니라 의미 축 객체로 다루도록 구조를 바꿨습니다.
active topic을 유지할지, 기존 topic에 attach할지, 새 topic을 만들지를 판단하도록 했고, topic과 project_id도 분리했습니다.
즉 topic은 의미 축이고, project_id는 출처 축이라는 원칙을 세운 겁니다.
핵심 교훈은, topic은 분류용 문자열이 아니라 기억의 적용 범위를 정하는 핵심 좌표축이라는 점입니다.”

슬라이드 14. 트러블슈팅 10

일반 / 후보 / 확정 경계가 채널마다 달라지던 문제

“다섯 번째 문제는 candidate와 confirmed의 의미가 채널마다 다르게 해석될 위험이 있었다는 점입니다.
채팅에서는 일반 정보로만 남고, 첨부에서는 후보가 되고, project artifact에서는 거의 확정처럼 작동한다면 시스템 전체 일관성이 무너집니다.
그래서 tier 의미를 가능한 한 통일하는 쪽으로 재정렬했습니다.
general은 저장 가치는 있지만 기본 주입 대상은 아닌 층, candidate는 승격 가능한 evidence 집합, confirmed는 현재 사용자 모델에 반영되는 층이라는 공통 정의를 유지하도록 했습니다.
즉 채널별 편의 규칙이 아니라, 시스템 전체에서 같은 뜻을 갖는 기억 계층을 만들려고 한 것입니다.”

슬라이드 15. 트러블슈팅 13

텍스트 첨부 답변이 템플릿처럼 느껴지던 문제

“여섯 번째 문제는 첨부 기반 답변이 실제 대화처럼 생성되지 않고, 너무 템플릿처럼 보이던 문제였습니다.
이렇게 되면 답변이 기계적으로 느껴지고, 바뀐 memory 상태를 충분히 반영하지 못합니다.
그래서 최종 답변도 모델이 생성하도록 바꾸고, 템플릿은 설명용 보조 구조로만 축소했습니다.
즉 MK4는 정해진 문구를 출력하는 시스템이 아니라, 갱신된 근거를 바탕으로 그때그때 말할 수 있는 시스템이어야 한다는 방향으로 수정했습니다.”

슬라이드 16. 트러블슈팅 14

사용자 질문을 그대로 복사하는 답변이 나오던 문제

“일곱 번째 문제는 echo 문제였습니다.
특히 첨부 기반 경로에서 사용자의 질문을 거의 그대로 되받아 말하는 현상이 나타났습니다.
이건 이해가 아니라 입력 복제에 가깝고, 작은 모델일수록 더 자주 나타났습니다.
그래서 질문 원문 전체를 그대로 넘기기보다, 질문의 요지를 요약해서 답변 입력으로 전달하는 방식으로 바꿨습니다.
또 사용자 문장을 그대로 베껴 답하지 않고, 인칭 표현을 기계적으로 따라 쓰지 않도록 원칙을 분명히 했습니다.
핵심 교훈은, 질문 복사는 이해의 신호가 아니라 입력 설계 실패의 신호라는 점입니다.”

슬라이드 17. 트러블슈팅 15

인칭과 말투가 흔들리던 문제

“여덟 번째 문제는 화자 일관성 문제였습니다.
첨부 텍스트나 프로필 경로에서 나와 너의 기준이 뒤섞이거나, 존댓말과 반말이 섞이는 일이 반복됐습니다.
표면적으로는 스타일 문제처럼 보이지만, 실제로는 같은 시스템이 계속 말하고 있다는 신뢰를 깨뜨리는 문제였습니다.
그래서 사용자 발화의 ‘나/내’는 사용자 기준, ‘너/네’는 AI 기준으로 고정하고, 답변은 존댓말만 유지하도록 별도 경로에도 강하게 적용했습니다.
핵심 교훈은, 말투 문제는 단순 문체 문제가 아니라 동일한 인지 주체처럼 보이느냐의 문제라는 점입니다.”

슬라이드 18. 트러블슈팅 20

Correction target kind 분류 체계 도입

“아홉 번째는 correction의 종류를 더 세분화한 문제입니다.
초기에는 correction을 거의 한 종류처럼 다뤘는데, 실제로는 무엇을 정정하는지에 따라 영향 범위가 완전히 달랐습니다.
예를 들어 사용자 모델 자체를 정정하는 경우와, AI 설명의 사실 오류를 정정하는 경우, 그리고 답변 방식만 수정하는 경우는 각각 다르게 처리돼야 합니다.
그래서 correction을 profile, topic_fact, response_behavior 세 종류로 나눴습니다.
profile은 사용자 모델 자체 정정, topic_fact는 설명이나 정보 정정, response_behavior는 답변 길이와 톤 같은 스타일 정정입니다.
이렇게 나누면서 profile 안전성과 향후 확장성이 모두 좋아졌습니다.”

슬라이드 19. 트러블슈팅 21

문제 의존성을 제거하지 않고 Python 상한으로만 막으려던 문제

“열 번째는 기술 선택과 관련된 문제였습니다.
특정 외부 라이브러리가 Python 3.14 환경에서 호환성 불안을 만들고 있었는데, 처음에는 interpreter 상한으로만 막는 방향으로 굳어질 위험이 있었습니다.
그런데 그렇게 하면 문제 원인은 의존성인데, 프로젝트 전체가 버전 제약을 떠안게 됩니다.
그래서 방향을 바꿔, 문제 의존성을 직접 제거하는 쪽으로 갔습니다.
구체적으로는 chunker 관련 외부 의존성을 제거하고 내부 구현으로 전환했습니다.
이 과정에서 얻은 교훈은, 장기 프로젝트에서는 문제 라이브러리 때문에 interpreter를 묶는 것보다, 가능한 한 구조 자율성을 높이는 것이 더 건강하다는 점입니다.”

슬라이드 20. 트러블슈팅 2

입력 채널별로 기억 의미론이 갈라지던 문제 요약

“이 슬라이드는 앞에서 설명드린 핵심을 더 짧게 다시 요약한 장입니다.
결국 MK4의 구조 수정은 여러 기능을 따로 붙이는 방향이 아니라, 서로 다른 입력을 하나의 memory substrate 위로 모으는 방향으로 정리되고 있습니다.
즉 채팅, 첨부, project artifact가 다르게 보이더라도, 기억 계층 안에서는 같은 언어를 써야 한다는 점이 핵심입니다.”

슬라이드 21. 트러블슈팅 3

preview-first가 실제 업데이트를 가로막던 문제 요약

“이 슬라이드도 같은 맥락의 요약입니다.
겉으로 잘 설명하는 것과 실제로 다음 대화에 반영될 수 있는 기억이 남는 것은 다릅니다.
그래서 MK4는 preview-first가 아니라 update-first를 택했습니다.
사용자에게 그럴듯해 보이는 일회성 해석보다, 실제로 다음 턴에 남는 기억 구조를 우선한 것입니다.”

슬라이드 22. 향후 확장 계획

What is not finished yet

“이제 아직 남아 있는 과제를 말씀드리겠습니다.
가장 큰 과제는 일반 채팅 품질을 첨부나 artifact 수준까지 끌어올리는 것입니다.
현재는 대략 75에서 80퍼센트 수준으로 보고 있고, general에서 candidate로의 승격, 재사용 안정화, 짧음이나 반복 같은 문제를 계속 잡아야 합니다.
그 외에도 recent source 관리와 active topic 정리를 위한 source lookup layer, correction의 topic 수준 확장, merge와 split, rebuild 기준, correction 기반 demote 로직 같은 것들이 남아 있습니다.
또 응답 시간, 승격 빈도, attach 정확도 같은 품질 메트릭을 추적해서 개선 루프를 더 정교하게 만드는 작업도 필요합니다.
즉 구조 골격은 어느 정도 세워졌지만, 실사용 품질을 더 밀어 올리는 단계가 아직 남아 있습니다.”

슬라이드 23. 향후 발전 방향 · MK5

Next Stage

“마지막으로 MK5 방향입니다.
여기서 다시 MACHI 맥락으로 돌아가 보면, MK4는 memory substrate를 구현하는 단계이고, MK5는 그 위에 더 넓은 cognition loop를 얹는 단계로 생각하고 있습니다.
즉 기억만 저장하고 재사용하는 수준을 넘어, source lookup, reasoning과 planning, 그리고 사용자 ownership과 governance를 더 강하게 연결하려고 합니다.
조금 더 크게 보면, MK5는 단순히 ‘더 큰 모델’을 쓰는 방향이 아니라, 인지, 사고, 결론, 언어화가 분리된 구조를 더 강하게 지향하는 단계라고 볼 수 있습니다.
그래서 MK4가 기억 중심의 기반 공사였다면, MK5는 그 기반 위에 더 넓은 인지 구조를 얹는 다음 단계라고 보시면 됩니다.”

슬라이드 24. 정리

Closing

“정리하겠습니다.
MACHI는 장기적으로 사용자를 구조적으로 이해하고 보조하는 로컬 인지 시스템을 지향하는 상위 프로젝트입니다.
그리고 그 안에서 MK4는, 사용자를 어떻게 기억하고, 틀리면 어떻게 고치고, 다음 대화에서 어떻게 다시 활용할지를 다루는 memory substrate 구현 단계입니다.
즉 MK4는 단순히 ‘나를 잘 기억하는 챗봇’을 만들려는 프로젝트가 아니라, 기억을 evidence로 만들고, correction으로 다시 고칠 수 있게 하며, 계정이나 플랫폼이 바뀌어도 가져갈 수 있는 로컬 기억 구조를 만들려는 시도입니다.
또 개인화를 프롬프트가 아니라 memory가 담당하게 하고, LLM을 단순 답변 엔진이 아니라 cognition substrate의 일부로 재배치하려는 프로젝트이기도 합니다.
아직 남은 과제는 많지만, 지금 단계에서는 개념적 memory 구조를 실제 시스템으로 옮기는 데까지는 꽤 왔다고 생각합니다.
이상으로 발표를 마치겠습니다. 감사합니다.”
