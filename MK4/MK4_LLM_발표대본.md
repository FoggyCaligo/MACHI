슬라이드 3. 왜 기존 대화형 LLM만으로는 부족했는가

Problem Definition

“이 프로젝트의 출발점은 기존 대화형 LLM의 한계였습니다.
LLM은 분명히 대화를 잘 이어가고, 맥락도 어느 정도 기억하는 것처럼 보입니다.
하지만 제가 느낀 문제는, 대화 기록은 남아도 이해 구조는 남지 않는 경우가 많다는 점이었습니다.
예를 들어 모델이 저를 어느 정도 이해하고 있는 것처럼 보여도, 왜 그렇게 이해했는지, 어떤 기억이 지금 답변에 영향을 주는지, 무엇이 틀렸다면 어떻게 수정해야 하는지는 따로 보이지 않습니다.
또 플랫폼이나 계정이 바뀌면 그동안 쌓인 맥락이 쉽게 끊기고, 저장된 기억이 틀렸더라도 다시 깨고 고치는 구조는 상대적으로 약합니다.
즉 기존 LLM은 대화를 이어갈 수는 있어도, 사용자를 장기적으로 구조적으로 이해하는 시스템이라고 보기에는 부족하다고 느꼈습니다.”

슬라이드 4. 프로젝트 목적

Origin Story

“이 문제의식은 개인적인 경험에서 시작됐습니다.
이전에 퇴사하면서 회사 계정으로 쓰던 GPT에 더 이상 접근할 수 없게 됐습니다.
그래서 그 GPT에게 ‘나를 어떻게 이해하고 있었는지 정리해 달라’고 요청해서 문서로 옮긴 뒤, 새 개인 계정에 넣어봤습니다.
그런데 제가 기대했던 건 ‘이해의 이전’이었지만, 실제로는 모델이 그 문서를 그대로 이어받는 것이 아니라 다시 검증하고 다시 해석하더라고요.
그때 느낀 것이, 기억의 이전이 곧 이해의 이전은 아니다라는 점이었습니다.
그리고 더 나아가, 기억이 계정이나 플랫폼에 묶여 있으면 사용자는 자기 장기 맥락을 직접 소유하지 못한다는 문제의식으로 이어졌습니다.
그래서 MK4는 기억을 플랫폼의 부속물이 아니라, 사용자가 보유하고 통제할 수 있는 구조로 만들자는 목적에서 시작됐습니다.”

슬라이드 5. MK4의 핵심 아이디어

Evidence-first / Correction-first / Revisable Memory

“그래서 나온 핵심 아이디어가 세 가지입니다.
첫 번째는 evidence-first입니다.
채팅, 첨부 텍스트, ZIP artifact 같은 입력을 곧바로 ‘이게 바로 이 사용자의 성향이다’라고 확정하지 않고, 먼저 근거, 즉 evidence로 저장합니다.

두 번째는 correction-first입니다.
사람을 이해하다 보면 틀릴 수도 있기 때문에, 한 번 저장한 이해를 그대로 믿는 것이 아니라 나중에 correction으로 깨고 다시 세울 수 있어야 합니다.

세 번째는 revisable memory입니다.
현재 맞다고 보고 있는 이해조차도 절대 진실이 아니라, 현재 시점의 가장 그럴듯한 가설로 다룹니다.

즉 MK4는 ‘정답을 빨리 말하는 모델’이라기보다, ‘사용자를 더 일관되게 이해하고, 틀리면 다시 고칠 수 있는 로컬 기억 구조’를 목표로 하고 있습니다.”

슬라이드 6. 기존 LLM 활용과 MK4의 차이

Role of the Model

“여기서 중요한 건 MK4가 LLM을 버리는 프로젝트는 아니라는 점입니다.
오히려 LLM은 evidence를 뽑아내거나, 답변을 생성하거나, 관련 기억을 찾는 데 도움을 주는 역할을 합니다.
다만 일반적인 챗봇이 프롬프트와 최근 대화 맥락을 중심으로 그때그때 답을 만드는 구조라면, MK4는 기억 구조 자체를 중심에 둡니다.
즉 LLM은 답변 엔진이지만, 동시에 더 큰 기억 시스템의 일부로 들어갑니다.
정리하면, 일반적인 챗봇은 ‘이번 답변’을 잘 만드는 데 초점이 있고, MK4는 ‘시간이 갈수록 더 일관된 사용자 모델’을 만드는 데 더 큰 초점이 있습니다.”

슬라이드 7. 시스템 전체 구조

입력 채널 → 공통 처리 계층 → 저장 / 재사용

“전체 구조를 보면, 입력은 여러 방식으로 들어옵니다.
일반 채팅도 있고, 첨부 텍스트도 있고, ZIP 프로젝트 파일도 있고, 정정 입력도 있습니다.
하지만 중요한 점은, 이 입력들이 들어오는 형태가 다르더라도 기억으로 들어갈 때는 비슷한 절차를 거쳐야 한다는 것입니다.

그래서 중간에 공통 처리 계층을 두었습니다.
입력이 들어오면 먼저 근거 형태로 정리하고, 그다음 어떤 주제에 관한 이야기인지 판단하고, 실제 기억에 반영한 뒤, 필요할 때 다시 꺼내 써서 다음 답변에 활용합니다.

즉 입력은 여러 갈래지만, 기억으로 들어가고 다시 재사용되는 과정은 하나의 구조로 통일하려고 했습니다.”

슬라이드 8. 핵심 memory pipeline

Input → Evidence → Reuse

“이 과정을 조금 더 단순하게 보면 이렇습니다.
입력이 들어오면 먼저 evidence를 추출합니다.
그리고 이 입력이 지금 하던 이야기의 연장인지, 이전에 다루던 관련 주제인지, 아니면 새 주제인지를 판단합니다.
그다음 그 정보를 어느 정도 무게로 저장할지 정하고, 필요하면 반복성과 신뢰도를 바탕으로 더 중요한 기억으로 올립니다.
마지막으로 다음 턴에서 현재 주제에 맞는 기억을 다시 불러와 답변에 재사용합니다.

여기서 기억은 크게 세 층으로 봅니다.
하나는 일단 저장은 하지만 아직 기본 답변에는 바로 쓰지 않는 층,
둘째는 나중에 더 중요한 기억으로 올라갈 수 있는 후보 층,
셋째는 현재 사용자 모델에 실제로 반영되는 층입니다.

즉 MK4는 대화를 그냥 쌓아두는 시스템이 아니라, 해석 가능한 기억 계층으로 바꿔서 다시 사용하는 시스템입니다.”

슬라이드 9. 현재 구현 상태

What is already built

“현재 구현 상태를 말씀드리면, 아이디어 수준에만 머무는 상태는 이미 지나왔습니다.
일반 채팅, 첨부 텍스트, ZIP artifact, project 질문처럼 입력 경로를 나눠서 처리할 수 있게 되어 있고,
새 입력이 들어왔을 때 이게 지금까지 하던 이야기의 연장인지, 예전에 다루던 관련 주제로 다시 붙는 것인지, 아니면 완전히 새로운 주제인지를 구분하는 구조도 들어가 있습니다.

또 입력 종류가 달라도 결국 같은 처리 흐름을 거치게 했습니다.
즉 채팅이든 문서든 프로젝트 파일이든, 먼저 근거 형태로 바꾸고, 형식을 정리하고, 그다음 실제 기억에 반영하는 공통 파이프라인을 만들었습니다.

그 위에 기억의 단계 구분, 승격 구조, correction 종류 분류, 의미 기반 retrieval, 응답 이어쓰기, 프로젝트 파일 검색 같은 기능들도 1차 반영했습니다.
즉 지금 단계는 개념 설명에 머무는 것이 아니라, 개념적 구조가 실제 코드로 꽤 내려온 단계라고 보시면 됩니다.”

슬라이드 10. 트러블슈팅 03

입력 채널별로 기억 의미론이 갈라지던 문제

“이제부터는 개발하면서 실제로 부딪힌 문제들을 말씀드리겠습니다.
첫 번째 큰 문제는, 입력 채널마다 기억의 의미가 달라지려는 경향이 있었다는 점입니다.
예를 들어 일반 채팅은 그냥 지나가는 정보처럼 취급되고, 첨부 텍스트는 분석용, 프로젝트 ZIP은 별도 프로젝트 자료처럼 따로 놀기 쉬웠습니다.
그렇게 되면 같은 정보라도 어떤 방식으로 들어왔느냐에 따라 버려지기도 하고, 과하게 확정되기도 합니다.
그래서 해결 방향은 단순했습니다.
입력은 다를 수 있지만, 기억으로 들어갈 때는 같은 기준과 같은 단계 구조를 따르도록 재정렬한 것입니다.
핵심 교훈은, 입력 방식의 차이는 기술적 차이일 뿐, 기억의 의미까지 달라지는 근거가 되면 안 된다는 점입니다.”

슬라이드 11. 트러블슈팅 04

preview-first가 실제 업데이트를 가로막던 문제

“두 번째 문제는 preview-first 구조였습니다.
첨부 텍스트를 넣으면 겉으로는 분석을 잘해주는 것처럼 보였지만, 실제로는 다음 대화에 남는 기억이 거의 없는 상태가 한동안 있었습니다.
즉 사용자는 이해받은 것처럼 느끼는데, 시스템은 실제로는 배우지 못하는 구조였던 겁니다.
그래서 이 부분을 update-first로 바꿨습니다.
먼저 source를 저장하고, relevant passage를 고르고, evidence를 추출하고, memory에 반영한 다음에, 그 결과를 바탕으로 답변을 만들도록 순서를 바꿨습니다.
이 문제를 통해 얻은 핵심 교훈은, MK4에서 첨부 기반 프로필 요청은 설명 서비스가 아니라 실제 memory update 이벤트여야 한다는 점이었습니다.”

슬라이드 12. 트러블슈팅 08

저장된 memory를 너무 쉽게 믿던 문제

“세 번째 문제는 저장된 기억을 너무 쉽게 믿었던 점입니다.
처음에는 저장된 내용을 거의 참처럼 바로 꺼내서 쓰다 보니, 오래된 오염이나 잘못된 해석이 현재 응답을 덮어쓰는 일이 생겼습니다.
예를 들어 예전의 잘못된 응답 스타일이 계속 반복되거나, 이미 correction이 들어온 내용보다 오래된 profile이 더 강하게 작동하는 문제가 있었습니다.
그래서 저장된 memory를 ‘정답 저장소’가 아니라, 언제든 다시 검토할 수 있는 가설 저장소로 다루는 방향으로 바꿨습니다.
즉 이미 저장된 내용도 검증 대상이고, correction이 있으면 더 우선하며, 꺼내 쓸 때도 최소한의 필터링이 필요하다는 원칙으로 재정렬했습니다.”

슬라이드 13. 트러블슈팅 09

topic 구조가 약해서 대화 의미 축이 흔들리던 문제

“네 번째 문제는 대화 주제를 관리하는 구조가 약했던 점입니다.
초기에는 topic이 단순한 라벨처럼 취급되거나, 애매하면 그냥 한 군데로 뭉개지는 경우가 많았습니다.
그러다 보니 서로 다른 이야기들이 한 bucket에 섞이고, correction이 어느 범위에 적용되는지도 모호해졌습니다.
그래서 topic을 단순 꼬리표가 아니라, 지금 이 대화가 무엇에 대한 이야기인지 보여주는 의미 축으로 다루도록 바꿨습니다.
새 입력이 들어오면 현재 주제를 계속 이어가는지, 예전 관련 주제로 다시 붙는지, 새 주제를 시작하는지 판단하게 만들었습니다.
핵심 교훈은, topic은 보기 좋은 라벨이 아니라 어떤 기억을 지금 꺼내 써야 하는지를 결정하는 좌표축이라는 점입니다.”

슬라이드 14. 트러블슈팅 10

일반 / 후보 / 확정 경계가 채널마다 달라지던 문제

“다섯 번째 문제는 기억의 단계가 채널마다 다르게 해석될 위험이 있다는 점이었습니다.
예를 들어 어떤 정보가 채팅에서는 그냥 지나가는 정보로 남고, 첨부에서는 중요한 후보가 되고, 프로젝트 파일에서는 거의 확정 정보처럼 작동한다면 시스템 전체 일관성이 깨집니다.
그래서 기억의 단계를 가능한 한 통일된 의미로 다시 정리했습니다.
즉 아직 바로 쓰진 않지만 저장할 가치가 있는 정보, 나중에 더 중요한 기억으로 올라갈 수 있는 후보, 현재 사용자 모델에 반영되는 기억, 이렇게 같은 뜻을 유지하려고 했습니다.
핵심은 채널마다 편의적으로 다르게 해석하지 않고, 시스템 전체에서 같은 뜻을 갖는 기억 계층을 만드는 것이었습니다.”

슬라이드 15. 트러블슈팅 13

텍스트 첨부 답변이 템플릿처럼 느껴지던 문제

“여섯 번째 문제는 첨부 기반 답변이 실제 대화처럼 느껴지지 않고, 너무 정형화된 템플릿처럼 보이던 문제였습니다.
이렇게 되면 답변이 기계적으로 보이고, 실제로 바뀐 기억 상태를 충분히 반영하지 못합니다.
그래서 최종 답변도 모델이 직접 생성하게 바꾸고, 템플릿은 설명용 보조 구조로만 줄였습니다.
즉 MK4는 미리 정해둔 문구를 뱉는 시스템이 아니라, 그때까지 갱신된 근거를 바탕으로 실제로 말할 수 있어야 하는 시스템이라는 방향으로 수정했습니다.”

슬라이드 16. 트러블슈팅 14

사용자 질문을 그대로 복사하는 답변이 나오던 문제

“일곱 번째 문제는 echo 문제였습니다.
특히 첨부 기반 경로에서 사용자의 질문을 거의 그대로 되받아 말하는 현상이 나타났습니다.
이건 이해라기보다 입력 복제에 가깝고, 모델이 작을수록 더 자주 나타났습니다.
그래서 질문 원문 전체를 그대로 넘기기보다, 질문의 핵심 요지를 정리해서 답변 입력으로 전달하는 방식으로 바꿨습니다.
또 사용자 문장을 그대로 베껴 답하지 않고, 인칭 표현도 기계적으로 따라 쓰지 않도록 원칙을 분명히 했습니다.
핵심 교훈은, 질문 복사는 이해의 신호가 아니라 입력 설계 실패의 신호라는 점입니다.”

슬라이드 17. 트러블슈팅 15

인칭과 말투가 흔들리던 문제

“여덟 번째 문제는 화자 일관성 문제였습니다.
첨부 텍스트나 프로필 경로에서 나와 너의 기준이 뒤섞이거나, 존댓말과 반말이 섞이는 일이 반복됐습니다.
겉보기에는 말투 문제처럼 보일 수 있지만, 실제로는 같은 시스템이 계속 말하고 있다는 신뢰를 깨뜨리는 문제였습니다.
그래서 사용자 발화의 ‘나’는 사용자 기준, ‘너’는 AI 기준으로 고정하고, 답변 톤도 존댓말로 일관되게 유지하도록 적용했습니다.
핵심 교훈은, 말투 문제는 단순한 문체 문제가 아니라 동일한 인지 주체처럼 보이느냐의 문제라는 점입니다.”

슬라이드 18. 트러블슈팅 20

Correction target kind 분류 체계 도입

“아홉 번째는 correction을 더 세분화한 문제입니다.
초기에는 정정을 거의 한 종류처럼 다뤘는데, 실제로는 무엇을 정정하는지에 따라 영향 범위가 완전히 달랐습니다.
예를 들어 사용자 모델 자체를 고치는 경우와, AI 설명의 사실 오류를 고치는 경우, 그리고 답변 방식만 바꾸는 경우는 서로 다르게 처리돼야 합니다.
그래서 correction을 세 종류로 나눴습니다.
사용자 모델 자체를 고치는 정정, 설명이나 정보 내용을 바로잡는 정정, 답변 길이나 톤 같은 스타일 정정입니다.
이렇게 나누면서 프로필 오염을 줄일 수 있었고, 이후 확장성도 좋아졌습니다.”

슬라이드 19. 트러블슈팅 21

문제 의존성을 제거하지 않고 Python 상한으로만 막으려던 문제

“열 번째는 기술 선택과 관련된 문제였습니다.
특정 외부 라이브러리가 새 Python 버전에서 호환성 불안을 만들고 있었는데, 처음에는 그냥 Python 버전을 제한해서 막는 방향으로 굳어질 위험이 있었습니다.
그런데 그렇게 하면 원인은 라이브러리인데, 프로젝트 전체가 버전 제약을 떠안게 됩니다.
그래서 방향을 바꿔, 문제 라이브러리를 직접 제거하고 내부 구현으로 대체했습니다.
이 과정에서 얻은 교훈은, 장기 프로젝트에서는 문제를 덮기 위해 환경을 묶는 것보다, 구조 자율성을 높이는 선택이 더 건강하다는 점입니다.”

슬라이드 20. 트러블슈팅 2 요약

입력 채널별 의미론 문제 요약

“이 슬라이드는 앞에서 설명한 핵심을 더 짧게 다시 정리한 장입니다.
결국 MK4의 구조 수정은 여러 기능을 따로 붙이는 방향이 아니라, 서로 다른 입력을 하나의 기억 구조 위로 모으는 방향으로 정리되고 있습니다.
즉 채팅, 첨부, 프로젝트 자료가 다르게 보이더라도, 기억 계층 안에서는 같은 언어와 같은 규칙을 써야 한다는 점이 핵심입니다.”

슬라이드 21. 트러블슈팅 3 요약

preview-first 문제 요약

“이 슬라이드도 같은 맥락의 요약입니다.
겉으로 잘 설명하는 것과 실제로 다음 대화에 남는 기억이 생기는 것은 다릅니다.
그래서 MK4는 preview-first가 아니라 update-first를 택했습니다.
사용자에게 그럴듯해 보이는 일회성 해석보다, 실제로 다음 턴에 남는 기억 구조를 우선한 것입니다.”

슬라이드 22. 향후 확장 계획

What is not finished yet

“이제 아직 남아 있는 과제를 말씀드리겠습니다.
가장 큰 과제는 일반 채팅 품질을 첨부 텍스트나 artifact 수준까지 끌어올리는 것입니다.
현재는 대략 75에서 80퍼센트 수준으로 보고 있고, 아직 짧은 답변, 반복, 재사용 안정성 같은 문제를 더 다듬어야 합니다.

또 recent source 관리와 active topic 정리를 위한 source lookup layer,
correction을 더 넓은 수준으로 확장하는 작업,
그리고 응답 시간, 승격 빈도, attach 정확도 같은 지표를 측정해서 개선 루프를 정교하게 만드는 작업도 남아 있습니다.

즉 구조의 뼈대는 어느 정도 세워졌지만, 실사용 품질을 더 밀어 올리는 단계가 아직 남아 있습니다.”

슬라이드 23. 향후 발전 방향 · MK5

Next Stage

“마지막으로 MK5 방향입니다.
다시 상위 프로젝트인 MACHI 맥락으로 돌아가 보면, MK4는 기억 계층을 구현하는 단계이고, MK5는 그 위에 더 넓은 인지 루프를 얹는 단계로 생각하고 있습니다.
즉 기억을 저장하고 다시 쓰는 수준을 넘어, source lookup, reasoning과 planning, 그리고 사용자 ownership과 governance를 더 강하게 연결하려고 합니다.
조금 더 크게 보면, MK5는 단순히 ‘더 큰 모델’을 쓰는 방향이 아니라, 인지, 사고, 결론, 언어화가 더 분리된 구조를 지향하는 다음 단계라고 볼 수 있습니다.
그래서 MK4가 기억 중심의 기반 공사였다면, MK5는 그 기반 위에 더 넓은 인지 구조를 얹는 다음 단계라고 보시면 됩니다.”

슬라이드 24. 정리

Closing

“정리하겠습니다.
MACHI는 장기적으로 사용자를 구조적으로 이해하고 보조하는 로컬 인지 시스템을 지향하는 상위 프로젝트입니다.
그리고 그 안에서 MK4는, 사용자를 어떻게 기억하고, 틀리면 어떻게 고치고, 다음 대화에서 어떻게 다시 활용할지를 다루는 memory substrate 구현 단계입니다.

즉 MK4는 단순히 ‘나를 잘 기억하는 챗봇’을 만들려는 프로젝트가 아닙니다.
기억을 근거 형태로 만들고, 정정으로 다시 고칠 수 있게 하며, 계정이나 플랫폼이 바뀌어도 가져갈 수 있는 로컬 기억 구조를 만들려는 시도입니다.
또 개인화를 프롬프트가 아니라 memory가 담당하게 하고, LLM을 단순 답변 엔진이 아니라 더 큰 인지 구조의 일부로 재배치하려는 프로젝트이기도 합니다.

아직 남은 과제는 많지만, 지금 단계에서는 개념적 memory 구조를 실제 시스템으로 옮기는 데까지는 꽤 왔다고 생각합니다.
이상으로 발표를 마치겠습니다. 감사합니다.”
