# MK4 handoff 문서

## 1. 현재 프로젝트 상태 요약

MK4는 단순 챗봇이 아니라, 시간이 지날수록 사용자 이해를 정교화하는 로컬 개인화 인지 시스템을 목표로 한다.

현재까지 작업 방향은 크게 다음 축으로 진행되었다.

- 일반 채팅 / 첨부 텍스트 / ZIP artifact / project 질문을 하나의 기억 구조로 수렴시키려는 방향
- topic은 의미 축, project_id는 출처 축으로 유지
- memory-driven personalization을 지향하고, prompt 하드코딩을 줄이는 방향
- candidate / confirmed 중심 구조를 일반 채팅에도 붙이려는 작업 진행
- 문자열 휴리스틱 / 의미 하드코딩 제거를 우선 과제로 두는 방향
- 실패는 하드코딩으로 숨기지 않고 오류처럼 드러나게 두는 원칙 채택

## 2. 현재까지 확인된 핵심 문제

### A. memory.db 오염
최근 확인 결과, memory.db에는 사용자 기억이라기보다 모델의 메타 발화나 잘못된 요약이 저장된 흔적이 있었다.

예:
- "알겠습니다."
- "확인합니다. 질문의 의도와 방식에 대한 확인입니다."
- topic/summary가 사용자 주제가 아니라 모델 출력 찌꺼기로 잡힘
- states에 assistant_reply, reasoning, understanding 같은 잡음성 항목이 들어감

핵심 원인 가설:
1. ChatEvidenceService.extract()가 사용자 입력뿐 아니라 모델의 직전 답변도 함께 보고 있음
2. 그 결과, 사용자 특성이 아니라 assistant의 메타 발화가 evidence candidate로 잘못 뽑힘
3. EvidenceNormalizationService.normalize_chat_update_bundle()가 첫 evidence 문장을 topic seed로 사용함
4. 잘못 추출된 candidate가 topic seed가 되어 topic까지 연쇄 오염됨
5. 일반 채팅 경로에서 general이 실제 저장되지 않고 candidate 위주로만 살아남음
6. apply 단계에서 잘못 들어온 profile candidate를 충분히 걸러내지 못함

### B. 일반 채팅의 tier 구조가 반쪽짜리
현재 일반 채팅에서는 correction / episode / state / 일부 profile candidate는 저장되지만,
일반(general) profile memory는 실제 저장 경로가 부족하거나 빠져 있다.

즉 현재 일반 채팅은 개념적으로는 tier 구조를 말하고 있어도,
실제로는 candidate 중심 저장에 가깝고 general은 소실된다.

### C. recent conversation 주입 문제
최근 대화를 원문 transcript처럼 주입하던 구조 때문에,
작은 모델이 이를 참고 문맥이 아니라 출력 템플릿처럼 따라 쓰는 문제가 있었다.

이 문제를 줄이기 위해:
- recent conversation은 원문 transcript가 아니라 1문단 요약으로 넘기는 방향으로 수정
- 첨부 텍스트 답변 프롬프트에서는 원문 재낭독 / 감상문 / 위로문 출력 금지 방향으로 정리

### D. continuation scaffolding 누수
ResponseRunner의 continuation prompt에서 괄호 플레이스홀더나 메타 안내문이 출력으로 새는 문제가 있었다.
해당 부분은 최신 로컬에서 사용자가 직접 수정한 response_runner.py를 최신본으로 간주해야 한다.

## 3. 현재 상태에서 신뢰 가능한 것 / 불안정한 것

### 비교적 방향이 맞는 것
- topic / project_id 축 분리
- artifact → evidence → sync 방향
- prompt 사용자 하드코딩 축소
- update-first 방향
- project/profile prompt 외부화
- ResponseRunner가 config.py 기반 설정을 사용하는 방향
- recent conversation을 요약해서 넘기려는 방향

### 아직 불안정한 것
- 일반 채팅 evidence 추출 품질
- topic seed 산출 방식
- general 실제 저장
- promotion 의미론 정리
- memory.db 오염 방지 규칙
- assistant 발화가 기억 계층으로 섞이는 문제
- retrieval 품질 및 response retrieval의 신뢰도

## 4. 다음 채팅에서 우선적으로 이어서 해야 할 작업

아래 작업이 다음 우선 과제다.

### 1) ChatEvidenceService.extract()에서 모델의 직전 답변을 아예 제외
목표:
- 일반 채팅 evidence 추출은 사용자 입력만 재료로 사용
- assistant 발화는 로그로만 남기고, evidence/profile/general/candidate/confirmed 어떤 tier에도 입력 금지

의도:
- memory.db 오염의 가장 앞단 차단
- assistant 메타 문장이 candidate로 저장되는 문제 제거

### 2) EvidenceNormalizationService.normalize_chat_update_bundle()의 topic seed를 첫 문장이 아니라 "요약된 현재 대화"로 변경
목표:
- 첫 evidence envelope.content 하나에 topic을 맡기지 않음
- 현재 대화의 요약본을 topic seed로 사용
- 잘못 뽑힌 첫 문장 하나가 topic 전체를 오염시키는 구조 완화

의도:
- topic 추출 안정화
- evidence 추출 오류의 연쇄 전염 완화

### 3) ExtractionPolicy.extract()에서 general 저장 추가
목표:
- 일반 채팅에서도 general을 실제 저장 경로로 보냄
- candidate만 살아남고 general이 날아가는 현재 구조 수정

의도:
- general → candidate → confirmed로 이어질 수 있는 재료층 확보

### 4) MemoryClassificationPolicy에서 general이 실제 저장되도록 수정
목표:
- current structured signal을 보고 candidate / general / confirmed를 분기하되,
  general도 discard가 아니라 저장 계층으로 내려가게 정리
- 문자열 의미 추정 규칙은 제거 유지
- 숫자 문턱값은 필터용 상수로만 유지

의도:
- tier 의미론을 실제 DB 반영과 일치시키기

### 5) promotion 정리
목표:
- 2회 반복이면 승격
- 1회라도 매우 높은 신뢰도면 승격 가능
- direct confirm은 채널 공통 기준
- 채널 간 교차 evidence도 승격 근거로 허용
- promotion은 의미 추정 규칙이 아니라 structured signal + evidence pool 기준으로 계산

### 6) MemoryApplyService.apply_extracted()에서 general을 실제로 저장하도록 수정
목표:
- 현재 apply 단계가 candidate / confirmed 위주로만 반영되는 문제 해소
- general도 저장되되, 기본 응답 주입은 하지 않는 방향 유지

## 5. 다음 작업에서 유지해야 할 강한 규칙

- assistant 발화는 general / candidate / confirmed 어떤 profile tier에도 입력 금지
- 실패는 담백하게 오류로 처리하고, 하드코딩 fallback으로 숨기지 않기
- 문자열 의미 추정 규칙 금지
- 숫자 문턱값은 임시 정책 상수로는 허용하되, 의미 결정값처럼 쓰지 않기
- 일반 채팅도 첨부/프로젝트처럼 후보 저장 → 승격 → 확정 구조를 가져야 함
- general은 넓게 저장하되 기본 주입은 하지 않음
- 채널을 넘는 승격 허용
- recent conversation은 원문 transcript가 아니라 요약으로 넘기기
- 최신 로컬 MK4를 기준본으로 삼고, 수정 파일만 파일 단위로 제공하기

## 6. sync 주의사항

- 사용자가 다음 채팅에서 최신 MK4.zip을 다시 줄 수 있음
- 반드시 그 최신 로컬 기준으로 sync를 다시 맞춘 뒤 이어서 작업할 것
- 예전 ZIP/과거 패치본을 기준으로 작업하지 말 것
- response_runner.py는 사용자가 직접 수정한 최신본이 있을 수 있으므로, 먼저 비교 확인할 것

## 7. 한 줄 요약

현재 MK4의 가장 큰 문제는 "기억이 없다"가 아니라,
"무엇을 기억 재료로 써야 하는지"가 아직 오염되어 있다는 점이다.

다음 작업의 핵심은:
assistant 발화를 기억 재료에서 제외하고,
chat evidence → topic seed → tier 저장 → promotion의 앞단을 다시 안정화하는 것이다.
