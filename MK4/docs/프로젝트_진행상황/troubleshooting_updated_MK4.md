# MK4 Troubleshooting / 방향 수정 기록

이 문서는 **MK4에서 지금까지 실제로 겪은 문제, 그 문제를 통해 바뀐 설계 방향, 그리고 그에 따라 수정된 개념적 원칙**을 정리한 문서다.  
목적은 단순한 버그 로그가 아니라, 다음 작업에서 다시 같은 우회를 반복하지 않도록 **“왜 이 방향으로 바뀌었는가”**를 남기는 데 있다.

기준:
- 정리 기준일: 2026-04-13
- 정리 단위: **코드 파일 변경 내역이 아니라 개념/구조 단위**
- 반영 범위: 현재 프로젝트 상태 + 이전 대화들에서 합의된 방향성 수정 + 문제 해결 과정
- 통합 기준:
  - 이 문서는 기존 `troubleshooting.md`와 `troubleshooting_project_only.md`의 핵심 내용을 흡수한 최신 기준본이다.
  - 예전 문서에 있던 개별 버그 사례와 project 전용 메모는, 여기서 더 큰 구조 변화와 운영 원칙 기준으로 재정리했다.

---

## 0. 이 문서가 필요한 이유

MK4는 단순 CRUD 프로젝트가 아니라,  
**“사용자를 더 일관되게 이해하는 로컬 개인화 인지 시스템”**을 목표로 하다 보니  
작은 구현 실수도 곧바로 구조 철학의 왜곡으로 이어졌다.

즉, 여기서의 troubleshooting은 보통의
- 에러 원인 찾기
- 함수 시그니처 맞추기
- import 수정

수준에서 끝나지 않았다.

실제로는 계속 다음 질문이 따라붙었다.

- 이 문제는 진짜 버그인가, 아니면 설계 철학이 틀어진 신호인가?
- 지금 붙이려는 해결책은 임시 우회인가, 아니면 장기 구조와 맞는가?
- 채팅 / 첨부 / 프로젝트 artifact를 서로 다르게 처리하는 것이 맞는가?
- prompt에 더 많이 써 넣는 것이 해결인가, 아니면 memory 구조를 바꾸는 것이 해결인가?

그래서 이 문서는 “무엇을 고쳤는가”보다  
**“어떤 잘못된 접근을 버렸고, 어떤 원칙으로 재정렬했는가”**를 기록한다.

## 1. 기준본이 흔들리면서 작업이 분기되던 문제

### 문제
프로젝트를 ZIP 스냅샷 단위로 다루면서, 수정이 누적되지 않고 여러 갈래로 갈라지는 문제가 반복되었다.

### 왜 문제가 되었는가
이 방식은 겉으로는 안전해 보이지만, 실제로는 다음 문제가 생겼다.

- 어떤 수정이 최신 기준인지 불명확해짐
- 이전 대화에서 반영한 구조 변경이 새 ZIP 기준 작업에서 사라짐
- “이전 수정 위에 다음 수정을 얹는” 것이 아니라, 매번 다른 가지에서 따로 발전하는 상태가 됨

### 방향 수정
작업 기준을 “대화 중 임시 산출물”이 아니라 **사용자의 최신 로컬 프로젝트 상태**로 다시 고정했다.

### 현재 원칙
- 기준본은 여러 개가 아니라 **항상 최신 로컬 기준본 1개**
- 새 ZIP은 새 출발점이 아니라 **현재 기준본과의 sync 재료**
- 이전 채팅에서 만든 파생 ZIP 여러 개를 병렬 기준으로 삼지 않음

### 핵심 교훈
MK4 같은 구조 프로젝트는 “한 번 맞춘 구조를 계속 누적”해야 한다.  
스냅샷 중심 작업은 편해 보여도, 실제로는 설계 기억을 계속 잃게 만든다.

## 2. 프로젝트의 목표가 “답변 생성”으로 오해되던 문제

### 문제
초기 흐름에서는 MK4가 “잘 답하는 챗봇”처럼 다뤄지는 순간이 자주 있었다.

### 왜 문제가 되었는가
그렇게 되면 최적화의 기준이 아래처럼 잘못 이동한다.

- 답변이 그럴듯한가
- 말투가 자연스러운가
- 당장 응답이 나오기만 하는가

하지만 MK4의 실제 목표는 그게 아니다.

### 방향 수정
프로젝트의 중심을 다시 아래로 재정의했다.

- 답변 품질보다 **사용자 이해의 일관성**
- fact 적재보다 **user model 형성**
- append-only 기억보다 **update / correction / rebuild 가능한 기억**
- 그럴듯한 답변보다 **근거 있는 해석과 교정 가능성**

### 핵심 교훈
MK4에서 답변은 결과물이지 본질이 아니다.  
본질은 **시간이 지나도 더 정교해지는 사용자 모델**이다.

## 3. 입력 채널별로 기억 의미론이 갈라지던 문제

### 문제
일반 채팅, 첨부 텍스트, 프로젝트 artifact가 서로 다른 시스템처럼 취급되는 경향이 있었다.

예를 들면:
- 채팅은 임시 업데이트
- 첨부 텍스트는 별도 preview
- 프로젝트 ZIP은 project 분석 전용

처럼 분리되기 쉬웠다.

### 왜 문제가 되었는가
같은 사용자 정보라도 들어오는 채널이 다르면 전혀 다른 수준으로 취급되는 모순이 생겼다.

이 상태에서는
- 같은 의미가 채팅에서는 버려지고
- 첨부에서는 후보가 되고
- 프로젝트에서는 다른 규칙으로 저장되는
식의 불일치가 발생한다.

### 방향 수정
입력 채널은 달라도, 기억으로 들어갈 때는 **공통 의미론**으로 수렴시켜야 한다는 방향으로 바꿨다.

현재 핵심 원칙:
- 채널은 다를 수 있다
- 하지만 기억의 tier 의미는 최대한 같아야 한다
- `general / candidate / confirmed`의 해석은 전 채널에서 일관되어야 한다
- chat / uploaded_text / project_artifact가 결국 같은 apply 계층으로 모여야 한다

### 핵심 교훈
입력 채널 분리는 기술적 편의일 뿐, 의미론 분리의 근거가 될 수 없다.

## 4. preview-first가 실제 업데이트를 가로막던 문제

### 문제
첨부 텍스트를 이용한 프로필 형성 요청이 한동안 “분석해주는 응답”만 만들고 실제 기억에는 거의 반영되지 않는 구조로 흘렀다.

### 왜 문제가 되었는가
이 방식은 사용자가 보기에 뭔가 이해한 것처럼 보이지만,
실제로는 시스템이 **배운 것이 없는 상태**가 된다.

즉:
- 지금은 잘 말하는 것처럼 보임
- 하지만 다음 대화에서는 그 결과가 남지 않음
- 결국 “이해”가 아니라 일회성 해석문 생성에 머무름

### 방향 수정
`preview-first`에서 `update-first`로 전환했다.

핵심 흐름은 다음과 같이 재정렬되었다.

- source를 저장하고
- relevant passage를 고르고
- evidence를 추출하고
- candidate/confirmed 판단을 거쳐
- memory sync/apply를 수행하고
- 그 다음 자연어 답변을 만든다

### 핵심 교훈
MK4에서 첨부 기반 프로필 요청은 “설명 서비스”가 아니라  
**실제 memory update 이벤트**여야 한다.

## 5. project와 profile을 지나치게 분리하던 문제

### 문제
프로젝트 분석과 사용자 프로필 형성을 너무 별개 작업처럼 다루려는 흐름이 있었다.

### 왜 문제가 되었는가
실제 artifact에는 둘이 섞여 있다.

예를 들어:
- 프로젝트 구조를 어떻게 나누는지
- 디버깅 방식이 어떤지
- 무엇을 중요하게 여기는지
- 어떤 기준으로 설계를 비판하는지

이런 것은 project 정보이면서 동시에 profile evidence이기도 하다.

### 방향 수정
`project`와 `profile`은 배타적 분기가 아니라,  
하나의 입력이 **둘 다의 근거가 될 수 있다**는 쪽으로 정리했다.

현재 원칙:
- `project_id`는 출처 축
- `topic`은 의미 축
- 둘은 대체 관계가 아님
- project artifact도 profile evidence를 만들 수 있어야 함
- “이건 project니까 profile이 아니다” 같은 강한 분리를 지양

### 핵심 교훈
MK4의 artifact는 문서가 아니라 **행동의 흔적**이다.  
그래서 project 데이터는 종종 profile 데이터이기도 하다.

## 6. 문자열 하드코딩이 구조 판단을 대신하던 문제

### 문제
초기/중간 단계에서는 여러 곳에서 문자열 휴리스틱이 의미 판단을 대신하는 문제가 있었다.

예:
- 특정 표현이 들어가면 profile 요청으로 본다
- 특정 문장이면 일반 채팅으로 본다
- 특정 길이나 말끝이면 답변이 덜 끝난 것으로 본다
- 기억 후보 여부를 문장 패턴으로 가늠한다

### 왜 문제가 되었는가
이런 방식은 빠르게 붙일 수는 있어도, MK4가 지향하는 방향과 정면으로 충돌한다.

문제는 다음과 같다.

- 일반화가 약하다
- 예외가 늘어난다
- 새 상황이 오면 규칙이 계속 추가된다
- 결국 “사람을 이해하는 시스템”이 아니라 “문자열 분기기”가 된다

### 방향 수정
문자열 규칙을 줄이고, 다음으로 이동시켰다.

- 모델 기반 route resolver
- structured signal 기반 memory classification
- topic router 기반 의미 축 결정
- 임베딩 기반 selection / retrieval
- 규칙은 의미 판단이 아니라 최소한의 안전장치에만 사용

### 핵심 교훈
하드코딩은 임시 응급처치로는 쓸 수 있어도,  
MK4의 장기 구조를 맡길 수는 없다.

## 7. system prompt가 사용자 모델을 하드코딩하던 문제

### 문제
한 시기에는 system prompt 안에 사용자의 특성, 사고방식, 반응 구조, 설명 선호 등이 과하게 들어가 있었다.

### 왜 문제가 되었는가
그렇게 되면 기억 시스템이 아무리 바뀌어도,  
실제 답변 레이어는 여전히 **고정된 사용자상**을 강제로 따르게 된다.

즉:
- correction이 들어와도 prompt가 이기고
- 새 evidence가 쌓여도 오래된 persona가 유지되고
- personalization이 memory-driven이 아니라 prompt-driven이 된다

### 방향 수정
system prompt를 얇게 만들고, 운영 원칙만 남기는 방향으로 바꿨다.

남겨야 하는 것은:
- 정직성
- 검증 우선
- correction 반영
- 과장 금지
- 저장 메모도 검증 대상이라는 점

반대로 최대한 빼야 하는 것은:
- 사용자 개인에 대한 단정적 서술
- 기억 시스템이 담당해야 할 성향 정보
- 장황한 페르소나 설명

### 핵심 교훈
MK4에서 personalization은 prompt가 아니라  
**memory / correction / evidence 구조**가 담당해야 한다.

## 8. 저장된 memory를 너무 쉽게 믿던 문제

### 문제
한동안 저장된 memory를 거의 “참인 정보”처럼 바로 주입하면서, 예전 오염이나 잘못된 해석이 현재 응답을 덮어쓰는 일이 생겼다.

### 왜 문제가 되었는가
기억은 원본 진실이 아니라, 과거의 해석 산물이다.  
따라서 틀릴 수 있고, 낡을 수 있고, system-like 오염이 섞일 수도 있다.

실제로 다음 같은 문제가 나타났다.

- 오래된 잘못된 응답 스타일이 계속 재주입됨
- correction보다 과거 profile이 더 강하게 작동함
- system prompt 냄새가 나는 긴 문장이 memory인 척 들어감

### 방향 수정
기억을 “신뢰 가능한 사실 저장소”가 아니라  
**검증과 필터링이 필요한 가설 저장소**로 취급하도록 바꿨다.

현재 원칙:
- 저장된 memory도 검증 대상
- correction이 있으면 이전 profile보다 우선
- 주입 전 최소한의 위생 필터 필요
- 오염 레코드는 DB 차원에서도 정리해야 함

### 핵심 교훈
기억을 무조건 믿는 순간, MK4는 학습 시스템이 아니라  
과거 오류의 증폭기가 된다.

---

## 9. topic 구조가 약해서 대화 의미 축이 흔들리던 문제

### 문제
초기에는 topic이 단순 문자열 라벨처럼 취급되거나, `general`로 뭉개지는 경우가 많았다.

### 왜 문제가 되었는가
이 상태에서는 다음 문제가 생긴다.

- 서로 다른 의미 축이 같은 bucket에 쌓임
- correction이 어느 범위에 적용되는지 모호해짐
- active topic 유지/전환 판단이 불명확해짐
- “이 정보가 무엇에 대한 기억인가”가 흐려짐

### 방향 수정
topic을 단순 꼬리표가 아니라 **의미 축 객체**로 다루는 방향으로 바꿨다.

핵심 변화:
- active topic 유지 / 기존 topic attach / 새 topic create 판단
- topic과 project_id를 분리
- topic은 의미 축, project_id는 출처 축으로 구분
- `general`은 기본 수거함이 아니라 예외적 fallback로 취급
- general 계층은 기본 주입하지 않고 보존/정리 정책을 둠

### 핵심 교훈
MK4에서 topic은 분류용 문자열이 아니라  
**기억의 적용 범위를 정하는 핵심 좌표축**이다.

---

## 10. 일반 / 후보 / 확정 경계가 채널마다 달라지던 문제

### 문제
candidate와 confirmed의 의미가 채널마다 다르게 해석될 위험이 있었다.

### 왜 문제가 되었는가
예를 들어 어떤 정보가
- 채팅에서는 일반 정보로만 남고
- 첨부에서는 후보가 되고
- project artifact에서는 거의 확정처럼 작동한다면

시스템 전체 일관성이 무너진다.

### 방향 수정
tier 의미를 가능한 한 통일하는 쪽으로 재정렬했다.

현재 지향:
- 일반: 저장 가치는 있지만 기본 주입 대상은 아님
- 후보: 승격 가능성이 있는 evidence 집합
- 확정: 현재 사용자 모델에 반영되는 층위

추가 원칙:
- direct confirm은 강한 신호지만 절대 자동 진실은 아님
- 반복, 출처 강도, correction 충돌 여부 등을 함께 봐야 함
- 확정은 영구 고정값이 아니라 다시 깨질 수 있는 상태

### 핵심 교훈
기억 계층은 채널별 편의 규칙이 아니라  
**시스템 전체에서 같은 뜻을 가져야 한다.**

---

## 11. retrieval과 selection이 표면적 유사도에 머물던 문제

### 문제
초기 retrieval/selection은 단순 lexical matching이나 얕은 점수표 중심으로 흐르기 쉬웠다.

### 왜 문제가 되었는가
MK4가 찾고 싶은 것은 단순 키워드 일치가 아니라,  
**사용자의 안정적 특성, 반복 기준, 사고 방식, correction 포인트를 잘 드러내는 근거**다.

그런데 lexical 중심 접근은 다음 한계가 있었다.

- 표현이 조금만 바뀌어도 놓친다
- 의미는 비슷하지만 단어가 다른 문단을 잘 못 잡는다
- topic 요약, passage selection, project retrieval 품질이 낮아진다

### 방향 수정
retrieval/selection을 점차 다음 구조로 옮겼다.

- passage 선택: profile 형성에 유용한 문단을 의미 기반으로 선별
- project retrieval: chunk 임베딩 기반 검색
- topic 관련 판단: 임베딩 + active topic 유지 로직
- keyword 휴리스틱은 보조 또는 축소 대상

### 핵심 교훈
MK4의 retrieval은 “단어 찾기”가 아니라  
**의미 있는 evidence 찾기**여야 한다.

---

## 12. 응답이 중간에 잘리거나 반복되던 문제

### 문제
로컬 Ollama 환경에서 긴 응답이 문장 중간에 끊기거나, 이어 말하기에서 반복이 생기거나, 불완전한 응답이 정상 응답처럼 보이는 문제가 있었다.

### 왜 문제가 되었는가
이 문제는 단순 UX 이슈가 아니다.

- 완전하지 않은 답변이 정상처럼 보이면 평가가 왜곡됨
- continuation이 잘못 붙으면 의미 반복이 누적됨
- “모델이 생각은 했는데 답변이 덜 나온 상태”와 “정상 종료”를 구분 못하면 디버깅이 어려워짐

### 방향 수정
응답 생성의 운영 원칙을 아래처럼 다시 세웠다.

- `done_reason == length`를 실제 truncation으로 본다
- continuation은 **정말 잘렸을 때만** 붙인다
- continuation 여부를 모호한 문자열 휴리스틱으로 판단하지 않는다
- `num_predict`는 충분히 확보하되, 무작정 크게만 잡지는 않는다
- 이어붙일 때는 overlap similarity를 이용해 반복을 줄인다
- 이 로직은 경로별 임시 구현이 아니라 공용 runner에서 다루는 편이 맞다

### 핵심 교훈
응답 품질 문제의 상당수는 “모델이 멍청해서”가 아니라  
**생성 완료 판정과 continuation 정책이 부정확해서** 생긴다.

---

## 13. 텍스트 첨부 답변이 템플릿처럼 느껴지던 문제

### 문제
첨부 텍스트 기반 응답이 실제 대화처럼 생성되지 않고, 미리 짜둔 템플릿처럼 반복되던 시기가 있었다.

### 왜 문제가 되었는가
이 상태에서는 사용자는 “기억 업데이트가 된 것 같은 느낌”만 받을 뿐,
실제 시스템은 다음 두 면에서 약해진다.

- 답변이 지나치게 기계적이라 품질 신뢰가 떨어짐
- 템플릿 구조가 바뀐 메모리 상태를 충분히 반영하지 못함

### 방향 수정
최종 답변도 모델이 생성하도록 바꾸고, 템플릿은 설명용 보조 구조에서만 쓰는 방향으로 조정했다.

### 핵심 교훈
MK4는 정해진 문구를 뱉는 시스템이 아니라,  
**갱신된 근거를 바탕으로 그때그때 말할 수 있어야 하는 시스템**이다.

---

## 14. 사용자 질문을 그대로 복사하는 답변이 나오던 문제

### 문제
특히 첨부 기반 경로에서 사용자의 질문을 거의 그대로 되받아 말하는 echo 문제가 있었다.

### 왜 문제가 되었는가
이런 답변은 실제 이해가 아니라 입력 복제를 보여준다.  
작은 모델일수록 더 자주 발생했고, prompt가 길수록 더 심해졌다.

### 방향 수정
답변 입력에 질문 원문 전체를 그대로 넘기기보다,
**질문의 요지**를 요약해서 전달하는 방향으로 바꿨다.

또한 명시적으로 다음 원칙을 세웠다.

- 사용자 문장을 그대로 베껴 답하지 않는다
- 인칭 표현을 기계적으로 따라 쓰지 않는다
- 첨부 기반 답변도 일반 대화와 같은 화자 일관성을 유지한다

### 핵심 교훈
질문 복사는 이해의 신호가 아니라  
**입력 설계 실패의 신호**다.

---

## 15. 인칭과 말투가 흔들리던 문제

### 문제
첨부 텍스트/프로필 경로에서 특히 다음 문제가 반복되었다.

- 나/너 기준이 뒤섞임
- 존댓말과 반말이 섞임
- 일반 대화와 첨부 답변의 말투가 다른 사람처럼 느껴짐

### 왜 문제가 되었는가
MK4의 핵심은 사용자 이해의 일관성인데,  
화자 일관성이 깨지면 시스템 전체의 신뢰가 떨어진다.

### 방향 수정
다음 원칙을 별도 경로에도 강하게 적용했다.

- 사용자 발화의 `나/내`는 사용자 기준
- `너/네`는 AI 기준
- 답변은 존댓말만 유지
- 첨부 경로도 일반 채팅과 너무 다른 보고서 톤으로 분리하지 않음

### 핵심 교훈
말투 문제는 사소한 스타일 이슈가 아니라  
**동일한 인지 주체처럼 보이느냐의 문제**다.

---

## 16. reply guard와 fallback이 응답 자체를 대신하던 문제

### 문제
일부 보정 계층이 원래 목적을 넘어서, 응답 생성의 본체처럼 작동할 위험이 있었다.

### 왜 문제가 되었는가
guard/fallback은 실패를 감추기 위해 커지기 쉽다.  
하지만 그렇게 되면 모델 기반 시스템 위에 또 다른 매크로 시스템이 얹히게 된다.

### 방향 수정
보정 계층의 역할을 축소해서 정리했다.

- guard는 guard 역할만
- 의미 판단은 가능하면 upstream 구조에서
- fallback은 최소한의 실패 안전장치로만
- 오류는 오류처럼 드러나게 두는 원칙 유지

### 핵심 교훈
보조 레이어가 똑똑해질수록 시스템이 좋아지는 것이 아니라,  
오히려 **본체의 문제를 가리는 레거시층**이 될 수 있다.

---

## 17. 문서와 실제 코드 상태가 자꾸 어긋나던 문제

### 문제
README, todo, handoff, troubleshooting 문서와 실제 현재 코드가 어긋나는 일이 반복되었다.

### 왜 문제가 되었는가
이 상태에서는 같은 항목도
- 어떤 문서에서는 “아직 안 됨”
- 다른 문서에서는 “반영됨”
- 실제 코드에서는 “부분 반영”
으로 보이게 된다.

그러면 다음 채팅에서 판단 비용이 매우 커진다.

### 방향 수정
문서를 단순 기록이 아니라 **현재 구조를 이어받기 위한 운영 도구**로 취급하기 시작했다.

즉:
- handoff는 다음 대화 진입용
- troubleshooting은 왜 방향이 바뀌었는지 기록
- todo는 “현재 코드 기준”으로만 유지
- 완료/부분완료/미완료를 분리해서 적는다

### 핵심 교훈
MK4에서는 문서도 기억 계층의 일부다.  
문서가 틀리면 다음 작업도 틀어진다.

---

## 18. 일반 채팅을 너무 약한 입력으로 취급하던 문제

### 문제
첨부 텍스트나 project artifact에 비해, 일반 채팅은 상대적으로 빈약한 업데이트 경로로 남을 위험이 계속 있었다.

### 왜 문제가 되었는가
실제 사용자 모델은 일반 채팅에서 가장 많이 드러난다.  
그런데 이 채널이 너무 약하면, 시스템은 정작 가장 풍부한 입력원을 덜 활용하게 된다.

### 방향 수정
일반 채팅도 결국 다음 수준까지 끌어올려야 한다는 방향이 분명해졌다.

- evidence 추출
- candidate 축적
- 조건부 승격
- correction 반영
- topic 기반 적용

즉, 일반 채팅도 첨부/프로젝트 수준의 기억 의미론으로 수렴해야 한다.

### 핵심 교훈
일반 채팅은 부가 입력이 아니라,  
**가장 중요한 장기 evidence 공급원**이다.

---

## 19. 현재까지 정착된 핵심 원칙

지금까지의 문제 해결을 거치며, 아래 원칙들은 거의 프로젝트 헌법에 가깝게 정착했다.

### 19-1. 기준본 원칙
- 최신 로컬 기준본 1개를 중심으로 sync한다.

### 19-2. 기억 원칙
- memory는 append-only 로그가 아니다.
- correction, rebuild, discard가 가능해야 한다.

### 19-3. personalization 원칙
- prompt-driven persona보다 memory-driven personalization을 지향한다.

### 19-4. 입력 원칙
- 채널은 달라도 기억 의미론은 최대한 통일한다.

### 19-5. retrieval 원칙
- 단어 일치보다 evidence relevance를 본다.

### 19-6. 응답 원칙
- truncation은 실제 상태로 감지한다.
- continuation은 잘렸을 때만 수행한다.
- 반복은 overlap 기준으로 줄인다.

### 19-7. 문서 원칙
- 문서는 다음 작업을 위한 구조 기억 장치다.
- “완료처럼 보이게 포장”하지 않는다.

이 문서는 완료 보고서가 아니다.

4. correction이 topic 단위 재구성에 더 강하게 연결되도록 고도화하기
5. 속도/품질 측정 루프를 구조적으로 만들기

## 21. 최종 요약

지금까지 MK4에서 가장 큰 전환은 아래 여섯 가지다.

1. **ZIP 중심 작업**에서 **최신 기준본 sync 중심 작업**으로 이동  
2. **답변 중심 사고**에서 **사용자 모델 중심 사고**로 이동  
3. **채널별 분리 처리**에서 **공통 기억 의미론**으로 이동  
4. **preview-first**에서 **update-first / evidence-first**로 이동  
5. **문자열 규칙 중심**에서 **모델 판단 + 구조화 신호 중심**으로 이동  
6. **prompt 하드코딩 personalization**에서 **memory-driven personalization**으로 이동  

즉 MK4는 단순히 “버그를 고친 프로젝트”가 아니라,  
문제를 겪을 때마다 **무엇을 시스템의 본질로 볼 것인가**를 다시 정리해 온 프로젝트라고 보는 것이 맞다.
