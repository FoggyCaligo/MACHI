# MK4 Troubleshooting Log

이 문서는 **MK4 프로젝트 자체에서 실제로 부딪힌 문제**와,
그 문제를 어떤 방향으로 해결했는지, 그리고 그 결과 구조가 어떻게 바뀌었는지를
**개념 기준으로** 정리한 기록이다.

정리 원칙은 다음과 같다.

- 사용자-어시스턴트 협업 과정에서 생긴 운영 문제는 제외한다.
- 코드 조각이나 파일명보다 **설계 문제 / 동작 문제 / 해결 원리**를 우선 적는다.
- “무엇을 고쳤는가”보다 **왜 그 문제가 생겼고, 어떤 구조로 바뀌었는가**를 남긴다.

---

## 1. 첨부 텍스트가 “프로필 업데이트”가 아니라 “일회성 해석”으로 끝나던 문제

### 문제
사용자가 블로그 글이나 자기소개 텍스트를 첨부해도,
시스템이 그것을 **장기 기억 갱신의 입력**으로 다루지 않고
그 순간의 해석 답변만 반환하는 경향이 있었다.

### 원인
첨부 텍스트 경로가 “이 자료를 근거로 사용자 모델을 갱신한다”가 아니라
“이 자료를 읽고 지금 한 번 설명한다”는 흐름으로 설계되어 있었기 때문이다.

### 해결
첨부 텍스트 경로를 **preview-first**에서 **update-first**로 바꿨다.

즉, 첨부 자료를 받으면 다음 순서로 처리하도록 구조를 바꿨다.

1. 첨부 자료를 source로 저장
2. 의미 있는 passage를 선별
3. profile evidence를 추출
4. evidence를 정규화
5. memory 정책에 따라 반영
6. 그 결과를 바탕으로 답변

### 결과
첨부 자료가 더 이상 “읽고 끝나는 참고자료”가 아니라,
**사용자 모델을 갱신하는 실제 입력 채널**로 동작하게 되었다.

## 2. 채팅 / 첨부 / 프로젝트 산출물이 서로 다른 기억 규칙으로 들어가던 문제

### 문제
같은 성격의 정보라도 어디서 들어왔는지에 따라
저장 기준과 승격 기준이 달라지는 문제가 있었다.

예를 들어,
- 채팅에서 나온 정보
- 첨부 텍스트에서 추출한 정보
- 프로젝트 산출물에서 추출한 정보

이 셋이 서로 다른 규칙을 타면,
결국 기억 시스템이 **의미 기준이 아니라 입력 채널 기준**으로 갈라지게 된다.

### 원인
초기 구조가 경로별 기능 구현에 집중되어 있었고,
공통적으로 써야 할 판단 레이어가 충분히 통합되지 않았기 때문이다.

### 해결
기억 반영 흐름을 **공통 ingress 구조**로 재정리했다.

핵심은 다음 두 가지다.

- 추출은 경로별로 달라도,
  최종 반영은 공통 정책을 타게 한다.
- “이게 candidate인가, confirmed인가, general인가”를
  입력 채널이 아니라 **구조화된 evidence 의미**로 판정한다.

### 결과
프로젝트는 점점 “채널별 기능 모음”이 아니라
**하나의 기억 시스템 위에 여러 입력 경로가 연결된 구조**로 바뀌었다.

## 3. 프로젝트 ZIP이 단순 검색 자료에 머물고, 사용자 프로필 형성으로 잘 연결되지 않던 문제

### 문제
프로젝트 업로드 기능은 있었지만,
그 안의 문서나 코드에서 드러나는 사용자의 습관·선호·사고방식을
장기 프로필로 끌어올리는 흐름이 약했다.

즉, 프로젝트는 질문에 답하기 위한 검색 대상일 뿐,
사용자 이해를 깊게 만드는 자료로는 충분히 활용되지 못했다.

### 원인
프로젝트 처리 경로가
- 파일 추출
- 청크 검색
- 질문 응답

에는 초점이 맞춰져 있었고,
프로젝트 artifact 자체로부터 **profile evidence를 추출하고 동기화**하는 흐름이 약했기 때문이다.

### 해결
프로젝트 ingestion 이후에,
프로젝트 문서 중 프로필 형성에 의미 있는 문서를 선별하고
그 안에서 evidence를 추출해 memory 쪽으로 동기화하는 경로를 붙였다.

또한 프로젝트 질문도
- 프로젝트 자체에 대한 질문
- 프로젝트를 통해 드러나는 사용자 특성에 대한 질문

을 구분해 처리하도록 바꿨다.

### 결과
프로젝트는 더 이상 “질문 답변용 자료집”만이 아니라,
**사용자 모델을 보강하는 장기 evidence source**로 편입되었다.

## 4. topic이 약해서 기억이 퍼지고, 관련 없는 정보가 한 덩어리처럼 섞이던 문제

### 문제
대화와 evidence가 축적될수록,
어떤 기억이 어떤 주제 축에 속하는지 불명확해지는 문제가 있었다.

그 결과
- 새 정보가 엉뚱한 기존 맥락에 붙거나
- 너무 쉽게 general로 흩어지거나
- 반대로 너무 잘게 쪼개진 topic이 생기는 문제가 발생했다.

### 원인
기억 저장은 있었지만,
그 기억을 묶는 **주제 단위(topic) 관리**가 충분히 강하지 않았기 때문이다.

### 해결
topic을 별도 객체처럼 다루는 방향으로 바꿨다.

핵심은 다음과 같다.

- 현재 active topic을 유지할지 판단
- 기존 topic에 attach할지 판단
- 정말 필요할 때만 새 topic 생성
- 이 판단에 embedding 유사도와 요약 기반 재매칭을 함께 사용

### 결과
기억이 무작정 누적되는 대신,
**어떤 주제 위에 무엇이 쌓이는지**를 더 안정적으로 관리할 수 있게 되었다.

## 5. 특정 단어에 의존한 규칙 기반 라우팅이 쉽게 오작동하던 문제

### 문제
“블로그”, “글”, “화자”, “프로필” 같은 특정 표현에 과하게 의존하면,
실제 의도보다 **표면 단어**에 끌려 라우팅이 흔들리는 문제가 있었다.

예를 들어, 문맥상은 일반 질문인데 특정 단어가 들어갔다는 이유로
프로필 경로로 오인되거나,
반대로 프로필 업데이트 의도인데 단어가 부족해서 놓치는 식의 문제가 생길 수 있다.

### 원인
초기에는 빠르게 기능을 붙이기 위해
문자열 트리거와 규칙 기반 분기를 많이 사용했기 때문이다.

### 해결
라우팅을 점차 **의미 판단 중심**으로 옮겼다.

- 질문의 성격을 모델이 좁은 스키마 안에서 판정하게 하고
- 규칙은 최소한의 안전장치로만 남기고
- 특정 도메인 단어 하드코딩은 줄였다.

### 결과
라우팅이 “이 단어가 있나?”보다
**“이 질문이 실제로 무엇을 요구하나?”**에 더 가깝게 움직이게 되었다.

## 6. 프로필 추출이 과장되거나 이론명 중심으로 흘러가던 문제

### 문제
프로필 추출 결과가
- 지나치게 분류표처럼 딱딱하거나
- 원문에 없는 학자/이론/심리 용어를 끌어오거나
- evidence보다 해석이 앞서는 식으로 과장되는 문제가 있었다.

### 원인
추출 프롬프트가 “예쁘게 정리된 프로필”을 만들도록 유도하는 면이 있었고,
작은 모델은 이런 상황에서 보수적 추출보다 **그럴듯한 구조화**를 더 쉽게 선택했다.

### 해결
추출의 기준을 “정리된 프로필”에서 “근거 중심 evidence”로 옮겼다.

핵심 원칙은 다음과 같다.

- 애매하면 넣지 않는다.
- 원문 밖의 이론 이름을 끌어오지 않는다.
- 해석보다 evidence를 우선한다.
- 많이 뽑는 것보다 적고 정확하게 뽑는다.

### 결과
프로필 추출은 점점 “성격 분석 보고서 생성”이 아니라
**검증 가능한 후보 진술 수집**에 가까워졌다.

## 7. 프롬프트 같은 텍스트가 기억 DB에 저장되어 답변을 오염시키던 문제

### 문제
과거에 잘못 들어간 메타 텍스트나 system-like 문장이
기억처럼 저장되고,
그것이 다시 답변 생성에 주입되어 현재 대화보다 더 강하게 작동하는 문제가 있었다.

이 문제는 특히
- 이상하게 과장된 공감 문구
- 현재 맥락과 맞지 않는 답변 태도
- 사용자의 최근 정정보다 과거의 오염된 문장에 끌리는 현상

으로 드러났다.

### 원인
기억 저장 단계에서 “저장하면 안 되는 텍스트”를 충분히 거르지 못했고,
응답 생성 단계에서도 저장된 기억을 너무 쉽게 신뢰했기 때문이다.

### 해결
두 층에서 방어했다.

1. **저장 위생 강화**
   - 프롬프트 냄새가 나는 텍스트, 메타 문장, 과도하게 긴 규범 문장을 경계
2. **주입 위생 강화**
   - 저장된 기억을 그대로 진실로 쓰지 않고
   - 답변에 넣기 전에 필터링

### 결과
기억은 “저장된 모든 것의 합”이 아니라,
**다시 검증되어 주입되는 정보층**이라는 방향이 더 분명해졌다.

## 8. 사용자 특성이 system prompt에 과하게 박혀 있어, 기억 갱신성이 떨어지던 문제

### 문제
사용자에 대한 설명이 system prompt 안에 과하게 박혀 있으면,
새로운 evidence나 correction이 들어와도
시스템이 이미 정해진 사용자상을 계속 재생산하는 문제가 생긴다.

### 원인
초기에는 개인화 강도를 빠르게 확보하려고
사용자 특성을 상위 프롬프트에 강하게 넣는 쪽으로 기울었기 때문이다.

### 해결
system prompt는 **운영 원칙만 남기는 얇은 층**으로 줄이고,
사용자 특성은 memory / correction / evidence 쪽에서 관리하도록 방향을 바꿨다.

남긴 것은 주로 다음과 같다.

- 정직성
- 검증 우선
- 과장 금지
- 정정 반영
- 기억도 검증 대상이라는 원칙

### 결과
개인화가 “상단 프롬프트의 하드코딩”이 아니라
**갱신 가능한 기억 구조**로 이동했다.

## 9. 첨부 기반 답변 경로가 일반 대화와 다른 화자처럼 보이던 문제

### 문제
첨부 텍스트를 처리한 뒤의 답변이
일반 대화와 말투·화자 기준·톤이 달라져
마치 다른 시스템이 말하는 것처럼 보이는 문제가 있었다.

구체적으로는
- 질문 문장을 그대로 되풀이하거나
- 나/너 기준이 흔들리거나
- 존댓말/반말이 섞이거나
- 지나치게 템플릿 같은 답변이 나오는 현상이 있었다.

### 원인
첨부 처리 경로가 일반 대화 경로와 별도로 굴면서,
최종 답변 생성 방식과 프롬프트 기준이 달랐기 때문이다.

### 해결
다음 방향으로 정리했다.

- 최종 답변을 코드 템플릿이 아니라 모델 생성으로 통일
- 질문 원문을 그대로 복사하지 않도록 요약 입력 사용
- 인칭 기준을 별도 규칙으로 명시
- 존댓말 유지와 과한 템플릿화를 억제

### 결과
첨부 기반 응답도 점차 일반 대화와 같은 시스템의 연장선처럼 보이게 되었다.

## 10. 답변이 길이 제한으로 중간에서 잘려도 정상처럼 보이던 문제

### 문제
로컬 모델이 길이 제한에 걸려 중간 종료된 답변을 내놓아도,
시스템은 이를 정상 완료처럼 취급하는 문제가 있었다.

이 경우 사용자 입장에서는
“답변이 그냥 좀 어색하게 끝났다” 정도로만 보이지만,
실제로는 **불완전한 출력**이었다.

### 원인
응답 본문이 비어 있지 않다는 이유만으로 정상으로 간주했고,
모델의 종료 사유를 충분히 해석하지 않았기 때문이다.

### 해결
응답 처리에서 **truncation을 명시적으로 감지**하도록 바꿨다.

- 길이 제한 종료를 구분
- 완전성이 필요한 호출은 실패로 처리
- 일반 응답은 잘림 여부를 표시
- 필요하면 continuation으로 이어받도록 연결

### 결과
“중간에 잘렸지만 모르는 척 넘어가는 시스템”에서
**잘림을 인식하고 대응하는 시스템**으로 바뀌었다.

---

## 11. continuation이 반복과 재시작을 만들어 답변 품질을 망치던 문제

### 문제
응답이 잘렸을 때 이어서 생성해야 하는데,
실제로는
- 앞부분을 다시 반복하거나
- 이미 한 말을 조금 바꿔 다시 말하거나
- 문단을 새로 시작해 중복을 만들거나
- 꼬리만 조금 바꾸는 식의 저품질 continuation

이 나오는 문제가 있었다.

### 원인
“이어 써라”는 지시만으로는 부족했고,
모델이 **어디까지 이미 말했는지**를 충분히 받지 못했기 때문이다.

### 해결
continuation을 다음 원칙으로 바꿨다.

- **정말 잘렸을 때만** continuation 수행
- 직전 답변의 완결 구간과 잘린 꼬리를 분리해서 제공
- 이어쓰기 전용 프롬프트 사용
- tail similarity와 overlap 제거로 반복 차단
- 합쳐진 결과가 실제로 새 내용이 아닐 경우 중단

### 결과
continuation은 단순 재시도가 아니라,
**이전 답변의 정확한 끝지점을 기준으로 이어 붙이는 보정 단계**로 바뀌었다.

## 12. 작은 로컬 모델 환경에서 timeout / 빈 응답 / thinking-only 응답이 발생하던 문제

### 문제
로컬 CPU 환경과 작은 모델 조합에서,
긴 입력이나 무거운 프롬프트가 들어오면
- timeout
- content 없는 빈 응답
- thinking만 있고 최종 답변이 없는 응답

이 발생하는 문제가 있었다.

### 원인
프로젝트가 실제 배포형 대형 추론 환경이 아니라
제약 있는 로컬 실행을 전제로 하기 때문에,
프롬프트 길이와 생성량을 잘못 잡으면 쉽게 병목이 생긴다.

### 해결
다음 방식으로 대응했다.

- 경로별 timeout과 생성 길이를 재조정
- 추출용/답변용/분류용 호출 budget을 분리
- 너무 긴 입력은 passage selection으로 줄임
- 빈 응답 유형을 분류해 원인 파악 가능하게 함
- continuation은 한 번 더 모델을 돌리는 비용이므로 제한적으로 사용

### 결과
프로젝트는 “모델이 알아서 다 해주겠지”가 아니라,
**로컬 추론 제약을 전제로 budget을 관리하는 구조**로 바뀌었다.

## 13. 프로젝트 내부의 중복 구현이 수정 전파를 막던 문제

### 문제
비슷한 역할을 하는 로직이 여러 경로에 따로 있으면,
한 곳에서 문제를 고쳐도 다른 경로에는 그대로 남는 문제가 생긴다.

이 문제는 특히 다음 영역에서 두드러졌다.

- evidence 추출
- passage selection
- 기억 분류
- 응답 생성 / continuation

### 원인
기능이 늘어나는 과정에서 경로별로 구현이 증식했고,
공통 레이어보다 개별 기능 완성이 우선되었기 때문이다.

### 해결
중복 로직을 가능한 한 공통 서비스로 끌어올렸다.

핵심은 다음과 같다.

- evidence extraction 공통화
- passage selection 공통화
- memory classification 공통화
- response running / continuation 공통화

### 결과
프로젝트는 기능 추가형 구조에서,
**핵심 레이어를 공유하는 구조**로 조금씩 재정렬되었다.

## 14. 현재까지의 핵심 변화 요약

초기 MK4는 경로별 기능이 먼저 붙은 구조에 가까웠고,
문제가 생길 때마다 개별 패치를 얹는 경향이 있었다.

지금은 방향이 다음처럼 바뀌었다.

- 첨부 자료는 해석용이 아니라 **기억 갱신 입력**
- 프로젝트 artifact는 검색 자료가 아니라 **장기 evidence source**
- 기억 반영은 채널별이 아니라 **공통 ingress**
- topic은 문자열 꼬리표가 아니라 **맥락 관리 단위**
- 추출은 멋진 해석보다 **보수적 evidence 수집**
- 개인화는 하드코딩 prompt보다 **갱신 가능한 memory**
- continuation은 재시도가 아니라 **잘린 응답 복구 단계**

## 15. 아직 완전히 끝난 것은 아닌 부분

다음 항목들은 방향은 잡혔지만, 계속 검증이 필요한 영역이다.

1. topic 생성/attach 품질의 장기 안정성
2. evidence가 너무 적게 또는 너무 보수적으로 추출되는 경우의 균형 조정
3. correction이 topic 수준 재구성까지 얼마나 확장될지
4. 로컬 모델 크기 차이에 따른 라우팅/추출/응답 품질 편차
5. 일반 대화 경로와 첨부/프로젝트 경로 사이의 톤 일관성
6. retrieval budget과 answer budget의 추가 최적화

---

## 16. 재발 방지 원칙

- 입력 채널이 달라도 기억 반영 기준은 가능하면 공통화한다.
- evidence는 해석보다 보수적으로 뽑는다.
- 저장된 기억도 검증 대상이지 절대 진실이 아니다.
- 개인화는 prompt 하드코딩보다 갱신 가능한 memory 구조에 둔다.
- continuation은 “더 말해”가 아니라 “어디서 끊겼는지 복구”로 다룬다.
- 로컬 추론 환경에서는 모델 능력보다 budget 설계가 더 중요할 수 있다.
