**Tier 1: 일반 채팅 품질 고도화 (1-5개)**
1. 일반 채팅 품질을 첨부 텍스트 / project artifact 수준의 기억 처리 품질까지 끌어올리기
2. `general / candidate / confirmed`의 의미론을 전 채널에서 더 엄밀하고 동일하게 맞추기
3. chat 경로의 `general`이 저장은 되더라도, 이후 승격 / 재사용 / 검색까지 포함한 전체 의미론을 더 안정화하기
4. candidate -> confirmed 승격 기준을 더 안정화하고, 충돌 / correction / 출처 강도 반영을 더 정교화하기
5. 일반 채팅 답변 경로의 짧음 / `...` / 반복 / 얼버무림 / 모르는 걸 모른다고 못 하는 문제를 재현 테스트하며 수정

**Tier 2: 구조 정리 및 책임 경계 (6-10개)**
6. recent source / DB / 과거 대화를 모델이 필요할 때만 찾는 source lookup layer 설계 및 구현
7. recent source를 최근 3개 유지 + 사용자 발화 4턴 미사용 시 후보 제외 정책으로 실제 운영 구조에 반영
8. `reply_guard.py`에 남아 있는 구조 판단 역할을 더 축소하고 source lookup layer와의 책임 경계 재정리
9. `app/api.py`와 상위 orchestration에서 recent source를 질문마다 자동 검사하지 않는 방향으로 재설계
10. artifact / profile attachment 경로를 topic router 철학에 더 깊게 맞추기

**Tier 3: Topic & Correction 확장 (11-15개)**
11. artifact 경로에 맞는 topic create-or-attach 정책을 더 세분화하기
12. 새 topic 생성 제약을 프롬프트 수준을 넘어 후처리 단계까지 강화하기
13. correction 기반 재구성을 profile 수준에서 topic 수준 merge / split / rebuild까지 확장하기
14. retrieval 예산 (budget) 2차 축소와 context compaction 설계 (지금은 profile 1개만이므로 이미 줄어있음)
15. `response_retriever` 응답 컨텍스트 품질 재검증 (embedding 기반 retrieval로 변경 후)

**Tier 4: Recent Context & Handoff (16-20개)**
16. 새 채팅으로 넘길 자동 handoff 요약 흐름 만들기
17. `ResponseRunner`와 프론트 렌더링 경로를 함께 점검해서 continuation이 UI에서 실제로 정상 동작하는지 검증
18. 텍스트 첨부 후 follow-up 질문에서, 직전 source를 자동 병합하지 않고도 필요 시 참조 가능한 구조 설계

20. retrieval budget 관리 (지금은 profiles 1, corrections 2, summaries 1, episodes 2 수준)

24. 남아 있는 하드코딩 / 매크로 응답 / 도메인 특정 문자열 의존 / 응답 경로 우회 브리지 / 중복 상수 / 레거시 참조를 전체 파일 기준으로 재탐색

최종 메모 (2026-04-14)
==================================================

- Retrieval이 이제 응답/회상 경로 모두 의미 기반으로 정리됨 (embedding 기반 또는 active context 우선)
- ProfileStore/CorrectionStore/SummaryStore의 문자열 search() 메소드는 주요 retrieval 경로에서 더 이상 호출되지 않음
- EpisodeStore.find_relevant()는 embedding 기반 의미 검색으로 전환됨
