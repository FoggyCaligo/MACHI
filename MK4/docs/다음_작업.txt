해야 하는 것들 우선순위

프롬프트 경량화

지금 가장 급해. 네 노트북 환경에선 답변 품질보다 먼저 응답이 나와야 하니까, system/user prompt 길이와 불필요한 메타정보부터 줄여야 해.

긴 채팅을 새 채팅처럼 넘기는 context compaction 구조 설계

이건 속도 문제의 근본 과제야.
“최근 원문 전부”가 아니라 “핵심 상태 요약 + 필요시 retrieval” 구조로 바꿔야 진짜 빨라질 수 있어.

auto-promotion 의도 재정렬

지금 철학 메모와 실제 코드가 어긋나는 핵심 지점이야.
“정말 자동 승격을 할 건지”, “할 거면 어떤 조건에서만 할 건지”를 먼저 확정해야 해.

첨부 텍스트 답변 경로 프롬프트 단순화

현재 여기가 느리고 말투도 흔들리기 쉬워.
answer prompt를 더 짧게 줄이고, 내부 상태 설명도 최소화해야 해.

인칭/존댓말 안정화

작은 모델에서는 이게 금방 무너져.
프롬프트를 길게 보강하는 방식보다, 입력 포맷 자체를 더 단순하게 만드는 쪽이 효과적일 가능성이 커.

profile evidence가 너무 쉽게 0건이 되는지 점검

승격 이전에 추출 자체가 막히면 전체 구조가 의미가 없어져.
“보수적 추출”과 “아무것도 못 뽑는 상태”는 구분해서 봐야 해.

memory DB 오염 레코드 실제 상태 재확인

문서상 이슈가 컸던 만큼, 현재 작업본 DB에도 진짜 남아 있는지 확인이 필요해.
이미 정리됐는데 문서만 남아 있을 수도 있고, 반대로 일부만 남아 있을 수도 있어.

correction policy 2차 고도화

지금은 최신 correction 우선 1차 버전이라 해석 충돌이 생기면 너무 단순하게 처리될 수 있어.
장기적으로는 topic별 누적 correction 해석이 필요해.

topic 분류 개선

지금 규칙 기반은 빠르지만 거칠어.
다만 속도가 급한 현재 단계에서는 1순위는 아니고, 프롬프트/컨텍스트 경량화 다음이 맞아.

trusted_search 실제 구현

중요하긴 하지만, 현재 병목이 속도와 메모리 구조 쪽이라 우선순위는 그보다 아래야.
먼저 내부 구조를 가볍게 만든 뒤 붙이는 게 맞아 보여.

구조가 더 안정된 뒤 맞추는 편이 효율적이야.

작업 순서표
1. topic 분류 규칙 기반 제거

extraction_policy.py의 TOPIC_RULES와 규칙 기반 infer_topic()를 제거한다.
이 파일은 앞으로 분류기가 아니라 topic decision 결과 검증기 / 정규화기 역할만 맡긴다.

2. topic decision 스키마 확정

출력을 keep_active | attach_existing | create_new | general 중 하나로 고정한다.
즉 topic 추론은 “이름 붙이기”가 아니라 유지/부착/생성/보류 결정이 되게 바꾼다.

3. active topic 유지 판별기 추가

매 턴마다 먼저 “현재 대화가 기존 active topic 안에 아직 속하는가”만 본다.
속하면 끝, 아니면 그때만 기존 topic 검색 단계로 넘어간다.

4. existing topic summary 비교 경로 추가

기존 topic들의 짧은 summary + 이름만 후보로 넣고, 현재 턴이 붙을 곳이 있는지 판단한다.
텍스트명 일치가 아니라 “이 대화가 저 대분류 안에 들어갈 수 있는가” 기준으로 붙인다.

5. 새 topic 생성 제약 추가

새 topic을 만들 때는 너무 구체적인 이름 금지 규칙을 넣는다.
예: long sword history보다 history, weapon system 같은 넓은 축으로 생성하게 한다.

6. general 저장 정책 확정

timeout이면 버리고, malformed JSON이나 low confidence면 general로 저장한다.
단 general은 매 턴 주입 금지, 필요할 때만 검색 대상으로 쓴다.

7. confirmed profile 승격 정책 공격형 반영

1개 high-confidence evidence도 신규 특성/신규 profile 형성 시작 허용으로 바꾼다.
대신 source trace와 correction 재구성 경로를 반드시 남긴다.

8. correction 기반 재구성 경로 보강

confirmed/general/topic 모두 깨지고 재구성될 수 있어야 한다는 전제를 코드에 명시한다.
즉 오염 방지는 승격 금지가 아니라, 나중에 다시 고칠 수 있는 구조로 해결한다.

9. topic prompt 신설 및 경량화

topic inference 전용 프롬프트 파일을 따로 만들고, 아주 짧은 JSON만 출력하게 한다.
현재 프롬프트 병목과 작은 모델의 불안정성을 줄이려면, 지시는 적고 출력도 짧아야 한다.

10. system/answer prompt 추가 경량화

system_prompt.txt와 profile_attachment_answer_system_prompt.txt는 운영 원칙만 남기고 더 줄인다.
현재 문서상 방향도 사용자 개인 특성 하드코딩 제거 + 운영 원칙만 유지 쪽이다.

11. retrieval budget 축소

response_builder.py 쪽은 오염 필터는 유지하고, 주입량만 줄인다.
특히 general은 기본 주입에서 빼고, confirmed/correction 위주로만 얇게 넣는다.

12. topic store / summary persistence 추가

topic별로 name, summary, last_used_at, confidence 정도는 저장해야 한다.
그래야 다음 턴에서 active topic 유지 여부와 existing topic attach 판단을 가볍게 할 수 있다.

13. artifact 경로부터 우선 적용

처음엔 일반 채팅 전체보다 profile attachment / explicit correction / artifact 분석 경로부터 붙인다.
원래 철학도 evidence-first이므로, 먼저 이 경로가 살아야 한다.

14. 일반 채팅 경로는 2차 적용

artifact 경로가 안정화된 뒤에 일반 턴에도 topic decision을 확장한다.
처음부터 전 경로에 붙이면 속도 문제와 디버깅 난이도가 같이 커진다.

15. 마지막에 실제 속도/품질 점검

각 단계마다 응답시간, topic 생성 정확도, general 누적량, confirmed 승격 빈도를 확인한다.
지금 MK4는 “정교함”보다 “실제로 돌아감”이 먼저라서, 매 단계 성능 확인이 필수다.

지금 상태 기준 남은 작업 우선순위
1. artifact / profile attachment 경로를 topic router 철학에 더 깊게 맞추기

지금 일반 채팅 쪽은 topic routing이 많이 들어왔는데,
첨부 텍스트·artifact 쪽은 아직 extract prompt 중심 흐름이 더 강해.

2. confirmed / candidate / general 경계를 시스템 전체에서 통일하기

지금은 경로마다 기준이 조금씩 달라.
네가 정한 “기억 가치가 크면 general보다 위로” 철학을 일반 채팅, 첨부 텍스트, project evidence 경로 전체에 일관되게 맞춰야 해.

3. correction 기반 재구성을 profile 수준에서 topic 수준까지 확장하기

지금도 재구성 뼈대는 있지만, 아직은 profile/summary 쪽이 중심이야.
앞으로는 topic attach 결과나 confirmed/general 상태도 correction으로 다시 깨고 재구성할 수 있어야 해.

4. 새 topic 생성 제약을 후처리까지 넣기

지금은 프롬프트로 “너무 구체적으로 만들지 말라” 정도만 걸려 있어.
생성된 topic summary가 지나치게 좁거나 고유명사 중심이면 후처리에서 다시 걸러내는 단계가 필요해.

5. retrieval budget 2차 축소

1차로는 많이 줄었지만, 아직 더 깎을 여지가 있어.
특히 recent messages, summaries, episodes, raw expansion 쪽은 속도 기준으로 한 번 더 다이어트할 수 있어.

6. 실제 속도 / 품질 측정 루프 만들기

지금은 구조가 많이 바뀌었는데, 아직 실측 루프가 부족해.
응답시간, topic 유지율, attach 정확도, general 누적량, confirmed 승격 빈도를 실제로 찍어봐야 다음 판단이 쉬워져.

7. artifact 경로에 맞는 topic create-or-attach 정책 세분화

지금은 일반 채팅에서의 active topic 중심 판단이 더 강해.
artifact는 한 번의 긴 텍스트에서 여러 evidence가 나오기 때문에, 같은 topic에 묶을지 새 topic을 팔지 기준을 더 세밀하게 잡아야 해.

8. 남아 있는 레거시/미사용 코드 계속 삭제

지금도 꽤 정리됐지만, 아직 중복 흔적과 미사용 함수 후보가 남아 있어.
이건 구조가 더 안정된 뒤 계속 잘라내면 돼.

9. general의 보존/검색 정책 미세조정

90일 TTL과 비주입화는 이미 들어갔지만,
“어떤 general을 나중에 다시 끌어올릴지”와 “어떤 general은 그냥 잊을지” 기준은 더 다듬을 수 있어.

10. topic 생성 품질이 부족할 때만 embedding 모델 상향 검토

지금은 multilingual-e5-small로 시작하는 게 맞아.
나중에 실제 attach/keep 품질이 부족하다고 느껴질 때만 더 큰 모델을 검토하면 돼.


최소한의 임베딩 기반 selection으로 passage_selection_service.py를 나중에 손봐야 할 것 같아.

그 대신 아직 모델이 필요할 때 recent source / DB / 과거 대화를 스스로 찾는 구조는 안 만들어졌어.

2-4. 구체적인 런타임 문제 후보가 하나 보인다

이건 그냥 철학 문제가 아니라 실제 버그 후보다.

project_analysis/services/project_profile_evidence_service.py 안의 _resolve_candidate_topic()에서:

resolution = self.topic_router.resolve(
    user_message=routing_text,
    model=model,
    use_active_topic=False,
    persist_active=False,
)

이렇게 호출하는데,
현재 memory/services/topic_router.py의 resolve() 시그니처는 내가 본 기준으로:

def resolve(self, user_message: str, model: str | None = None) -> TopicResolution:

이거다.

즉 지금 코드 기준으로는 use_active_topic, persist_active를 안 받는다.
그러면 이 경로가 실제로 실행될 때 TypeError 날 가능성이 높다.

즉:

py_compile은 통과해도
실제 실행 경로에서는 깨질 수 있다

이건 우선순위 높다.

2-5. project retrieval은 아직 “검색기”라기보다 “점수 기반 grep 강화판”에 가깝다

project_analysis/retrieval/project_retriever.py는 솔직히 말해서 아직 약하다.

현재 방식은 대략:

토큰 추출
path bonus
내용 포함 점수
상위 chunk 반환

즉 의미 기반 retrieval이라기보다 파일 경로 + 단어 포함 기반 휴리스틱 랭킹이다.

이 구조는 다음에는 잘 작동한다.

파일명 직접 언급
함수명/클래스명 직접 언급
“구조”, “리팩토링”, “에러” 같은 큰 카테고리 질문

하지만 다음에는 약하다.

구현 의도 질문
함수명이 아니라 역할로 묻는 질문
서로 떨어진 chunk를 종합해야 답이 나오는 질문
문맥상 연관된 여러 파일을 함께 봐야 하는 질문

즉 project Q&A는 지금 **“읽을 만한 근거 몇 개를 뽑아주는 단계”**까지는 왔는데,
아직 “프로젝트를 깊게 이해하는 retriever”까지는 아니다.

2-6. response retrieval과 topic routing의 품질 축이 섞여 있다

이 부분도 구조상 애매하다.

topic routing은 임베딩 기반
project retrieval은 lexical scoring 기반
profile/summaries 검색은 SQL LIKE 기반

즉 기억 retrieval 계층 전체가 하나의 retrieval 철학으로 통일돼 있지 않다.

현재는 기능별로:

어떤 곳은 임베딩
어떤 곳은 문자열 포함
어떤 곳은 최근순 fallback

으로 섞여 있다.

초기 프로토타입에선 괜찮지만,
이제 이 정도 규모면 슬슬 문제가 된다.

왜냐하면 나중에 사용자는 “왜 어떤 기억은 잘 잡히고, 어떤 기억은 안 잡히지?”를 체감하게 되는데,
그 원인이 모델 문제가 아니라 retrieval 방식 혼재에서 생기기 때문이다.






2-7. correction 충돌 판단이 다소 거칠다

_has_conflicting_active_correction() 류 로직은 지금 너무 거칠 가능성이 크다.

현재 구조상 같은 topic 안에 active correction이 있고,
candidate_content와 “같은 의미”가 아니면 충돌로 보는 식인데,
이건 문제를 일으킬 수 있다.

왜냐하면 topic 자체를 넓게 잡으려 하고 있기 때문이다.

즉 topic이 넓을수록 한 topic 안에 들어가는 내용 폭이 커지는데,
그 상태에서 correction 충돌 판단이 거칠면
실제로는 병존 가능한 두 프로필 내용도 서로 막아버릴 수 있다.

예를 들면 같은 “설명 선호” topic 안에서도

구조적 설명 선호
감정적 위로도 나중 층위에서 중요

이 둘은 충돌이 아니라 병존인데,
현재의 단순 의미 비교 방식은 이런 다층 구조를 잘 못 다룰 수 있다.



2-8. UpdateRetriever와 chat-side extraction은 아직 너무 얕다

memory/retrieval/update_retriever.py는 사실상 규칙 기반 초기 버전이다.

correction 힌트
state 힌트
길이 20자 이상이면 new_episode

이 정도다.

즉 현재 일반 채팅에서의 memory extraction은
네가 구상하는 MK4 철학에 비해 아직 너무 얕고 기계적이다.

Artifact/profile evidence 쪽은 꽤 무겁게 설계했는데,
일반 채팅 쪽은 아직 간단한 규칙층에 많이 기대고 있다.

그래서 지금 MK4의 중심 축은 사실
“대화형 memory OS” 전체라기보다
artifact-driven profile formation 쪽이 더 진화한 상태라고 보는 게 맞다.

이번 배치의 중요한 한계

여기는 꼭 솔직히 말할게.

지금 구조엔 chat-side separate general/candidate store가 아직 없다.
그래서 이번 배치에서는 candidate-level chat memory만 profile로 쓰고,
general은 아직 독립 저장층으로 구현하지 않았다.

이건 의도적이다.

왜냐하면 지금 general까지 기존 profiles 테이블에 그냥 넣어버리면
네가 원하지 않는 방식으로 기본 주입 surface에 섞여 들어갈 위험이 크기 때문이다.

즉 지금은:

잘못 저장해서 오염시키는 것
보다
일단 candidate만 저장하는 것

을 택했다.

그래서 이 단계는 휴리스틱 제거 + 잘못된 저장 방지까지이고,
general/candidate/confirmed를 진짜 계층으로 만드는 건 다음 단계다.






























///////////////////////////////









일단 문자열 휴리스틱 등의 하드코딩 들을 전부 걷어내는 작업을 먼저 하고, 그 다음에 일반 채팅 쪽을 다른 부분과 같은 수준으로 올려놓는 작업을 진행하자.


이미 방향이 맞는 부분
topic/project 축 분리
artifact → evidence → sync 방향
project에서도 profile evidence를 뽑으려는 구조
general/candidate/confirmed 계층 의식
correction 중심 사고
update-first
prompt의 사용자 하드코딩 축소
passage selection의 임베딩 기반 전환 시도
아직 방향과 구현이 어긋나는 부분
memory_classification_policy.py에 남아 있는 규칙 기반 신호
일반 채팅 쪽 extraction의 얕은 규칙 기반 분기
project retrieval의 lexical scoring 성격
recent source / response retrieval 쪽의 혼합 구조
계층 철학은 공통인데 구현이 채널별로 따로 자란 흔적

즉 지금 MK4는 철학은 꽤 선명한데,
구현은 아직 transition state다.



1단계: 문자열 휴리스틱/하드코딩 전면 제거

먼저 해야 할 건 “더 똑똑하게 만들기”가 아니라
잘못된 기반을 걷어내는 것이다.

우선 제거 대상 핵심 파일은 이쪽이다.

memory/policies/memory_classification_policy.py
memory/policies/extraction_policy.py
memory/retrieval/update_retriever.py
project_analysis/retrieval/project_retriever.py
memory/retrieval/response_retriever.py
app/api.py
reply_guard.py








2단계: 공통 ingress 정책 확정

그 다음엔 채널별로 따로 자란 걸 하나로 묶어야 한다.

공통 정책은 대략 이렇게 가는 게 맞다.

입력:

chat
attachment
project artifact
correction

공통 중간 구조:

source
evidence
route
topic attach/create/keep
source trace
confidence

공통 결과:

discard
general
candidate
confirmed

즉 채널은 다르지만 ingress policy는 하나여야 한다.





3단계: 일반 채팅 쪽을 artifact 수준으로 끌어올리기

지금 일반 채팅이 약한 이유는,
애초에 다른 부분보다 뒤늦게 붙었기 때문이다.

그래서 제거 후에는

chat도 evidence-first
chat도 topic-aware
chat도 promotion-aware
구조로 올려야 한다.

즉 일반 채팅만 예외적인 얕은 규칙층으로 남겨두면 안 된다.







4단계: retrieval 전면 의미 기반화

그 다음에 retrieval을 통일해야 한다.

project에서도 의미 기반
profile에서도 의미 기반
response context에서도 의미 기반

이렇게 묶어야 시스템 전체가 한 철학으로 간다.
