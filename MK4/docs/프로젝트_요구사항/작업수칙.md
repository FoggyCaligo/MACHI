# MK4 작업수칙 (업데이트본)

이 문서는 MK4를 수정할 때 앞으로 계속 지켜야 하는 작업 원칙을 업데이트한 버전이다.

## 1. 기준본 원칙
- 작업 기준은 항상 사용자가 가장 최근에 올린 로컬 MK4다.
- 예전 ZIP 스냅샷이나 과거 패치본을 기준으로 덮어쓰지 않는다.
- 새 작업을 시작할 때는 먼저 최신 로컬 기준본과 현재 작업 상태를 sync 맞춘다.

## 2. sync 맞추는 방법
- 사용자가 최신 MK4.zip을 올리면, 그 ZIP을 풀어서 현재 구조와 파일 상태를 먼저 확인한다.
- 이후 수정은 반드시 그 기준본 위에서만 이어서 적용한다.
- 이전 단계에서 만든 여러 ZIP 버전을 각각 따로 발전시키지 않는다.
- 항상 "최신 로컬 1개 기준 + 순차 패치" 방식으로 간다.

## 3. 파일 제공 방식
- 전체 ZIP 재배포보다 수정된 파일만 파일 단위로 제공하는 것을 기본으로 한다.
- 각 파일은 반드시 아래 형식으로 준다.
  - 원래 경로
  - 다운로드 링크
  - 필요 시 반영 순서
- 1번용 ZIP, 2번용 ZIP, 3번용 ZIP처럼 분기된 결과물을 만들지 않는다.

## 4. 오류 처리 원칙
- 실패했을 때는 담백하게 오류로 처리한다.
- 실패를 하드코딩, 문자열 비교, 임시 우회로 숨기지 않는다.
- 모델 호출 실패, 파싱 실패, 필수 필드 누락은 가능한 한 명확하게 드러나게 둔다.
- 지금 단계에서는 "오류도 오류처럼 보이게" 만드는 것이 우선이다.
- 추가 fallback이 필요하면 나중에 정책적으로 붙인다.

## 5. 하드코딩 금지
- 의미 해석을 문자열 하드코딩으로 처리하지 않는다.
- 규칙 기반 키워드 매칭, 특정 문장 패턴 하드코딩, 임의의 의미 추정용 문자열 비교는 금지한다.
- 특히 사용자 성향, 정정, 선호, NEED 같은 의미 판단을 규칙 몇 개로 처리하지 않는다.
- 하드코딩은 현재 프로젝트 철학에 정면으로 위배되는 레거시로 본다.

## 6. 정책 판단이 필요한 경우
- 구조적 선택지가 갈리거나 철학에 영향을 주는 판단이 필요한 경우에는 먼저 사용자에게 묻는다.
- 예:
  - 확정 승격 기준
  - general / candidate / confirmed 경계
  - correction 우선순위
  - source lookup 동작 방식
  - 채널 간 promotion 허용 범위
- 애매한 경우 혼자 밀어붙이지 않는다.

## 7. 프로젝트 방향성
MK4는 답변을 잘하는 챗봇보다, 시간이 지날수록 더 일관되게 사용자를 이해하는 로컬 개인화 인지 시스템을 목표로 한다.

핵심 방향:
- 기억은 append-only 로그가 아니다.
- fact 저장보다 user model 형성이 우선이다.
- 문서, 첨부 텍스트, ZIP artifact도 evidence-first로 다룬다.
- correction은 부가기능이 아니라 핵심 메커니즘이다.
- prompt-driven persona보다 memory-driven personalization을 지향한다.

## 8. 현재 구조 이해
- topic은 의미 축이다.
- project_id는 출처 축이다.
- 둘은 대체 관계가 아니다.
- 일반 채팅, 첨부 텍스트, 프로젝트 artifact는 입력 채널이 다르더라도 결국 공통 기억 구조로 수렴해야 한다.

현재 핵심 구조:
- MemoryIngressService: 입력 정규화와 라우팅
- MemoryApplyService: 반영, 승격, 재구성
- ExtractionPolicy: envelope를 memory action으로 변환
- MemoryClassificationPolicy: tier 판정
- TopicRouter: active topic 유지/attach/create
- ResponseBuilder: 응답용 문맥을 조립하되, recent conversation은 원문 transcript보다 요약을 우선

## 9. 기억 구조 정책
기억은 최소한 다음 층위를 가진다.
- general: 저장 가치는 있지만 기본 주입은 하지 않는 정보
- candidate: 승격 후보
- confirmed: 현재 모델이 보고 있는 사용자상

원칙:
- confirmed는 절대 진실이 아니다.
- correction이 오면 언제든 깨지고 재구성될 수 있다.
- 채널이 달라도 tier 의미론은 최대한 같아야 한다.
- 일반 채팅도 첨부/프로젝트처럼 후보 저장 → 승격 → 확정 경로를 가져야 한다.
- general은 넓게 저장해야 한다.
- 채널 간 교차 evidence도 승격 근거로 허용한다.

## 10. assistant 발화 처리 규칙
- assistant 발화는 general / candidate / confirmed 어떤 profile tier에도 입력 금지
- assistant 발화는 raw log나 디버깅용 참고, recent conversation 요약 재료 정도로만 제한적으로 사용할 수 있다
- assistant의 메타 발화("알겠습니다", "확인합니다", "기억합니다" 등)는 절대 기억 후보가 되면 안 된다

## 11. 프롬프트 작업 원칙
- 프롬프트는 길이보다 계약이 중요하다.
- 장황한 설명보다 역할, 판단 기준, 출력 스키마를 짧게 유지한다.
- 모델을 과하게 유도하는 긴 문장은 줄인다.
- 프롬프트 다이어트는 기본 원칙으로 본다.
- "성격 강제"보다 "태스크 경계와 출력 형식 정리"가 우선이다.
- 첨부 텍스트 분석에서는 원문 재낭독 대신 분석 결과를 우선한다.
- recent conversation은 transcript가 아니라 1문단 요약으로 넘기는 방향을 기본으로 한다.

## 12. 구현 작업 시 주의
- 새 파일을 만들면 왜 필요한지 분명해야 한다.
- wrapper와 레거시 브리지는 가능하면 줄인다.
- 단, 안정적인 전환이 필요하면 임시 wrapper는 허용하되 이후 제거 계획을 남긴다.
- 타입 경고가 반복되면 "실행은 된다"로 넘기지 말고 구조적으로 정리한다.
- 실행 검증과 정적 검증을 함께 본다.
- 일반 채팅 경로의 general 저장이 실제로 살아 있는지 항상 확인한다.
- topic seed는 잘못 뽑힌 첫 문장 하나에 과도하게 의존하지 않도록 주의한다.

## 13. 결과 보고 방식
- 무엇을 바꿨는지
- 왜 바꿨는지
- 아직 안 끝난 건 무엇인지
를 분리해서 말한다.
- 완료처럼 보이게 포장하지 않는다.
- 부분 완료면 부분 완료라고 분명히 말한다.

## 14. 당분간의 실제 우선 작업
1. ChatEvidenceService.extract()에서 모델의 직전 답변 제외
2. EvidenceNormalizationService.normalize_chat_update_bundle()에서 topic seed를 요약된 현재 대화로 전환
3. ExtractionPolicy.extract()에서 general 저장 추가
4. MemoryClassificationPolicy에서 general이 실제 저장되도록 정리
5. promotion 의미론 정리
6. MemoryApplyService.apply_extracted()에서 general 실제 저장
7. memory.db 오염 방지 검증 및 재시험
