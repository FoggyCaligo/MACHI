## 1. 이 문서의 역할

이 README는 단순 실행 안내가 아니라, **MK6의 정책·개념·작동방식·최근 구현을 통해 새로 확인된 해석 기준**까지 함께 담는 기준 문서다.

즉 이 문서는 다음을 먼저 고정한다.

- MK6가 무엇을 만들려는 프로젝트인지
- MK5와 무엇이 근본적으로 달라졌는지
- 단어, 의미그래프, 임시 사고 그래프, 세계그래프가 각각 무엇인지
- 검색, 번역, 사고, 커밋, 언어화가 어떤 순서와 의미로 동작하는지
- 코드가 흔들리거나 문서 간 표현 차이가 있을 때 무엇을 우선 기준으로 읽어야 하는지

핵심 원칙은 하나다.

**MK6는 텍스트를 저장하는 시스템이 아니라, 의미그래프를 중심으로 인식하고 사고한 뒤, 필요할 때만 언어를 사용하는 시스템이다.**

---

## 2. MK5에서 MK6로 넘어오며 바뀐 것

MK5는 언어체계 위에 장기기억과 판단 구조를 붙인 프로젝트였다.
MK6는 거기서 기능을 더 얹는 방향이 아니라, **언어를 앞단 인터페이스로 내리고 의미그래프를 본체로 두는 구조**로 넘어간다.

핵심 전환은 아래와 같다.

1. **노드는 단어가 아니다.**  
   노드는 그래프가 체감한 개념이다.

2. **단어는 의미의 본체가 아니다.**  
   단어는 의미그래프에 접근하기 위한 앞단 표면 표현이다.

3. **언어 입력은 곧바로 저장되지 않는다.**  
   먼저 언어는 그래프로 번역되어야 한다.

4. **결론은 문장이 아니라 구조다.**  
   최종 문장은 결론 구조를 다시 언어화한 결과다.

5. **생각은 임시 사고 그래프에서 하고, 저장은 세계그래프에 선택적으로 한다.**

## 3. MK6의 한 줄 정의

MK6는 아래 구조를 목표로 한다.

```text
언어입력
→ 언어를 그래프로 번역
→ 현재 알고 있는 의미그래프를 호출
→ 임시 사고 그래프에서 정합성/의도 기준으로 조정
→ 필요한 경우만 검색과 세계그래프 업데이트 수행
→ 결론 그래프를 다시 단어 레이어를 통해 언어화
→ 언어출력
```

즉, MK6는 **언어입력 → 그래프 번역 → 그래프 사고 → 그래프 언어화** 구조다.

---

## 4. 두 레이어: 단어 레이어와 의미그래프 레이어

### 4.1 단어 레이어

단어 레이어는 그래프라기보다 **lookup table / address table**에 가깝다.

역할은 다음과 같다.

- 표면어를 저장한다.
- 표면어를 뒤쪽 의미 노드의 `address_hash`에 연결한다.
- 언어 입력 시 단어를 바로 의미그래프의 특정 국소그래프로 번역할 수 있게 한다.
- 언어화 시 의미 노드를 다시 표면어로 표현할 때 사용한다.

즉 단어 레이어는 의미를 담는 본체가 아니라, **의미그래프와 인간 언어를 연결하는 앞단 인터페이스**다.

### 4.2 의미그래프 레이어

의미그래프는 MK6의 본체다.

역할은 다음과 같다.

- 체감된 개념이 존재한다.
- 개념 간 관계와 상태가 엣지 구조로 표현된다.
- 정합성 조정, 분화/수렴, 의도 만족 여부 판단이 여기서 일어난다.

중요한 점은, **의미는 노드의 레이블보다 노드의 위치·연결·활성화·엣지 상태에 더 강하게 존재한다**는 것이다.

---

## 5. LangToGraph란 무엇인가

`LangToGraph`는 입력 문장을 저장하는 함수가 아니다.
또 단순 전처리 함수도 아니다.

`LangToGraph`는 **언어 도메인 → 그래프 도메인 번역기**다.

입력:
- 사용자 문장
- 검색 결과 텍스트

출력:
- `TranslatedGraph`

의미:
- 현재 시스템이 알고 있는 의미그래프를 바탕으로, 입력을 **국소그래프들의 집합**으로 번역한다.
- 찾지 못한 개념은 `EmptySlot`으로 남긴다.

### 5.1 중요한 해석 기준

MK6에서 번역 결과는 “문장 자체의 이상적 논리 구조”가 아니라,
**현재 세계그래프가 이해 가능한 범위 안에서 호출된 의미그래프**에 더 가깝다.

그래서 초기 사고 그래프는 종종 아래처럼 읽는 것이 맞다.

- 문장 관계를 그대로 보존한 그래프
  가 아니라
- **현재 알고 있는 의미그래프의 국소구조들을 호출해온 결과의 합집합**

이것은 버그가 아니라 의도다.

---

## 6. LocalSubgraph란 무엇인가

각 단어 또는 해석 단위가 의미그래프의 특정 노드와 연결되면,
MK6는 그 노드 하나만 가져오지 않고 **그 노드를 중심으로 한 국소그래프(LocalSubgraph)** 를 함께 가져온다.

즉 MK6의 이해 단위는 보통 다음에 가깝다.

- 중심 개념 노드
- 그 노드와 의미상 가까운 주변 노드
- 그들 사이의 현재 관계 엣지

그래서 사용자 입력은 최종적으로 “단어들의 집합”이 아니라,
**국소그래프들의 집합**으로 번역된다.

### 6.1 현재 구현에서 실제 생성 방식

현재 구현에서 `local_subgraph`는 다음 흐름으로 생성된다.

1. 토큰이 `words` exact match에 걸리면 해당 `address_hash`의 노드를 찾는다.
2. exact match가 없으면 임베딩 유사도로 가장 가까운 노드를 찾는다.
3. 찾은 노드를 중심으로 `LocalGraphExtractor`가 BFS로 N-hop 이웃을 추출한다.
4. 이렇게 만들어진 `LocalSubgraph(center_hash, nodes, edges, hop_radius)`가 `ConceptPointer` 안에 담긴다.

추가 정책:
- **중심 노드(depth=0)는 trust threshold와 무관하게 포함**한다.
- 주변 이웃만 threshold로 필터링한다.

이 예외는 구현 중 확인된 중요한 해석 기준이다. 중심 노드까지 필터링하면 words exact match로 찾은 개념이 임시 사고 그래프에서 사라질 수 있다.

---

## 7. TranslatedGraph의 의미

`TranslatedGraph`는 언어 입력의 그래프 번역 결과다.

구성 요소는 크게 둘이다.

- `ConceptPointer` / `EmptySlot`
- 번역된 문장 관계 엣지(`TranslatedEdge`)

하지만 해석 우선순위는 분명하다.

1. 본체는 각 포인터가 들고 온 `LocalSubgraph`다.
2. 문장 관계 엣지는 “문장에 보이는 관계 힌트”다.
3. 이 관계 힌트는 곧바로 세계그래프의 진실로 확정되지 않는다.

즉 초기 사고의 재료는 보통:

- 번역된 문장 관계 자체
  보다
- **현재 세계그래프에서 불러온 국소구조들의 합집합**

에 더 가깝다.

---

## 8. 검색의 위치

검색은 `LangToGraph`의 내부 본체가 아니다.

정책은 다음과 같다.

1. `LangToGraph`는 먼저 **알고 있는 것만으로** 번역한다.
2. 모르는 것은 `EmptySlot`으로 남긴다.
3. Think가 필요를 판단하면 검색한다.
4. 검색 결과도 다시 `LangToGraph`를 거친다.

즉 검색의 실제 위치는 아래다.

```text
LangToGraph
→ EmptySlot / 근거 부족 / 업데이트 필요 상태 남김
→ Think가 검색 필요 판단
→ graph > words > external search > LangToGraph
```

### 8.1 검색 실패

검색 결과가 없으면 그것은 그대로 하나의 상태다.

- “검색 결과도 없구나”
- “아직 채울 근거가 없구나”

로 받아들여야 한다.

검색 실패를 문자열 fallback으로 숨기거나, 억지 정답으로 덮는 방향은 정책과 맞지 않는다.

### 8.2 부트스트랩용 약한 ingest

현재 구현에는 부트스트랩을 위해,
검색으로도 채워지지 않을 때 약한 신규 노드를 생성하는 경로가 들어갈 수 있다.

이건 **search를 먼저 시도한 뒤의 보조 전략**이다.
검색 실패를 감추는 우회가 아니다.

## 9. 임시 사고 그래프와 세계그래프

### 9.1 세계그래프(WorldGraph)

- SQLite에 저장되는 영구 그래프
- 지금까지의 개념과 관계의 누적 상태
- 단어 테이블과 의미 노드/엣지의 실제 저장소

### 9.2 임시 사고 그래프(TempThoughtGraph)

- 현재 턴에서만 쓰이는 메모리상 그래프
- LangToGraph로 불러온 국소그래프들
- 목표 관련 노드
- 임시 연결/해제/조정 결과

핵심 정책은 아래다.

- **생각은 임시 사고 그래프에서 한다.**
- **세계그래프는 필요한 경우에만 갱신한다.**

즉, 현재 턴의 정합성 조정과 의도 튜닝은 우선 임시 사고 그래프에서 일어나야 한다.

## 10. AI 의도 구조

MK6에서 의도는 단순 문자열 태그가 아니다.

MVP 기준 기본 구조는 다음과 같다.

- 기본 노드로 `Goal Node`를 둔다.
- 그래프로 번역된 사용자 입력 의도를 여기에 임시 연결한다.
- Think는 현재 사고 국소그래프 집합이 이 목표와 얼마나 정합적인지 본다.

즉 사고는 전체 그래프를 무작정 개선하는 작업이 아니라,
**현재 목표와 입력 의도에 맞는 방향으로 국소그래프를 조정하는 과정**이다.

## 11. Think 루프

MK6의 Think는 1회성 판정이 아니라 루프다.

루프 안에서 할 수 있는 일은 다음과 같다.

- EmptySlot 처리 여부 판단
- 검색 필요 여부 판단
- 국소그래프 간 정합성 조정
- 임시 엣지 생성/해제
- 분화/수렴 후보 탐색
- 목표 만족 방향으로 그래프 튜닝

### 11.3 목표 지향 사고 루프(정책 보강)

Think의 목적은 "그래프를 업데이트하는 것" 자체가 아니다.  
정책상 더 중요한 목적은 다음과 같다.

- 세계그래프를 기준으로 현재 활성화된 국소그래프 집합을 조작한다.
- 목표(Goal)에 가장 유사하게 닿는 경로를 탐색한다.
- 그 과정에서 결론 서브그래프를 만들고, 이를 언어화한다.

즉 입력의 표면 토큰을 반복하는 것이 아니라, 목표 도달성 기준으로
관련 개념을 끌어오는 것이 맞다.

예시:
- 입력: "불을 끄려면?"
- 기대 결론: "물", "소방수", "소화기" 등 목표("진압/끄기")에 실제로 기여하는 노드
- 비목표: "불, 끄다" 같은 표면 재진술만으로 끝나는 응답

향후 구조 분리 계획:
- `update` 단계: 세계그래프 반영(강/약 커밋)
- `think` 단계: 목표 유사도 최대화를 위한 결론 탐색

현재 구현은 두 역할이 단일 Think 루프에 함께 들어있지만, 정책상은 분리 가능한
두 단계로 보는 것이 맞다.

### 11.1 수렴 판단

수렴은 단순 수치 카운터로만 보면 안 된다.

원칙적으로는 아래 감각에 가깝다.

- 더 수정할 곳이 딱히 안 보인다.
- 생각을 계속해도 개선점이 안 나온다.

따라서 수렴 판단은 단순 “전체 개수 변화 없음”보다,
**현재 목표와 연결된 활성 영역에서 의미 있는 구조 변화가 더 이상 나오지 않는 상태**를 봐야 한다.

### 11.2 현재 구현 해석 주의점

현재 구현은 delta나 개수 변화 중심으로 수렴을 단순화한 부분이 있다.
이건 정책의 최종 형태라기보다 MVP 구현의 임시 형태에 가깝다.

정책상 더 바람직한 방향은 다음에 가깝다.

- 큰 엣지 신규 생성
- 큰 엣지 삭제
- `connect_type` 변경
- EmptySlot 해결 상태 변화
- 분화/수렴으로 인한 구조 재배치

같은 **구조적 변화** 중심으로 수렴을 판단하는 것이다.

## 12. 세계그래프 커밋 정책

임시 사고 그래프에서 나온 모든 변화가 곧바로 세계그래프에 저장되면 안 된다.

### 12.1 강하게 커밋할 수 있는 경우

- 특수상황이 아닌 일반적 대응에 쓸 수 있는 업데이트
- 기존 세계그래프와 충돌하는 결론이 나와 trust / stability 조정이 필요한 경우
- 새로운 연결이 실제로 의미 있다고 판단되는 경우

### 12.2 약하게만 남기는 경우

- 아직 범용성 확신이 낮은 연결
- 이번 사고에는 유용했지만 일반화가 불명확한 변경
- 초기 학습 단계의 느슨한 연결

즉 MK6는 “생각에서 나온 모든 것 저장”이 아니라,
**사고 결과 중 어떤 것만 세계그래프로 반영할지 선택적으로 판단**해야 한다.

## 13. ConceptDifferentiation

MK6는 노드를 단순 표면어 버킷으로 늘리는 방향이 아니라,
점차 **체감된 개념을 더 잘 분리하고 수렴시키는 방향**을 지향한다.

현재 정책 방향은 다음과 같다.

- 임베딩 유사도만 보지 않는다.
- 가능하면 `LocalSubgraph`의 이웃 overlap도 함께 본다.
- 초기엔 그래프가 비어 있으므로 임베딩 비중이 높다.
- 그래프가 성숙할수록 구조 overlap 비중을 높인다.

즉, 기본 방향은 **적응형 복합 스코어**다.

### 13.1 개념병합(ConceptMerge) 필요성

분화만으로는 의미 그래프가 건강하게 유지되지 않는다.  
정책상 Think 루프에는 개념병합도 함께 있어야 한다.

- 분화: 과도하게 뭉친 개념을 분리
- 병합: 중복/동의/표현 차이 노드를 통합

둘은 대칭적인 정리 축이다.  
분화만 있고 병합이 없으면 그래프는 파편화되고, 병합만 있고 분화가 없으면
그래프는 과도하게 뭉친다.

---

## 14. 입력 타입 분류

MK6는 자연어만 다루고 끝나는 시스템이 아니다.
나중에 파일 읽기 / 코드 수정 도구가 붙을 것을 전제한다.

그래서 입력 처리 앞단에는 `InputTypeClassifier`가 있다.

최소 분류 대상:
- natural
- code
- path
- url

정책은 **규칙 우선 + 임베딩 폴백**이다.

이유:
- URL, 경로처럼 명확한 것은 빠르게 처리 가능하다.
- 모호한 경우만 임베딩으로 보조할 수 있다.
- 나중에 코드/파일 도구를 붙일 때 구조를 유지하기 쉽다.

---

## 15. GraphToLang

최종 출력은 곧바로 “그래프 설명문”이 아니다.

정책상 GraphToLang은 아래를 해야 한다.

1. 결론 그래프를 본다.
2. 각 노드를 다시 단어 레이어를 통해 표면어로 매핑한다.
3. 그 표면어와 구조를 바탕으로 LLM이 자연어를 생성한다.

즉 최종 문장은 **결론 구조를 다시 언어화한 결과**여야 한다.

### 15.1 현재 구현 해석 주의점

현재 MVP는 일부 구간에서 `node.labels`를 바로 사용해 언어화하는 경향이 있다.
정책상 최종 방향은 **단어 레이어를 통한 역매핑을 더 명시적으로 강화하는 것**이다.

## 16. 최근 구현을 통해 새로 확인된 것

이번 구현과 디버깅을 통해, 정책 해석상 중요하다고 확인된 것들을 정리하면 다음과 같다.

1. **`TranslatedGraph`의 핵심은 문장 엣지보다 포인터가 들고 온 `LocalSubgraph`다.**
2. **중심 노드는 local graph 추출 시 trust threshold로 잘라내면 안 된다.**
3. **임베딩 전용 모델은 generation 모델 목록에서 제외해야 한다.**
4. **검색 결과 없음은 실패를 숨겨야 하는 예외가 아니라 하나의 상태다.**
5. **수렴 판단은 전체 개수 변화보다 구조적 변화 중심으로 가는 것이 정책상 더 맞다.**
6. **README / overview / schema / file_structure보다 상위 정책 문서가 필요하며, 구현 해석은 정책을 우선해야 한다.**

## 20. 현재 한계

현재 MVP는 아직 아래 영역이 완전히 닫힌 상태는 아니다.

- 수렴 판단이 정책보다 단순하게 구현된 부분
- `TranslatedGraph.edges`를 임시 사고 그래프 초기 상태에 어떻게 반영할지의 정교화
- GraphToLang의 단어 레이어 역매핑 강화
- 세계그래프 커밋 기준의 세분화
- 목표 유사도 기반 결론 탐색 루프의 명시적 분리 (`update` vs `think`)
- ConceptDifferentiation과 대칭되는 ConceptMerge 루틴 부재
- 파일/코드 도구 레이어 연결

즉 지금 구현은 **정책을 향해 가는 첫 구현**이며,
정책 그 자체와 완전히 동일한 최종 상태라고 보면 안 된다.

## 22. 앞으로의 읽기 기준

MK6를 읽을 때 아래 질문으로 해석하면 방향을 잃지 않는다.

- 이 입력은 곧바로 저장되는가, 아니면 먼저 그래프로 번역되는가?
- 이 노드는 단어인가, 아니면 체감된 개념인가?
- 지금 변화는 임시 사고 그래프에서 일어난 것인가, 세계그래프에 커밋된 것인가?
- 이 검색은 LangToGraph의 일부인가, Think가 필요를 판단해서 부른 것인가?
- 지금 출력 문장은 그래프를 직접 말한 것인가, 결론 구조를 다시 언어화한 것인가?

이 다섯 질문에 일관되게 답할 수 있으면, MK6를 대체로 올바르게 읽고 있는 것이다.
