# MK6 Concept

## 1. 문서의 목적
이 문서는 MK6의 **정책 우선 문서**다.
구현 파일별 설명이나 일시적 버그 수정 이력보다 먼저 읽어야 하는 문서이며, 다음을 고정한다.

- MK6가 무엇을 만들려는 프로젝트인지
- 언어, 단어, 의미그래프, 사고그래프, 세계그래프가 각각 무엇인지
- 검색, 번역, 사고, 커밋, 언어화가 어떤 순서와 의미로 동작하는지
- 구현이 흔들릴 때 무엇을 기준으로 해석해야 하는지

이 문서는 “현재 코드가 우연히 어떻게 되어 있는가”보다 “MK6가 원래 어떤 철학과 방법론으로 설계되었는가”를 우선한다.

---

## 2. MK5에서 MK6로 넘어오며 바뀐 핵심
MK5는 언어체계 위에 장기기억과 판단 구조를 붙인 프로젝트였다.
MK6는 그 위에 기능을 덧붙이는 것이 아니라, **언어를 앞단 표면 레이어로 내리고, 뒤에 의미그래프를 본체로 두는 구조**로 넘어간다.

핵심 전환은 다음과 같다.

1. **노드는 단어가 아니다.**
   노드는 그래프가 체감한 개념이다.
2. **단어는 의미의 본체가 아니다.**
   단어는 의미그래프에 접근하기 위한 앞단 표면 표현이다.
3. **언어 입력은 곧바로 저장되지 않는다.**
   먼저 언어는 그래프로 번역되어야 한다.
4. **결론은 문장이 아니라 구조다.**
   최종 문장은 결론 구조를 다시 언어화한 결과다.

즉, MK6는 “텍스트를 저장하는 시스템”이 아니라, **의미그래프를 중심으로 인식하고 사고한 뒤, 필요할 때만 언어를 사용하는 시스템**을 목표로 한다.

---

## 3. MK6의 기본 파이프라인
기본 흐름은 아래와 같다.

```text
언어입력
→ 입력 분류
→ 토큰/문장 분해
→ LangToGraph
→ 임시 사고 그래프 구성
→ Think 루프
→ 결론 그래프 수렴
→ 단어 레이어로 역매핑
→ 언어출력
```

여기서 중요한 점은 다음이다.

- LangToGraph는 ingest가 아니다.
- search는 원칙적으로 LangToGraph 내부의 본체가 아니라, **Think가 필요를 감지했을 때 호출되는 보조 경로**다.
- 최종 출력은 “그래프를 설명하는 문장”이 아니라, **결론 그래프를 단어 레이어를 통해 다시 언어화한 결과**다.

---

## 4. 두 레이어: 단어 레이어와 의미그래프 레이어
### 4.1 단어 레이어
단어 레이어는 그래프라기보다 **lookup table / address table**에 가깝다.

역할:
- 표면어를 저장한다.
- 표면어를 뒤쪽 의미 노드의 `address_hash`로 연결한다.
- 언어 입력 시, 단어를 바로 의미그래프의 특정 국소그래프로 번역할 수 있게 한다.
- 언어화 시, 의미 노드를 다시 표면어로 표현할 때 사용한다.

즉 단어 레이어는 “의미를 담는 본체”가 아니라, **의미그래프를 인간 언어와 연결하는 앞단 인터페이스**다.

### 4.2 의미그래프 레이어
의미그래프는 MK6의 본체다.

역할:
- 체감된 개념이 존재한다.
- 개념 간 관계와 상태가 엣지 구조로 표현된다.
- 사고, 정합성 조정, 분화/수렴, 의도 만족 여부 판단이 여기서 일어난다.

의미는 노드의 레이블보다 **노드의 위치, 연결, 활성화, 엣지 상태**에 더 강하게 존재한다.

---

## 5. LangToGraph란 무엇인가
LangToGraph는 입력 문장을 저장하는 함수도 아니고, LLM에 전달하기 전 전처리만 하는 함수도 아니다.

LangToGraph는 **언어 도메인 → 그래프 도메인 번역기**다.

입력:
- 문장 또는 검색 결과 텍스트

출력:
- `TranslatedGraph`

의미:
- 사용자의 언어 입력을, 현재 시스템이 알고 있는 의미그래프에 기초한 **국소그래프들의 집합**으로 번역한다.
- 못 찾은 개념은 `EmptySlot`으로 남긴다.

### 5.1 LangToGraph의 핵심 원칙
1. **현재 알고 있는 의미그래프에 기인해 번역한다.**
   번역 결과는 “문장 자체의 이상적 논리 구조”가 아니라, **현재 세계그래프가 이해 가능한 범위 안에서의 의미그래프**다.
2. **모르는 것은 비워 둔다.**
   억지로 채우지 않는다. 못 찾으면 `EmptySlot`으로 남긴다.
3. **검색 결과는 LangToGraph를 거치지 않는다.**
   빈 DB에서 재파싱하면 EmptySlot cascade와 PoolTimeout이 발생한다.
   검색 결과는 `_ingest_slot`으로 직접 노드를 생성하고 `payload[“search_summary”]`에 요약만 저장한다.
   GraphToLang이 이 요약을 LLM 컨텍스트(`[검색 컨텍스트]`)로 활용한다.

---

## 6. LocalSubgraph의 의미
각 단어 또는 해석 단위가 의미그래프에서 특정 노드와 연결되면, 그 노드 자체만이 아니라 **그 노드를 중심으로 한 국소그래프(LocalSubgraph)** 가 함께 반환된다.

즉 MK6가 이해하는 단위는 단일 노드가 아니라, 보통 아래에 더 가깝다.

- 중심 개념 노드
- 그 노드와 의미상 가까운 주변 노드
- 그들 사이의 현재 관계 엣지

그래서 사용자 입력은 최종적으로 “단어들의 집합”이 아니라, **국소그래프들의 집합**으로 번역된다.

---

## 7. TranslatedGraph의 의미
`TranslatedGraph`는 언어 입력의 그래프 번역 결과다. 다만 이것을 해석할 때 주의가 필요하다.

### 7.1 노드 측면
TranslatedGraph의 본체는 각 토큰/표현이 참조한 **ConceptPointer와 그 내부 LocalSubgraph들**이다.

### 7.2 엣지 측면
TranslatedGraph에 들어 있는 문장 관계 엣지는 “문장에 보이는 관계 힌트”일 뿐, 곧바로 세계그래프의 진실로 확정되지 않는다.

즉 MK6에서 초기 해석은 보통 다음에 더 가깝다.

- 문장 자체의 구조를 우선 저장한다
  가 아니라
- **문장이 현재 알고 있는 의미그래프를 어떤 방식으로 호출하는지**를 먼저 잡는다

그래서 초기 사고 그래프는 “번역된 문장 관계 그 자체”보다 **현재 세계그래프에서 불러온 국소구조들의 합집합**에 가깝다. 이것은 버그가 아니라 의도다.

---

## 8. 입력 분해와 입력 타입 분류
MK6는 자연어만 다루는 시스템으로 끝나지 않는다. 이후 파일 읽기/코드 수정 도구가 붙을 것을 전제한다.

그래서 입력 처리 앞단에는 두 가지가 있다.

1. **문장/토큰 분해**
2. **입력 타입 분류**

입력 타입 분류는 최소한 아래를 구분할 수 있어야 한다.

- natural
- code
- path
- url

MVP 정책은 **규칙 우선 + 임베딩 폴백**이다.

이유:
- URL, 경로처럼 명확한 것은 빠르게 처리 가능하다.
- 애매한 경우만 임베딩으로 보조할 수 있다.
- 나중에 코드/파일 도구를 붙일 때 구조를 유지하기 쉽다.

---

## 9. 검색의 위치와 의미
검색은 LangToGraph의 내부 본체가 아니다.

기본 정책은 다음이다.

1. LangToGraph는 먼저 **알고 있는 것만으로** 번역한다.
2. 모르는 것은 `EmptySlot`으로 남긴다.
3. Think가 필요를 판단하면 검색한다.
4. 검색 결과는 LangToGraph 재파싱 없이 직접 ingest한다. (요약을 payload에 저장)

즉 검색의 진짜 위치는 아래다.

```text
LangToGraph
→ EmptySlot 상태 남김
→ Think가 검색 필요 판단
→ external search → _ingest_slot (hint당 직접 노드 생성 + payload["search_summary"] 저장)
→ GraphToLang이 payload를 [검색 컨텍스트]로 LLM에 주입
```

### 9.1 검색 실패
검색 결과가 없으면 `None` 또는 실패 상태를 그대로 수용한다.
그 의미는 “검색 결과도 없구나”이지, 숨겨야 할 예외가 아니다.

### 9.2 약한 초기 ingest
구현상 부트스트랩을 위해, 검색으로도 채워지지 않을 때 약한 신규 노드를 생성하는 전략을 사용할 수 있다.
하지만 이 역시 **search를 먼저 시도한 뒤** 보조적으로 쓰는 것이다. 검색 실패를 덮기 위한 우회가 아니다.

---

## 10. 임시 사고 그래프와 세계그래프
MK6는 사고용 그래프와 저장용 그래프를 분리한다.

### 10.1 세계그래프(WorldGraph)
- SQLite에 저장되는 영구 그래프
- 현재까지의 개념과 관계의 누적 상태
- 단어 테이블과 의미 노드/엣지의 실제 저장소

### 10.2 임시 사고 그래프(TempThoughtGraph)
- 현재 턴에서만 쓰이는 메모리상 그래프
- LangToGraph로 불러온 국소그래프들 + 현재 목적 관련 노드들 + 임시 조작 결과를 담는다
- 연결/해제/병합/분할/임시 엣지 조작은 우선 여기서 일어난다

핵심 정책:
- **생각은 임시 사고 그래프에서 한다.**
- **세계그래프는 필요한 경우에만 갱신한다.**

---

## 11. AI 의도 구조
MK6에서 “의도”는 단순한 문자열 태그가 아니다.

기본 MVP에서는 하나의 기본 노드, 즉 **목표(goal)** 를 둔다.
그리고 그래프로 번역된 사용자 입력 의도를 이 목표 노드에 임시 연결한다.

그 후 Think는 다음을 본다.

- 현재 사고 국소그래프 집합이
- 목표에 연결된 입력 의도와 얼마나 정합적인가
- 목표를 만족하는 방향으로 어떤 연결/해제/조정이 필요한가

즉 사고는 “무작정 전체 그래프를 개선”하는 게 아니라, **현재 목표와 입력 의도에 맞는 방향으로 국소그래프를 조정하는 과정**이다.

---

## 12. 사고 과정: Update 루프와 Think 루프

MK6의 사고 과정은 두 단계로 분리된다.

### 12.1 Update 루프 (현재 구현)

“재료 준비” 단계다. 세계그래프를 최신 상태로 갱신하는 것이 목적이다.

루프 안에서 할 수 있는 일:
- EmptySlot 탐지 → 검색 → ingest
- ConceptDifferentiation (분화)
- ConceptMerge (병합) [예정]
- WorldGraph 커밋

수렴 조건: 구조적 변화(노드/엣지 증감)가 더 이상 없을 때.

### 12.2 Think 루프 [예정]

“요리” 단계다. Update로 갱신된 그래프 위에서 목표에 닿는 결론을 만들어내는 것이 목적이다.

**핵심 방향:**  
세계그래프를 기준으로, '목표'에 가장 유사하게 닿기 위해 현재 활성화된 국소그래프들을 이리저리 만져보며 결론을 만들어내는 과정이다.

예시:
- 입력: “불을 끄려면?”
- 단순 재진술: “불, 끄다” → **정책상 불충분**
- 목표 달성 결론: “물, 소화기, 소방수” → **이것이 Think가 만들어야 하는 것**

**동작 방식:**  
Goal 노드 방향으로 관련 노드 경로를 탐색(spreading activation)하고, 목표 달성에 기여하는 노드들을 결론 후보로 수집한다.

WorldGraph는 이 단계에서 수정되지 않는다.

### 12.3 수렴 판단

Update와 Think 각각의 수렴 기준이 다르다.

- **Update 수렴**: 구조적 변화 없음 (노드/엣지 수 변화 없음)
- **Think 수렴**: 목표 방향으로 활성화할 새 경로가 더 이상 없음

두 루프 모두 `MAX_LOOPS` 상한을 안전장치로 유지한다.

---

## 13. 세계그래프 커밋 정책
임시 사고 그래프에서 나온 모든 변화가 곧바로 세계그래프에 저장되어서는 안 된다.

강하게 커밋될 수 있는 경우:
- 특수상황이 아닌 일반적 대응에 쓰일 수 있는 업데이트
- 기존 세계그래프와 충돌하는 결론이 나와 trust/stability 조정이 필요한 경우
- 새로운 연결이 실제로 의미 있다고 판단되는 경우

약하게만 남길 수 있는 경우:
- 아직 범용성 확신이 낮은 연결
- 이번 사고에선 유용했지만 일반화가 불명확한 변경
- 초기 학습 단계의 느슨한 연결

즉 커밋은 “무엇이 바뀌었는가”만이 아니라, **그 변화가 세계 수준 지식으로 승격될 가치가 있는가**를 함께 본다.

---

## 14. ConceptDifferentiation
개념 분화/수렴은 MK6의 핵심 정책 중 하나다.

기본 방향:
- 임베딩 유사도만으로 끝내지 않는다.
- 국소그래프 구조도 함께 본다.
- 초기 그래프가 비어 있을 때와 그래프가 성숙했을 때의 기준이 같으면 안 된다.

정책:
- 복합 스코어 사용
- `score = α × embedding_similarity + (1-α) × local_overlap`
- 이웃이 적은 초기에 α를 높이고, 그래프가 성숙할수록 overlap 비중을 늘린다.

즉 MK6의 개념 분화/수렴은 단순 문자열 병합이 아니라, **현재 체감 구조가 실제로 같은 개념인지, 혹은 더 분리되어야 하는지를 그래프적으로 판단하는 작업**이다.

---

## 15. words 테이블 동기화 정책
단어 레이어와 의미그래프 레이어는 분리되어 있지만, 완전히 따로 놀 수는 없다.

### merge 시
- 기존 두 개념 노드에 연결된 단어 연결을 병합 대상 노드로 옮긴다.

### differentiation 시
- 기존 단어 연결을 분화된 두 노드에 재배분한다.
- 불명확하면 중복 허용도 가능하다.

핵심은 단어 연결이 본체 의미를 결정하는 것이 아니라, **이미 재편된 의미 구조를 따라 다시 연결을 정리하는 것**이다.

---

## 16. GraphToLang
최종 언어화는 단순히 `node.labels`를 나열하는 작업이 아니다.

기본 원칙:
1. 결론 그래프가 먼저다.
2. 각 의미 노드는 자신과 연결된 단어들을 통해 표현된다.
3. 추상 노드나 레이블 없는 노드도 words 테이블과 연결 구조를 통해 표현을 끌어온다.
4. LLM은 “결론 구조를 자연스럽게 말하게 하는 마지막 계층”이다.

즉 LLM은 의미를 본체로 생성하는 것이 아니라, **이미 수렴된 그래프 구조를 언어로 드러내는 마지막 표면화 계층**이다.

## 17. 구현 해석 원칙
현재 코드나 문서가 어긋날 때는 아래 우선순위로 해석한다.

1. 이 문서의 정책
2. MK5 → MK6 전환 대화에서 확정된 철학
3. 세부 아키텍처 문서
4. 현재 구현 상태

즉 현재 구현이 이 문서와 다르면, 우선 “구현이 뒤처졌는지 / 문서가 오래됐는지 / 정책이 아직 반영되지 않았는지”를 먼저 의심해야 한다.

---

## 18. 향후 도구 레이어
파일 읽기, 로컬 코드 수정 등은 MK6의 2차 과제다.

중요한 점:
- 이것은 MK6의 본체가 아니다.
- 본체는 여전히 의미그래프 기반 인지 구조다.
- 도구는 나중에 붙더라도, MK6가 그것을 다룰 수 있게 입력 분류와 파이프라인을 미리 열어 두는 것이 목적이다.

---

## 19. 한 줄 요약
MK6는 **단어를 본체로 저장하는 시스템이 아니라, 단어를 앞단 주소 체계로 사용해 의미그래프를 불러오고, 임시 사고 그래프에서 목표 지향적으로 구조를 조정한 뒤, 그 결론 구조를 다시 언어로 번역하는 시스템**이다.
