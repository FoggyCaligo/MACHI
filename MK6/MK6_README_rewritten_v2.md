# Machi MK6 README

## 1. 한 문장 정의

MK6는 **다음 토큰을 예측하는 언어모델**이 아니라,  
**입력을 의미 그래프로 번역하고, 그 그래프 위에서 사고를 전개한 뒤, 마지막에만 언어로 치환하는 인지 시스템**을 목표로 한다.

즉, MK6의 기본 구조는 다음과 같다.

**언어 입력 → 그래프 번역 → 사고 → 결론 그래프 → 언어화**

---

## 2. MK6가 하려는 일

MK6는 문장을 그대로 저장하는 시스템이 아니다.  
문장 속의 단어와 표현을 그대로 지식으로 취급하지 않고,  
그 뒤에 있는 **의미 구조**를 그래프로 다루려는 프로젝트다.

핵심 관심사는 다음과 같다.

- 개념이 어떻게 생기는가
- 입력이 어떻게 기존 개념과 연결되는가
- 어떤 구조가 더 말이 되는 구조인가
- 결론이 어떻게 만들어지는가
- 언어는 어디까지나 마지막 표면화 단계가 되도록 어떻게 분리할 것인가

---

## 3. 핵심 철학

### 3-1. 노드는 단어가 아니라 체감된 개념이다

노드는 외부에서 받은 문장이나 정보 조각이 아니다.  
노드는 내부 그래프 안에서 반복 입력과 관계 구조를 거치며 안정화된 **체감된 개념**이다.

따라서 노드의 본체 의미는 텍스트가 아니라 다음에서 나온다.

- 어떤 노드들과 연결되어 있는가
- 어떤 엣지 타입으로 연결되어 있는가
- 어떤 공통부/차이 구조를 형성하고 있는가
- 사고 과정에서 어떤 구조적 역할을 하는가

### 3-2. 정책과 규칙은 노드가 아니라 엣지다

정책, 규칙, 관계, 차이 설명은 별도 텍스트 노드가 아니라  
**노드 사이의 연결**, 즉 엣지의 성질로 표현하는 방향을 따른다.

엣지는 단순 연결선이 아니라 내부 데이터를 가질 수 있어야 한다.

예:

- 왜 연결되었는가
- 어떤 점에서 유사한가
- 어떤 점에서 다른가
- 어떤 조건에서만 성립하는가
- 어떤 출처/형성 과정을 거쳤는가

### 3-3. 학습은 가중치 조정보다 구조 재작성에 가깝다

MK6에서 중요한 학습은 파라미터 미세 조정이 아니다.  
핵심은 다음이다.

- 노드 생성
- 공통부 추출
- 차이 분리
- 엣지 생성
- 연결 강화/약화
- 구조 재조직
- 개념 분화/수렴

### 3-4. 새 연결 자체는 문제가 아니다. 기존 연결을 부숴야 하면 문제다

MK6의 정합성 판단에서 가장 중요한 기준은 수치가 아니라 구조다.

새로운 연결이 생기는 것 자체는 자연스럽다.  
하지만 그 연결을 성립시키기 위해 **기존 연결을 부정하거나 끊어야 한다면**,  
그 가정은 우선 잘못된 것으로 본다.

즉 MK6는 기본적으로 **비파괴적 확장**을 우선한다.

## 4. 현재 구조 개념

### 4-1. 단어 레이어와 의미 그래프 레이어

MK6는 크게 두 층을 가진다.

#### 단어 레이어
- 표면어를 관리하는 앞단 레이어
- 본체 그래프라기보다 lookup / lexical memory / 주소록에 가깝다
- 단어를 통해 뒤쪽 의미 노드에 접근한다

#### 의미 그래프 레이어
- 실제 인지 구조가 존재하는 본체
- 노드 = 체감된 개념
- 엣지 = 관계 / 규칙 / 차이 / 정책 / 연결 상태

중요한 점은, **단어는 의미의 본체가 아니라 의미에 접근하거나 의미를 언어화할 때 쓰는 표면 표현**이라는 것이다.

### 4-2. LangToGraph

LangToGraph는 ingest가 아니다.  
문장을 저장하는 함수도 아니다.

LangToGraph는 **언어를 현재 세계가 이미 알고 있는 의미 그래프로 번역하는 인터페이스**다.

입력이 들어오면 다음을 수행한다.

1. 입력을 문장/토큰 수준으로 분해한다
2. 단어 레이어에서 대응 후보를 찾는다
3. 의미 그래프에서 해당 노드의 국소 구조를 가져온다
4. 찾지 못한 것은 `EmptySlot`으로 반환한다

즉 반환값은 텍스트가 아니라 다음에 가깝다.

- 국소 개념 그래프들
- 빈 슬롯들
- 현재 입력에 대한 구조적 번역 결과

### 4-3. 현재 사고 그래프

MK6는 사고용 별도 저장 세계를 하나 더 두려는 방향보다,  
**하나의 누적 그래프 위에서 매 턴마다 현재 사고 그래프(View)를 생성하는 방향**이 더 맞다고 본다.

즉 전체 개념 그래프는 누적되고,  
현재 입력과 현재 의도에 따라 그 일부가 활성화되어 **현재 사고 그래프**가 된다.

사고는 이 현재 사고 그래프 위에서 진행된다.

### 4-4. 의도와 목표

의도는 고정값이 아니다. 계속 변한다.  
다만 사고를 진행하는 동안에는 **지금 사고에 사용되는 의도 스냅샷**이 필요하다.

현재 구조 개념상:

- 장기 방향성: 기본 욕구
- 지금 계속 변하는 것: live intent
- 이번 사고 사이클 동안 기준점이 되는 것: intent snapshot / goal anchor

즉 사고는 “아무 그래프나 탐색”하는 것이 아니라,  
**현재 의도와 더 정합적인 방향으로 그래프를 재배열하는 과정**이다.

## 5. 현재 구현에서 중요한 개념 정리

### 5-1. `node.labels`의 의미

`node.labels`는 노드의 본체 의미가 아니다.  
최종 방향에서 `node.labels`는 다음 정도의 역할로 격하된다.

- lexical hint
- verbalization candidate
- alias cache
- debug용 관찰값

즉 **노드는 labels가 없어도 존재할 수 있어야 하고**,  
labels는 언어화에 도움이 되는 보조 정보여야 한다.

### 5-2. 한 노드에는 여러 단어가 붙을 수 있어야 한다

같은 의미 노드에는 여러 표면어가 붙을 수 있어야 한다.

예:
- 소방관
- 소방사
- 소방수

이 셋은 서로 다른 표면어이지만 같은 의미 노드와 연결될 수 있다.

### 5-3. 같은 단어가 여러 의미 노드에 매핑되는 구조도 필요하다

같은 단어가 문맥에 따라 여러 의미 노드로 연결되는 구조는 필수다.

즉 최종적으로 필요한 것은 다음이다.

- 하나의 단어 ↔ 여러 의미 노드
- 하나의 의미 노드 ↔ 여러 단어

따라서 lexical 구조는 **many-to-many**가 되어야 한다.

현재의 단순 `surface_form -> address_hash` 단일 연결 모델은 최종 방향이 아니다.  
장기적으로는 다음처럼 분리되는 방향이 맞다.

- `word_forms`: 단어 표면형 자체
- `word_meaning_links`: 단어 ↔ 의미 노드 연결

그리고 `word_meaning_links`는 단순 FK가 아니라 다음 같은 메타데이터를 가질 수 있어야 한다.

- link strength
- formation source
- 최근 사용 맥락
- 언어 / 레지스터
- 현재 대화에서의 활성도

즉, **words를 many-to-many 링크 구조로 재정의해야 한다.**

### 5-4. GraphToLang은 단어 복원 함수가 아니라 결론 그래프 직렬화 계층이다

GraphToLang의 목표는 단순히 노드명을 골라내는 것이 아니다.

GraphToLang은 다음을 해야 한다.

1. 결론 그래프의 노드들을 언어 후보와 함께 재구성
2. 결론 그래프의 엣지와 경로를 함께 직렬화
3. 언어화 모델이 최대한 오해 없이 읽을 수 있는 형태로 재표현
4. 마지막에만 LLM 또는 언어화 계층에 전달

즉 GraphToLang은 다음 구조에 가깝다.

- 선택된 노드
- 선택된 단어 후보
- 선택된 엣지
- 결론 경로
- 충돌/제외된 구조
- 논리 순서
- 언어화 힌트

**노드만 넘기면 안 되고, 관계(엣지)까지 같이 넘어가야 한다.**

그리고 이 계층은 단순한 lexical restore가 아니라,  
**“결론 그래프 직렬화 + 언어화 힌트 첨부 계층”**으로 재정의되어야 한다.

### 5-5. 현재 입력과 현재 사고에서 활성화된 단어 후보도 중요하다

언어화 시에는 영구 lexical memory만 보는 것이 아니라, 현재 턴의 사고 과정에서 실제로 활성화되거나 선택된 단어 후보도 중요하다.

따라서 장기적으로 표현 선택 우선순위는 다음에 가까워질 수 있다.

1. 현재 사고에서 활성화된 lexical candidate
2. 현재 입력에서 직접 사용된 단어
3. 영구 단어-의미 연결 테이블
4. 마지막 fallback으로서의 `node.labels`

## 6. 현재 파이프라인 개념

현재 목표 파이프라인은 다음에 가깝다.

1. 사용자 입력 수신
2. 입력 분해
3. LangToGraph를 통해 입력을 국소 의미 그래프로 번역
4. 현재 사고 그래프 구성
5. 사고 진행
   - 비어 있는 부분
   - 업데이트가 필요한 부분
   - 근거가 부족한 부분
   가 보이면 검색/보강
6. 더 말이 되는 구조로 수렴
7. 결론 그래프 형성
8. GraphToLang으로 결론 그래프 직렬화
9. 마지막에만 언어화 모델에 전달

검색은 독립 본체가 아니라,  
**그래프를 더 채우기 위한 보조 경로**다.

## 7. 지금까지 정해진 방향

다음 사항은 현재 기준으로 확정 방향으로 본다.

### 7-1. 확정 방향

- MK6는 언어모델이 아니라 인지 시스템을 목표로 한다.
- 사고와 언어화는 분리된다.
- 노드는 단어가 아니라 체감된 개념이다.
- 규칙/정책은 노드가 아니라 엣지에 실린다.
- LangToGraph는 ingest가 아니라 번역 인터페이스다.
- 현재 사고 그래프(View)는 전체 누적 그래프 위에서 매번 생성된다.
- 의도는 고정값이 아니라 계속 변하지만, 사고 중에는 스냅샷이 필요하다.
- 정합성의 1차 기준은 “기존 연결을 부숴야만 하는가”이다.
- `node.labels`는 본체 의미가 아니라 lexical hint 쪽으로 격하된다.
- words는 many-to-many 링크 구조로 재정의된다.
- GraphToLang은 결론 그래프 직렬화 계층으로 재정의된다.
- GraphToLang은 노드뿐 아니라 엣지와 경로까지 언어화 모델에 전달해야 한다.

### 7-2. 현재 구현이 이미 갖춘 것

- 기본 패키지 구조
- SQLite 기반 저장 계층
- LangToGraph의 기본 골격
- LocalSubgraph 추출
- Temp / current thought graph 관련 골격
- 검색 함수와 검색 후 보강 경로
- FastAPI 서버 / UI / 모델 선택 / 기본 실행 구조
- 테스트의 기본 토대

### 7-3. 현재 구현이 아직 과도기인 것

- `node.labels`가 여전히 비교적 강한 역할을 하고 있음
- lexical memory가 아직 단순 구조에 가까움
- 같은 단어 ↔ 여러 의미 노드 매핑이 구조적으로 완성되지 않음
- GraphToLang이 결론 그래프 직렬화보다 언어 힌트 직렬화에 가까움
- 공통부 추출 / 차이 분리 / 개념 분화가 본체 수준으로 완성되지 않음
- 노드가 완전히 “체감된 개념”으로 안정화되었다고 보기 어려움
- 언어 없는 추상 노드 운용이 아직 미완성

## 8. 아직 더 디벨롭이 필요한 것

### 8-1. 공통부 추출 / 차이 분리 / 개념 분화

이 축은 MK6의 핵심이다.  
데이터가 늘수록:

- 개념은 더 응축되고
- 차이는 더 분명해지고
- 구조는 더 정교해져야 한다.

즉 MK6는 단순히 더 많은 노드를 쌓는 방향이 아니라,  
**공통부를 구조적으로 응축하고 차이를 구조적으로 분리하는 방향**으로 더 발전해야 한다.

### 8-2. 노드의 “체감된 개념”성 강화

노드가 완전히 체감된 개념으로 작동하려면, 장기적으로 다음이 더 필요하다.

- 노드 정체성이 label이 아니라 관계 구조에서 나오기
- 분화 이력 / 재사용 이력이 더 구조적으로 남기
- 사고 과정에서 구조적 생존성을 통해 안정화되기
- 새로운 맥락에서도 기존 연결을 부수지 않고 확장 가능하기

### 8-3. lexical 구조 고도화

many-to-many 자체가 끝이 아니다.  
단어와 의미 사이의 연결은 다음을 더 다룰 수 있어야 한다.

- 다의어
- 동의어
- 맥락별 우선 표현
- 레지스터 차이
- 현재 답변에 더 맞는 표현 선택

### 8-4. GraphToLang 재구성

GraphToLang은 앞으로 다음 방향으로 더 발전해야 한다.

- 결론 그래프 직렬화의 표준 형식 정리
- 노드/엣지/경로/제외 구조를 모두 포함하는 전달 패키지 설계
- 언어 힌트와 구조 본체의 경계 명확화
- LLM이 최대한 구조를 오해하지 않도록 주석화 방식 정교화

### 8-5. 정합성 판단과 수렴 판단의 정교화

정합성은 단순 수치가 아니며,  
현재 가장 중요한 1차 기준은 “기존 연결을 부숴야만 하는가”다.

하지만 실제 사고 루프에서는 다음도 더 정교해질 필요가 있다.

- 무엇을 모순으로 볼 것인가
- 언제 더 생각할 이유가 없다고 볼 것인가
- 어떤 변화가 구조적으로 의미 있는 변화인가
- 어떤 경우를 세계 그래프에 강화 반영할 것인가

### 8-6. 단어 없는 의미 노드의 운용

장기적으로는 labels나 직접 단어가 없는 추상 노드도 운용되어야 한다.  
즉 언어가 없는 상태로도 개념이 존재할 수 있어야 하며,  
언어화 시점에만 주변 구조를 통해 간접적으로 표현될 수 있어야 한다.

---

## 9. 지금 코드에 대한 짧은 판단

현재 MK6는 철학과 방향은 꽤 많이 올라왔지만,  
아직 완전히 개념 형성형 인지 시스템으로 닫히진 않았다.

대략적으로 보면:

- 철학/방향성: 많이 올라옴
- 구조 스켈레톤: 상당 부분 있음
- lexical / GraphToLang / concept differentiation 완성도: 아직 과도기

즉 현재 MK6는 **좋은 방향의 뼈대와 철학은 확보했지만, 공통부 추출 / lexical 구조 / 진짜 결론 그래프 언어화 쪽은 더 디벨롭이 필요한 상태**라고 보는 게 맞다.

## 10. 실행 / 운영 메모

현재 서버/UI 실행 구조와 테스트 구조는 이미 존재한다.  
다만 README는 실행법만 보는 문서가 아니라, **정책과 방향을 우선하는 기준 문서**로 읽는 것이 맞다.

즉 새 작업을 시작할 때는 다음 순서로 판단한다.

1. 이 변경이 “노드는 체감된 개념”이라는 철학과 맞는가?
2. 이 변경이 단어를 의미의 본체로 되돌리지 않는가?
3. 이 변경이 엣지 중심 규칙 표현을 훼손하지 않는가?
4. 이 변경이 GraphToLang을 구조 직렬화가 아닌 단순 문자열화로 후퇴시키지 않는가?
5. 이 변경이 공통부 추출 / 차이 분리 / 개념 분화 방향과 충돌하지 않는가?

## 11. 최종 요약

MK6는 문장을 잘 이어 쓰는 시스템이 아니라,  
**입력을 의미 그래프로 번역하고, 현재 의도에 맞는 사고 그래프를 형성하고, 더 말이 되는 구조로 수렴시킨 뒤, 마지막에만 언어로 바꾸는 인지 시스템**을 목표로 한다.

현재까지 정해진 방향은 다음과 같다.

- 노드는 체감된 개념이다.
- 엣지는 규칙/관계/정책을 담는다.
- words는 many-to-many lexical 링크 구조로 재정의된다.
- GraphToLang은 결론 그래프 직렬화 + 언어화 힌트 첨부 계층으로 재정의된다.
- 공통부 추출 / 차이 분리 / 개념 분화는 더 강하게 디벨롭되어야 한다.

즉 MK6는 아직 완성본이 아니라,  
**철학과 방향은 꽤 많이 고정되었고, 그 철학을 실제 구조로 더 깊게 내려보내는 단계에 있는 프로젝트**다.
